Внести вклад в экосистему Nuxt можно по-разному.

## Экосистема

В экосистему Nuxt входят разные проекты и организации:

* [nuxt/](https://github.com/nuxt) — основные репозитории самого фреймворка. В [**nuxt/nuxt**](https://github.com/nuxt/nuxt) лежит Nuxt (версии 2 и 3).
* [nuxt-modules/](https://github.com/nuxt-modules) — модули и библиотеки от сообщества. Есть [процесс переноса модуля](/docs/4.x/guide/modules/ecosystem) в `nuxt-modules`. У модулей свои мейнтейнеры, но они не завязаны на одного человека.
* [unjs/](https://github.com/unjs) — многие эти библиотеки используются по всей экосистеме Nuxt. Они задуманы как универсальные, без привязки к фреймворку и среде. Мы приветствуем вклад и использование в других фреймворках и проектах.

## Как участвовать

### Разбор issue и помощь в обсуждениях

Смотрите issue и discussions того проекта, где хотите помочь. Например, для Nuxt — [доска issue](https://github.com/nuxt/nuxt/issues) и [обсуждения](https://github.com/nuxt/nuxt/discussions). Помощь пользователям, обходные пути, воспроизведения или небольшое расследование бага с выводами — огромная польза.

### Создание issue

Спасибо, что нашли время оформить issue! ❤️

* **Баги**: перед открытием посмотрите [наше руководство](/docs/4.x/community/reporting-bugs).

* **Запросы фич**: убедитесь, что нет подходящего issue или обсуждения. Если фича относится к другой части экосистемы (например, к модулю), сначала имеет смысл поднять вопрос там. Если идея общая или API неочевиден — откройте обсуждение в разделе **Ideas**, чтобы сначала обсудить с сообществом.

### Pull request

Мы всегда рады pull request! ❤️

#### Перед началом

Перед исправлением бага проверьте, **есть ли issue с описанием** — возможно, это недочёт документации или есть полезный контекст.

Если вы делаете фичу, **сначала откройте issue с запросом фичи** и обсудите с мейнтейнерами нужность и дизайн. Так экономится время и мейнтейнерам, и контрибьюторам, а фичи выходят быстрее. Issue **должен быть подтверждён** членом команды фреймворка, прежде чем писать большую реализацию в PR.

Опечатки лучше собирать пачкой в один PR — так история коммитов чище.

Для крупных изменений в самом Nuxt сначала рекомендуем [создать модуль Nuxt](/docs/4.x/community/contribution#create-a-module) и реализовать фичу там. Так быстрее получить proof-of-concept. Затем можно [оформить RFC](/docs/4.x/community/contribution#make-an-rfc) в виде обсуждения. По мере внедрения и обратной связи решение можно доработать и либо включить в core Nuxt, либо оставить отдельным модулем.

#### Соглашения по коммитам

Мы используем [Conventional Commits](https://www.conventionalcommits.org) для сообщений коммитов — по ним [автоматически собирается changelog](https://github.com/unjs/changelogen). Если формат незнаком, прочитайте гайд.

`fix:` и `feat:` — для **реальных изменений кода** (которые могут менять поведение). Для опечаток и правок документации — `docs:` или `chore:`:

* ~~`fix: typo`~~ → `docs: fix typo`

В монорепозитории вроде `nuxt/nuxt` указывайте основной scope в скобках. Например: `feat(kit): add 'addMagicStuff' utility`.

#### Оформление pull request

Если не знаете, как отправить PR, прочитайте [руководство GitHub](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/proposing-changes-to-your-work-with-pull-requests/creating-a-pull-request).

Заголовок PR тоже должен соответствовать [соглашениям по коммитам](/docs/4.x/community/contribution#commit-conventions).

Если PR закрывает существующие issue, обязательно укажите их в описании.

Несколько коммитов в одном PR — нормально; не обязательно делать rebase или force push: при мерже используется `Squash and Merge`.

Хуки на коммиты мы не вешаем, чтобы коммитить было быстрее. Перед PR убедитесь, что проходят lint и тесты.

В PR не должно быть _несвязанных_ изменений. Если редактор поправил пробелы или форматирование в других местах файла — откатите это, чтобы было видно суть правок. Не смешивайте в одном PR несколько несвязанных фич или фиксов: по возможности разбивайте на отдельные PR. В идеале PR делает _одну вещь_.

#### После создания PR

Мы постараемся быстро посмотреть PR.

Если назначили мейнтейнера — он уделит PR особое внимание и при необходимости внесёт правки.

Если просим изменения — не пугайтесь «красного текста»! Это не значит, что PR плохой — так удобнее видеть статус списка PR.

Метка `pending` обычно значит внутреннюю заметку: есть ещё шаг в ревью; это не обязательно про качество идеи. По возможности поясним в комментарии.

### Вклад с помощью ИИ

Мы приветствуем вдумчивое использование ИИ при участии в Nuxt и просим следовать [двум принципам](https://roe.dev/blog/using-ai-in-open-source).

#### Не позволяйте LLM говорить за вас

* Комментарии, issue и описания PR пишите своими словами
* Мы ценим ясное человеческое общение важнее идеальной грамматики
* Не копируйте «сводки» от ИИ, если они не отражают ваше понимание

#### Не позволяйте LLM думать за вас

* Использовать ИИ для кода или идей — можно
* Отправляйте только то, что сами понимаете и можете объяснить
* Вклад должен отражать вашу логику и способ решения задачи

Наша цель — качество и удовольствие от общения с людьми. Если есть идеи, как улучшить политику по ИИ в сообществе Nuxt — расскажите! ❤️

### Создание модуля

Сделали что-то крутое на Nuxt — [вынесите в модуль](/docs/4.x/guide/modules), чтобы поделиться с другими. У нас уже [много отличных модулей](/modules), но места всегда хватает.

Если нужна помощь при разработке — [напишите нам](/docs/4.x/community/getting-help).

### RFC

Сильно рекомендуем сначала [создать модуль](/docs/4.x/community/contribution#create-a-module) для крупных фич и набора пользователей.

Если это уже сделано или модуль неуместен — начните с нового обсуждения. Чётко изложите мысли, примеры кода или сигнатуры для новых API, ссылки на issue и болевые точки с примерами.

Если решим оформить RFC, сменим категорию на RFC и шире разошлём на фидбек.

Стадии RFC:

* `rfc: active` — открыт для комментариев
* `rfc: approved` — одобрено командой Nuxt
* `rfc: ready to implement` — создан issue и назначен исполнитель
* `rfc: shipped` — реализовано
* `rfc: archived` — не одобрено, но сохранено для справки

### Соглашения в экосистеме

Ниже — _обязательно_ для организации `nuxt/` и рекомендуется остальным мейнтейнерам.

#### Модули

Модули следует делать по [шаблону модуля Nuxt](https://github.com/nuxt/starter/tree/module). Подробнее — [гайд по модулям](/docs/4.x/guide/modules).

#### Библиотеки `unjs/` из core

Рекомендуемые библиотеки, используемые по экосистеме:

* [pathe](https://github.com/unjs/pathe) — универсальные пути (замена `path` в Node)
* [ufo](https://github.com/unjs/ufo) — разбор и склейка URL
* [obuild](https://github.com/unjs/obuild) — сборка на rolldown
* … остальное смотрите в организации [unjs/](https://github.com/unjs)

#### ESM и `type: module`

Большая часть экосистемы Nuxt ест ESM напрямую. Избегайте CJS-специфики вроде `__dirname` и `require`. [Подробнее про ESM](/docs/4.x/guide/concepts/esm).

#### Что такое Corepack

[Corepack](https://github.com/nodejs/corepack) подбирает нужную версию менеджера пакетов при запуске команд. В `package.json` может быть поле `packageManager`.

При конфигурации ниже Corepack поставит `pnpm` версии `v7.5.0` (если её ещё нет) и будет использовать её для команд.

```jsonc [package.json]
{
  "packageManager": "pnpm@7.5.0"
}
```

#### ESLint

Для линта и форматирования используем [ESLint](https://eslint.org) с [`@nuxt/eslint`](https://github.com/nuxt/eslint).

##### Настройка IDE

Рекомендуем [VS Code](https://code.visualstudio.com) и [расширение ESLint](https://marketplace.visualstudio.com/items?itemName=dbaeumer.vscode-eslint). При желании включите автофикс и форматирование при сохранении:

```json [settings.json]
{
  "editor.codeActionsOnSave": {
    "source.fixAll": "never",
    "source.fixAll.eslint": "explicit"
  }
}
```

#### Без Prettier

ESLint уже форматирует код — дублировать Prettier не нужно. Формат: `yarn lint --fix`, `pnpm lint --fix`, `bun run lint --fix` или `deno run lint --fix`, либо см. [раздел ESLint](/docs/4.x/community/contribution#use-eslint) для IDE.

Если Prettier стоит в редакторе — лучше отключить его в этом проекте, чтобы не было конфликтов.

#### Менеджер пакетов

Для модулей, библиотек и приложений рекомендуем `pnpm`.

Включите Corepack, чтобы версия менеджера совпадала с проектом. В новых версиях Node Corepack встроен.

Включение:

```bash [Terminal]
corepack enable
```

Достаточно один раз после установки Node.js.

## Руководство по стилю документации

Документация — важная часть Nuxt. Мы хотим быть интуитивным фреймворком, и сюда входят и DX, и качество доков по всей экосистеме. 👌

Советы по улучшению документации:

* По возможности избегайте слов вроде _просто_, _всего лишь_, _очевидно…_

  У читателей разный бэкграунд; такие слова мало что значат и могут задевать.

  ::caution{ icon="i-lucide-circle-x"}
  Просто убедитесь, что функция возвращает промис.
  ::

  ::tip{icon="i-lucide-circle-check"}
  Убедитесь, что функция возвращает [промис](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/Promise).
  ::

* Предпочитайте [активный залог](https://developers.google.com/tech-writing/one/active-voice).

  ::caution{icon="i-lucide-circle-x"}
  Ошибка будет выброшена Nuxt.

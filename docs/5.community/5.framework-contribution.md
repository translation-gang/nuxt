После [общего гайда по контрибуции](/docs/4.x/community/contribution) — несколько конкретных моментов про контрибуцию в репозиторий [`nuxt/nuxt`](https://github.com/nuxt/nuxt).

## Монорепо

- `packages/kit`: инструментарий для создания модулей Nuxt, публикуется как [`@nuxt/kit`](https://www.npmjs.com/package/@nuxt/kit).
- `packages/nuxt`: ядро Nuxt, публикуется как [`nuxt`](https://www.npmjs.com/package/nuxt).
- `packages/schema`: типы и дефолты Nuxt для разных версий, публикуется как [`@nuxt/schema`](https://www.npmjs.com/package/@nuxt/schema).
- `packages/rspack`: бандлер [Rspack](https://rspack.rs) для Nuxt, публикуется как [`@nuxt/rspack-builder`](https://www.npmjs.com/package/@nuxt/rspack-builder).
- `packages/vite`: бандлер [Vite](https://vite.dev) для Nuxt, публикуется как [`@nuxt/vite-builder`](https://www.npmjs.com/package/@nuxt/vite-builder).
- `packages/webpack`: бандлер [webpack](https://webpack.js.org) для Nuxt, публикуется как [`@nuxt/webpack-builder`](https://www.npmjs.com/package/@nuxt/webpack-builder).

## Настройка

Чтобы контрибьютить в Nuxt, нужна локальная среда.

1. [Сделайте fork](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/working-with-forks/fork-a-repo) репозитория [`nuxt/nuxt`](https://github.com/nuxt/nuxt) и [склонируйте](https://docs.github.com/en/repositories/creating-and-managing-repositories/cloning-a-repository) его локально.
2. Убедитесь, что используется актуальная [Node.js](https://nodejs.org/en).
3. Включите [Corepack](https://github.com/nodejs/corepack) для `pnpm` и `yarn`:
    ```bash [Terminal]
    corepack enable
    ```
4. Установите зависимости: `pnpm install --frozen-lockfile`
    ```bash [Terminal]
    pnpm install --frozen-lockfile
    ```
    ::note
    При добавлении зависимости используйте `pnpm add`. :br
    Файл `pnpm-lock.yaml` — источник истины для всех зависимостей Nuxt.
    ::
5. Активируйте систему пассивной разработки:
    ```bash [Terminal]
    pnpm dev:prepare
    ```
6. Создайте ветку и закоммитьте изменения:
    ```bash [Terminal]
    git checkout -b my-new-branch
    ```

Затем проверьте изменения в [playground](/docs/4.x/community/framework-contribution#playground) и [тестах](/docs/4.x/community/framework-contribution#testing) перед отправкой PR.

### Playground

При работе над PR удобно проверять изменения на примере приложения.

Можно править пример в `playground/` и запускать:

```bash [Terminal]
pnpm dev
```

::important
В ветку коммитить изменения playground не нужно, но можно добавить пример кода в описание PR — это поможет ревьюерам и пользователям Nuxt понять фичу.
::

### Тесты

У каждой новой фичи по возможности должен быть юнит-тест. Директория `test/` в репозитории в процессе развития; по возможности добавляйте тесты по аналогии с существующими.

Перед созданием PR или пометкой «ready to review» убедитесь, что все тесты проходят:

```bash [Terminal]
pnpm test
```

### Линтинг

Мы используем ESLint для единого стиля кода.

Перед коммитом проверьте стиль:

```bash [Terminal]
pnpm lint
```

::note
Можно запустить `pnpm lint --fix` для автоматического исправления большинства замечаний. :br
Оставшиеся ошибки нужно исправить вручную.
::

### Документация

При добавлении фичи, рефакторинге или изменении поведения Nuxt стоит обновить документацию. Включайте правки docs в тот же PR (не обязательно в первом коммите, но до созревания PR).

::important
Вносите изменения в соответствии с [руководством по стилю документации](/docs/4.x/community/contribution#documentation-style-guide).
::

### Чек-лист перед отправкой

При отправке PR нужно заполнить шаблон и отметить подходящие пункты в чек-листах.

## Гайд по документации

Если заметили, что можно улучшить документацию или сообщения об ошибках — откройте PR, даже если это только опечатка!

::important
Вносите изменения в соответствии с [руководством по стилю документации](/docs/4.x/community/contribution#documentation-style-guide).
::

### Быстрые правки

При опечатке или желании переформулировать фразу можно нажать **Edit this page** в боковой панели справа в разделе **Community**.

Внесите правку прямо в интерфейсе GitHub и откройте Pull Request.

### Объёмные правки

Контент документации лежит в директории `docs/` репозитория [nuxt/nuxt](https://github.com/nuxt/nuxt) и написан в markdown.

::note
Чтобы смотреть документацию локально, следуйте инструкциям в репозитории [nuxt/nuxt.com](https://github.com/nuxt/nuxt.com).
::

::note
Рекомендуем установить [расширение MDC](https://marketplace.visualstudio.com/items?itemName=Nuxt.mdc) для VS Code.
::

### Линтинг документации

Документация линтится с помощью [MarkdownLint](https://github.com/DavidAnson/markdownlint) и [case police](https://github.com/antfu/case-police) для единообразия.

```bash [Terminal]
pnpm lint:docs
```

::note
Можно запустить `pnpm lint:docs:fix` для подсветки и исправления замечаний линтера.
::

### Открытие PR

Заголовок PR должен соответствовать [conventional commits](https://www.conventionalcommits.org).

```bash [Пример заголовка PR]
docs: update the section about the nuxt.config.ts file
```

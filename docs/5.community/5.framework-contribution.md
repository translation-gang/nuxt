Прочитав [общее руководство по участию](/docs/4.x/community/contribution), обратите внимание на особенности работы с репозиторием [`nuxt/nuxt`](https://github.com/nuxt/nuxt).

## Монорепозиторий

- `packages/kit`: набор для авторов модулей Nuxt, публикуется как [`@nuxt/kit`](https://www.npmjs.com/package/@nuxt/kit).
- `packages/nuxt`: ядро Nuxt, публикуется как [`nuxt`](https://www.npmjs.com/package/nuxt).
- `packages/schema`: типы и значения по умолчанию для разных версий Nuxt, публикуется как [`@nuxt/schema`](https://www.npmjs.com/package/@nuxt/schema).
- `packages/rspack`: сборщик [Rspack](https://rspack.rs) для Nuxt, публикуется как [`@nuxt/rspack-builder`](https://www.npmjs.com/package/@nuxt/rspack-builder).
- `packages/vite`: сборщик [Vite](https://vite.dev) для Nuxt, публикуется как [`@nuxt/vite-builder`](https://www.npmjs.com/package/@nuxt/vite-builder).
- `packages/webpack`: сборщик [webpack](https://webpack.js.org) для Nuxt, публикуется как [`@nuxt/webpack-builder`](https://www.npmjs.com/package/@nuxt/webpack-builder).

## Настройка

Чтобы участвовать в разработке Nuxt, настройте локальное окружение.

1. [Сделайте форк](https://docs.github.com/en/pull-requests/collaborating-with-pull-requests/working-with-forks/fork-a-repo) репозитория [`nuxt/nuxt`](https://github.com/nuxt/nuxt) в свой аккаунт GitHub и [клонируйте](https://docs.github.com/en/repositories/creating-and-managing-repositories/cloning-a-repository) его на компьютер.
2. Используйте актуальную версию [Node.js](https://nodejs.org/en)
3. Включите [Corepack](https://github.com/nodejs/corepack), чтобы были доступны `pnpm` и `yarn`
    ```bash [Terminal]
    corepack enable
    ```
4. Выполните `pnpm install --frozen-lockfile`, чтобы установить зависимости через pnpm:
    ```bash [Terminal]
    pnpm install --frozen-lockfile
    ```
    ::note
    При добавлении зависимости используйте `pnpm add`. :br
    Файл `pnpm-lock.yaml` — единственный источник правды по версиям зависимостей Nuxt.
    ::
5. Активируйте пассивную систему разработки
    ```bash [Terminal]
    pnpm dev:prepare
    ```
6. Создайте ветку для работы и коммитов:
    ```bash [Terminal]
    git checkout -b my-new-branch
    ```

Затем проверьте изменения в [playground](/docs/4.x/community/framework-contribution#playground) и [прогоните тесты](/docs/4.x/community/framework-contribution#testing) перед отправкой pull request.

### Playground

Во время работы над PR обычно нужно убедиться, что всё работает.

Можно менять пример в `playground/` и запускать:

```bash [Terminal]
pnpm dev
```

::important
Не коммитьте изменения playground в ветку; зато полезно добавить пример кода в описание PR — так ревьюерам и пользователям проще понять фичу.
::

### Тесты

У каждой новой фичи должен быть unit-тест (если это возможно). Каталог `test/` в репозитории ещё развивается, но старайтесь писать тесты по аналогии с существующими.

Перед созданием PR или переводом в ready-to-review убедитесь, что все тесты проходят:

```bash [Terminal]
pnpm test
```

### Линтинг

Мы используем ESLint для единообразия кода.

Перед коммитом проверьте стиль:

```bash [Terminal]
pnpm lint
```

::note
Большую часть правок стиля исправит `pnpm lint --fix`. :br
Оставшиеся ошибки нужно править вручную.
::

### Документация

Если добавляете фичу, рефакторите или меняете поведение Nuxt — скорее всего, нужно обновить доки. Включайте правки документации в тот же PR. Не обязательно в первом коммите (но как только PR созрел — добавьте).

::important
Следуйте [руководству по стилю документации](/docs/4.x/community/contribution#documentation-style-guide).
::

### Финальный чеклист

При отправке PR заполните шаблон и отметьте подходящие пункты в чеклистах.

## Руководство по документации

Если видите, как улучшить документацию или тексты ошибок — открывайте PR, даже если это одна опечатка!

::important
Следуйте [руководству по стилю документации](/docs/4.x/community/contribution#documentation-style-guide).
::

### Быстрые правки

Для опечатки или переформулировки нажмите **Edit this page** справа в разделе **Community**.

Внесите правку в интерфейсе GitHub и откройте Pull Request.

### Более крупные правки

Контент документации лежит в каталоге `docs/` репозитория [nuxt/nuxt](https://github.com/nuxt/nuxt) в формате Markdown.

::note
Локальный предпросмотр доков — по инструкциям в репозитории [nuxt/nuxt.com](https://github.com/nuxt/nuxt.com).
::

::note
Рекомендуем установить [расширение MDC](https://marketplace.visualstudio.com/items?itemName=Nuxt.mdc) для VS Code.
::

### Линт документации

Документация проверяется [MarkdownLint](https://github.com/DavidAnson/markdownlint) и [case police](https://github.com/antfu/case-police) для единообразия.

```bash [Terminal]
pnpm lint:docs
```

::note
Можно запустить `pnpm lint:docs:fix`, чтобы подсветить и исправить замечания линтера.
::

### Открытие PR

Заголовок PR должен соответствовать [Conventional Commits](https://www.conventionalcommits.org).

```bash [Пример заголовка PR]
docs: update the section about the nuxt.config.ts file
```

Nuxt — full-stack-фреймворк, поэтому ошибки времени выполнения могут возникать в разных контекстах:

- В жизненном цикле рендеринга Vue (SSR и CSR)
- При старте на сервере и клиенте
- В жизненном цикле Nitro ([каталог `server/`](/docs/4.x/directory-structure/server))
- При загрузке JS-чанков

::tip
**SSR** — **Server-Side Rendering**, **CSR** — **Client-Side Rendering**.
::

## Ошибки Vue

Перехват ошибок Vue — [`onErrorCaptured`](https://vuejs.org/api/composition-api-lifecycle#onerrorcaptured).

Дополнительно Nuxt вызывает хук [`vue:error`](/docs/4.x/api/advanced/hooks#app-hooks-runtime), если ошибка доходит до верхнего уровня.

Для сервисов отчётов об ошибках задайте глобальный обработчик в [`vueApp.config.errorHandler`](https://vuejs.org/api/application#app-config-errorhandler) — туда попадут все ошибки Vue, даже обработанные.

```ts twoslash [plugins/error-handler.ts]
export default defineNuxtPlugin((nuxtApp) => {
  nuxtApp.vueApp.config.errorHandler = (error, instance, info) => {
    // handle error, e.g. report to a service
  }

  // Also possible
  nuxtApp.hook('vue:error', (error, instance, info) => {
    // handle error, e.g. report to a service
  })
})
```

::note
Хук `vue:error` опирается на [`onErrorCaptured`](https://vuejs.org/api/composition-api-lifecycle#onerrorcaptured).
::

## Ошибки при старте

При ошибках запуска приложения Nuxt вызывает хук `app:error`.

Сюда входят:
- выполнение [плагинов Nuxt](/docs/4.x/directory-structure/app/plugins)
- обработка хуков `app:created` и `app:beforeMount`
- рендер Vue в HTML (SSR)
- монтирование приложения (на клиенте) — лучше обрабатывать через `onErrorCaptured` или `vue:error`
- хук `app:mounted`

## Ошибки сервера Nitro

Отдельного серверного обработчика пока нет; можно отрисовать страницу ошибки — раздел [Страница ошибки](/docs/4.x/getting-started/error-handling#error-page).

## Ошибки загрузки JS-чанков

Сбой сети или новый деплой (старые URL с хешами чанков) дают ошибки загрузки чанков. Nuxt по умолчанию при навигации делает hard reload, если чанк не загрузился.

Поведение меняется через `experimental.emitRouteChunkError`: `false` — не перехватывать, `manual` — обрабатывать самим. Идеи — в [автоматической реализации](https://github.com/nuxt/nuxt/blob/main/packages/nuxt/src/app/plugins/chunk-reload.client.ts).

## Страница ошибки

::note
При фатальной ошибке (необработанная на сервере или `fatal: true` на клиенте) Nuxt отдаёт JSON (если запрос с `Accept: application/json`) или показывает полноэкранную страницу ошибки.
::

На сервере ошибка возможна при:
- выполнении плагинов
- рендере Vue в HTML
- выбросе ошибки в server API route

На клиенте — при:
- выполнении плагинов
- до монтирования (хук `app:beforeMount`)
- монтировании, если ошибку не поймали через `onErrorCaptured` или `vue:error`
- после инициализации и монтирования (`app:mounted`).

::read-more{to="/docs/4.x/api/advanced/hooks"}
Все хуки жизненного цикла Nuxt.
::

Своя страница ошибки — файл `~/error.vue` рядом с `app.vue` в корне исходников.

<!-- TODO:twoslash: Twoslash does not support tsconfig paths yet -->

```vue [error.vue]
<script setup lang="ts">
import type { NuxtError } from '#app'

const props = defineProps({
  error: Object as () => NuxtError,
})

const handleError = () => clearError({ redirect: '/' })
</script>

<template>
  <div>
    <h2>{{ error?.status }}</h2>
    <button @click="handleError">
      Clear errors
    </button>
  </div>
</template>
```

::read-more{to="/docs/4.x/directory-structure/app/error"}
Подробнее о `error.vue`.
::

Для своих ошибок лучше `onErrorCaptured` в setup страницы/компонента или хук `vue:error` в плагине.

```ts twoslash [plugins/error-handler.ts]
export default defineNuxtPlugin((nuxtApp) => {
  nuxtApp.hook('vue:error', (err) => {
    //
  })
})
```

Чтобы убрать страницу ошибки, вызовите [`clearError`](/docs/4.x/api/utils/clear-error) — можно передать путь редиректа на «безопасную» страницу.

::important
Перед использованием того, что зависит от плагинов (`$route`, `useRouter` и т.д.), проверьте контекст: если плагин упал, он не выполнится снова, пока ошибка не сброшена.
::

::note
Страница ошибки — отдельная загрузка; middleware выполнятся снова. В middleware можно проверить ошибку через [`useError`](/docs/4.x/getting-started/error-handling#useerror).
::

::note
На Node 16 при установке cookie при рендере страницы ошибки они могут [перезаписать ранее выставленные](https://github.com/nuxt/nuxt/pull/20585). Лучше новее Node: Node 16 EOL с сентября 2023.
::

## Утилиты ошибок

### `useError`

```ts [TS Signature]
function useError (): Ref<Error | { url, status, statusText, message, description, data }>
```

Возвращает текущую глобальную ошибку Nuxt, которую обрабатывают.

::read-more{to="/docs/4.x/api/composables/use-error"}
Подробнее о `useError`.
::

### `createError`

```ts [TS Signature]
function createError (err: string | { cause, data, message, name, stack, status, statusText, fatal }): Error
```

Создаёт объект ошибки с метаданными: строка станет `message` или передайте объект полей. Работает и во Vue, и на сервере; предполагается `throw`.

Если выбросить ошибку из `createError`:
- на сервере — полноэкранная страница ошибки, сброс через [`clearError`](/docs/4.x/getting-started/error-handling#clearerror);
- на клиенте — по умолчанию нефатальная ошибка для вашей обработки; полноэкран — с `fatal: true`.

```vue twoslash [pages/movies/[slug\\].vue]
<script setup lang="ts">
const route = useRoute()
const { data } = await useFetch(`/api/movies/${route.params.slug}`)

if (!data.value) {
  throw createError({
    status: 404,
    statusText: 'Page Not Found',
  })
}
</script>
```

::tip
`statusText` — короткий HTTP-совместимый текст (например «Not Found»): только табы, пробелы и видимые ASCII (`[\t\u0020-\u007E]`).

Подробные описания, многострочный текст и не-ASCII — в свойстве `message`.
::

::read-more{to="/docs/4.x/api/utils/create-error"}
Подробнее о `createError`.
::

### `showError`

```ts [TS Signature]
function showError (err: string | Error | { status, statusText }): Error
```

Вызов в любой момент на клиенте или на сервере в middleware, плагинах или `setup()` покажет полноэкранную ошибку; сброс — [`clearError`](/docs/4.x/getting-started/error-handling#clearerror).

Предпочтительнее `throw createError()`.

::read-more{to="/docs/4.x/api/utils/show-error"}
Подробнее о `showError`.
::

### `clearError`

```ts [TS Signature]
function clearError (options?: { redirect?: string }): Promise<void>
```

Сбрасывает текущую обрабатываемую ошибку Nuxt; опционально — редирект на «безопасную» страницу.

::read-more{to="/docs/4.x/api/utils/clear-error"}
Подробнее о `clearError`.
::

## Ошибка внутри компонента

Компонент [`<NuxtErrorBoundary>`](/docs/4.x/api/components/nuxt-error-boundary) обрабатывает клиентские ошибки внутри приложения без замены всего сайта страницей ошибки.

Ошибки в default-слоте на клиенте не всплывают наверх; рендерится слот `#error`.

Слот `#error` получает проп `error`. (`error = null` перезапустит default-слот — сначала убедитесь, что ошибка действительно устранена.)

::tip
При переходе на другой маршрут ошибка сбрасывается автоматически.
::

```vue [app/pages/index.vue]
<template>
  <!-- some content -->
  <NuxtErrorBoundary @error="someErrorLogger">
    <!-- You use the default slot to render your content -->
    <template #error="{ error, clearError }">
      You can display the error locally here: {{ error }}
      <button @click="clearError">
        This will clear the error.
      </button>
    </template>
  </NuxtErrorBoundary>
</template>
```

Полноценная поддержка unit- и e2e-тестов через `@nuxt/test-utils`: утилиты и конфигурация, на которых держатся [тесты самого Nuxt](https://github.com/nuxt/nuxt/tree/main/test) и экосистема модулей.

:video-accordion{title="Александр Лихтер: старт с @nuxt/test-utils" videoId="yGzwk9xi9gU"}

## Установка

Чтобы вы сами выбирали остальные зависимости для тестов, у `@nuxt/test-utils` опциональные peer dependencies:

- `happy-dom` или `jsdom` для runtime Nuxt в тестах
- `vitest`, `cucumber`, `jest` или `playwright` как раннеры e2e
- `playwright-core` — только если нужны встроенные браузерные утилиты (и вы не используете `@playwright/test` как раннер)

::code-group{sync="pm"}
```bash [npm]
npm i --save-dev @nuxt/test-utils vitest @vue/test-utils happy-dom playwright-core
```
```bash [yarn]
yarn add --dev @nuxt/test-utils vitest @vue/test-utils happy-dom playwright-core
```
```bash [pnpm]
pnpm add -D @nuxt/test-utils vitest @vue/test-utils happy-dom playwright-core
```
```bash [bun]
bun add --dev @nuxt/test-utils vitest @vue/test-utils happy-dom playwright-core
```
::

## Unit-тесты

Есть среда для кода, которому нужен runtime [Nuxt](https://nuxt.com). Сейчас поддерживается **только `vitest`** (PR с другими раннерами приветствуются).

### Настройка

1. Добавьте `@nuxt/test-utils/module` в `nuxt.config` (по желанию): интеграция Vitest в Nuxt DevTools и запуск unit-тестов в dev.

   ```ts twoslash
   export default defineNuxtConfig({
     modules: [
       '@nuxt/test-utils/module',
     ],
   })
   ```

2. Создайте `vitest.config.ts`:

   ```ts twoslash
   import { defineConfig } from 'vitest/config'
   import { defineVitestProject } from '@nuxt/test-utils/config'

   export default defineConfig({
     test: {
       projects: [
         {
           test: {
             name: 'unit',
             include: ['test/unit/*.{test,spec}.ts'],
             environment: 'node',
           },
         },
         {
           test: {
             name: 'e2e',
             include: ['test/e2e/*.{test,spec}.ts'],
             environment: 'node',
           },
         },
         await defineVitestProject({
           test: {
             name: 'nuxt',
             include: ['test/nuxt/*.{test,spec}.ts'],
             environment: 'nuxt',
           },
         }),
       ],
     },
   })
   ```

::tip
При импорте `@nuxt/test-utils` в конфиге Vitest в `package.json` нужен `"type": "module"` или расширение вроде `vitest.config.m{ts,js}`.
::

::tip
Переменные для тестов можно задать в `.env.test`.
::

### Runtime Nuxt в тестах

[Vitest projects](https://vitest.dev/guide/projects.html#test-projects) задают, в какой среде что запускается:

- **Unit**: обычные тесты в `test/unit/` — среда Node, быстрее
- **Nuxt**: тесты, зависящие от runtime Nuxt, в `test/nuxt/`

#### Упрощённый вариант

Все тесты в среде Nuxt — базовый конфиг:

```ts twoslash
import { defineVitestConfig } from '@nuxt/test-utils/config'
import { fileURLToPath } from 'node:url'

export default defineVitestConfig({
  test: {
    environment: 'nuxt',
    // you can optionally set Nuxt-specific environment options
    // environmentOptions: {
    //   nuxt: {
    //     rootDir: fileURLToPath(new URL('./playground', import.meta.url)),
    //     domEnvironment: 'happy-dom', // 'happy-dom' (default) or 'jsdom'
    //     overrides: {
    //       // other Nuxt config you want to pass
    //     }
    //   }
    // }
  },
})
```

При `environment: 'nuxt'` по умолчанию для отдельного файла можно [выйти из среды Nuxt](https://vitest.dev/guide/environment.html#test-environment).

```ts twoslash
// @vitest-environment node
import { test } from 'vitest'

test('my test', () => {
  // ... test without Nuxt environment!
})
```

::warning
Не рекомендуется: гибрид, где крутятся Vite-плагины Nuxt, но нет entry Nuxt и `nuxtApp` — сложно отлаживать.
::

### Структура тестов

Пример раскладки с проектами:

```bash [Directory structure]
test/
├── e2e/
│   └── ssr.test.ts
├── nuxt/
│   ├── components.test.ts
│   └── composables.test.ts
├── unit/
│   └── utils.test.ts
```

Структура может быть любой, но отделяйте runtime Nuxt от e2e — так стабильнее.

#### TypeScript в тестах

По умолчанию файлы в `test/nuxt/` и `tests/nuxt/` попадают в [TS-контекст приложения Nuxt](/docs/4.x/guide/concepts/typescript#how-nuxt-uses-project-references): алиасы `~/`, `@/`, `#imports` и auto-imports.

::tip
Соответствует рекомендации: в эти папки — только тесты, которым нужен runtime Nuxt. Unit в `test/unit/` при необходимости подключайте вручную.
::

##### Другие каталоги тестов

Если тесты в других путях идут в Nuxt Vitest, добавьте их в TS-контекст:

<!-- @case-police-ignore tsConfig -->

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  typescript: {
    tsConfig: {
      include: [
        // this path is relative to the generated .nuxt/tsconfig.json
        '../test/other-nuxt-context/**/*',
      ],
    },
  },
})
```

::important
Unit-тесты не должны зависеть от auto-imports и композаблов Nuxt. Алиасы путей добавляйте, если импортируете свои модули (`~/utils/helpers`), а не ради фич Nuxt.
::

#### Запуск

С проектами Vitest:

```bash
# Run all tests
npx vitest

# Run only unit tests
npx vitest --project unit

# Run only Nuxt tests
npx vitest --project nuxt

# Run tests in watch mode
npx vitest --watch
```

::warning
В среде Nuxt тесты идут в [`happy-dom`](https://github.com/capricorn86/happy-dom) или [`jsdom`](https://github.com/jsdom/jsdom). Перед запуском поднимается глобальное приложение Nuxt (плагины, код из `app.vue`).

Не мутируйте глобальное состояние без необходимости; если нужно — сбрасывайте после теста.
::

### 🎭 Встроенные моки

Моки DOM в `@nuxt/test-utils`.

#### `intersectionObserver`

По умолчанию `true` — заглушка без логики для IntersectionObserver

#### `indexedDB`

По умолчанию `false`; [`fake-indexeddb`](https://github.com/dumbmatter/fakeIndexedDB) — рабочий мок IndexedDB

Настройка в `environmentOptions` в `vitest.config.ts`:

```ts twoslash
import { defineVitestConfig } from '@nuxt/test-utils/config'

export default defineVitestConfig({
  test: {
    environmentOptions: {
      nuxt: {
        mock: {
          intersectionObserver: true,
          indexedDb: true,
        },
      },
    },
  },
})
```

### 🛠️ Хелперы

`@nuxt/test-utils` упрощает тесты Nuxt-приложений.

#### `mountSuspended`

Монтирует Vue-компонент в среде Nuxt: async setup и инъекции из плагинов.

::note
Внутри — обёртка над `mount` из `@vue/test-utils`; опции см. в [документации Vue Test Utils](https://test-utils.vuejs.org/guide/).
::

Пример:

```ts twoslash
// @noErrors
import { expect, it } from 'vitest'
import type { Component } from 'vue'

declare module '#components' {
  export const SomeComponent: Component
}
// ---cut---
// tests/components/SomeComponents.nuxt.spec.ts
import { mountSuspended } from '@nuxt/test-utils/runtime'
import { SomeComponent } from '#components'

it('can mount some component', async () => {
  const component = await mountSuspended(SomeComponent)
  expect(component.text()).toMatchInlineSnapshot(
    '"This is an auto-imported component"',
  )
})
```

```ts twoslash
// @noErrors
import { expect, it } from 'vitest'
// ---cut---
// tests/components/SomeComponents.nuxt.spec.ts
import { mountSuspended } from '@nuxt/test-utils/runtime'
import App from '~/app.vue'

// tests/App.nuxt.spec.ts
it('can also mount an app', async () => {
  const component = await mountSuspended(App, { route: '/test' })
  expect(component.html()).toMatchInlineSnapshot(`
      "<div>This is an auto-imported component</div>
      <div> I am a global component </div>
      <div>/</div>
      <a href="/test"> Test link </a>"
    `)
})
```

Объект опций — опции `mount` из `@vue/test-utils` плюс:

  - `route`: начальный маршрут или `false`, чтобы не менять маршрут (по умолчанию `/`).

#### `renderSuspended`

Рендер через `@testing-library/vue` в среде Nuxt — async setup и инъекции плагинов.

Вместе с `screen`, `fireEvent` и т.д. Установите [@testing-library/vue](https://testing-library.com/docs/vue-testing-library/intro/).

Testing Library для cleanup использует глобали тестов — включите в [конфиге Vitest](https://vitest.dev/config/globals).

Компонент рендерится внутри `<div id="test-wrapper"></div>`.

Примеры:

```ts twoslash
// @noErrors
import { expect, it } from 'vitest'
import type { Component } from 'vue'

declare module '#components' {
  export const SomeComponent: Component
}
// ---cut---
// tests/components/SomeComponents.nuxt.spec.ts
import { renderSuspended } from '@nuxt/test-utils/runtime'
import { SomeComponent } from '#components'
import { screen } from '@testing-library/vue'

it('can render some component', async () => {
  await renderSuspended(SomeComponent)
  expect(screen.getByText('This is an auto-imported component')).toBeDefined()
})
```

```ts twoslash
// @noErrors
import { expect, it } from 'vitest'
// ---cut---
// tests/App.nuxt.spec.ts
import { renderSuspended } from '@nuxt/test-utils/runtime'
import App from '~/app.vue'

it('can also render an app', async () => {
  const html = await renderSuspended(App, { route: '/test' })
  expect(html).toMatchInlineSnapshot(`
    "<div id="test-wrapper">
      <div>This is an auto-imported component</div>
      <div> I am a global component </div>
      <div>Index page</div><a href="/test"> Test link </a>
    </div>"
  `)
})
```

Опции рендера `@testing-library/vue` плюс:

  - `route`: начальный маршрут или `false` (по умолчанию `/`).

#### `mockNuxtImport`

Мок auto-imports Nuxt. Например `useState`:

```ts twoslash
import { mockNuxtImport } from '@nuxt/test-utils/runtime'

mockNuxtImport('useState', () => {
  return () => {
    return { value: 'mocked storage' }
  }
})

// your tests here
```

Можно типизировать мок и в фабрике использовать оригинальную реализацию для сложных случаев.

```ts twoslash [test/nuxt/import.test.ts]
import { mockNuxtImport } from '@nuxt/test-utils/runtime'

mockNuxtImport<typeof useState>('useState', (original) => {
  return (...args) => {
    return { ...original('some-key'), value: 'mocked state' }
  }
})

// or specify the target to mock
mockNuxtImport(useState, (original) => {
  return (...args) => {
    return { ...original('some-key'), value: 'mocked state' }
  }
})

// your tests here
```

::note
`mockNuxtImport` — один раз на импорт в файле: это макрос → `vi.mock` с hoisting, см. [Vitest](https://vitest.dev/api/vi#vi-mock).
::

Разные реализации между тестами — через [`vi.hoisted`](https://vitest.dev/api/vi#vi-hoisted) и затем `mockNuxtImport`. Между тестами [восстанавливайте моки](https://vitest.dev/api/mock#mockrestore).

```ts twoslash
import { vi } from 'vitest'
import { mockNuxtImport } from '@nuxt/test-utils/runtime'

const { useStateMock } = vi.hoisted(() => {
  return {
    useStateMock: vi.fn(() => {
      return { value: 'mocked storage' }
    }),
  }
})

mockNuxtImport('useState', () => {
  return useStateMock
})

// Then, inside a test
useStateMock.mockImplementation(() => {
  return { value: 'something else' }
})
```

Мок только внутри одного теста — так:

```ts twoslash
import { beforeEach, vi } from 'vitest'
import { mockNuxtImport } from '@nuxt/test-utils/runtime'

mockNuxtImport(useRoute, original => vi.fn(original))

beforeEach(() => {
  vi.resetAllMocks()
})

// Then, inside a test
const useRouteOriginal = vi.mocked(useRoute).getMockImplementation()!
vi.mocked(useRoute).mockImplementation(
  (...args) => ({ ...useRouteOriginal(...args), path: '/mocked' }),
)
```

#### `mockComponent`

Мок компонента Nuxt.
Первый аргумент — имя в PascalCase или относительный путь.
Второй — фабрика, возвращающая мок-компонент.

Пример для `MyComponent`:

```ts twoslash
import { mockComponent } from '@nuxt/test-utils/runtime'

mockComponent('MyComponent', {
  props: {
    value: String,
  },
  setup (props) {
    // ...
  },
})

// relative path or alias also works
mockComponent('~/components/my-component.vue', () => {
  // or a factory function
  return defineComponent({
    setup (props) {
      // ...
    },
  })
})

// or you can use SFC for redirecting to a mock component
mockComponent('MyComponent', () => import('./MockComponent.vue'))

// your tests here
```

> **Замечание**: локальные переменные в фабрике недоступны из-за hoisting. Vue API и прочее — импортируйте внутри фабрики.

```ts twoslash
import { mockComponent } from '@nuxt/test-utils/runtime'

mockComponent('MyComponent', async () => {
  const { ref, h } = await import('vue')

  return defineComponent({
    setup (props) {
      const counter = ref(0)
      return () => h('div', null, counter.value)
    },
  })
})
```

#### `registerEndpoint`

Создаёт эндпоинт Nitro с мок-данными — удобно для компонентов, тянущих API.

Первый аргумент — путь (например `/test/`).
Второй — фабрика ответа.

Пример `/test/`:

```ts twoslash
import { registerEndpoint } from '@nuxt/test-utils/runtime'

registerEndpoint('/test/', () => ({
  test: 'test-field',
}))
```

По умолчанию `GET`; другой метод — второй аргумент как объект.

```ts twoslash
import { registerEndpoint } from '@nuxt/test-utils/runtime'

registerEndpoint('/test/', {
  method: 'POST',
  handler: () => ({ test: 'test-field' }),
})
```

Свойства объекта:

- `handler`: обработчик
- `method`: (опц.) HTTP-метод ('GET', 'POST' и т.д.)
- `once`: (опц.) `true` — сработает один раз и удалится

> **Замечание**: для внешнего API можно обнулить `baseURL` через [переопределение конфига по окружению](/docs/4.x/getting-started/configuration#environment-overrides) (`$test`), чтобы запросы шли на Nitro.

#### Конфликт с e2e

`@nuxt/test-utils/runtime` и `@nuxt/test-utils/e2e` требуют разных сред — не в одном файле.

Используйте оба — разнесите файлы: комментарий `// @vitest-environment nuxt` или суффикс `.nuxt.spec.ts` для runtime unit.

`app.nuxt.spec.ts`

```ts twoslash
import { mockNuxtImport } from '@nuxt/test-utils/runtime'

mockNuxtImport('useState', () => {
  return () => {
    return { value: 'mocked storage' }
  }
})
```

`app.e2e.spec.ts`

```ts twoslash
import { $fetch, setup } from '@nuxt/test-utils/e2e'

await setup({
  setupTimeout: 10000,
})

// ...
```

## E2E-тесты

Раннеры: [Vitest](https://github.com/vitest-dev/vitest), [Jest](https://jestjs.io), [Cucumber](https://cucumber.io/), [Playwright](https://playwright.dev/).

### Настройка

В каждом `describe` с хелперами `@nuxt/test-utils/e2e` сначала вызовите `setup` для контекста.

```ts twoslash [test/my-test.spec.ts]
import { describe, test } from 'vitest'
import { $fetch, setup } from '@nuxt/test-utils/e2e'

describe('My test', async () => {
  await setup({
    // test context options
  })

  test('my test', () => {
    // ...
  })
})
```

`setup` в `beforeAll` / `beforeEach` / `afterEach` / `afterAll` поднимает среду Nuxt для тестов.

Опции `setup`:

#### Конфиг Nuxt

- `rootDir`: каталог приложения под тест
  - Type: `string`
  - Default: `'.'`
- `configFile`: имя файла конфигурации
  - Type: `string`
  - Default: `'nuxt.config'`

<!--
- `config`: Object with configuration overrides.
  - Type: `NuxtConfig`
  - Default: `{}` -->

#### Тайминги

- `setupTimeout`: мс на завершение `setupTest` (сборка/генерация и т.д.)
  - Type: `number`
  - Default: `120000` или `240000` на Windows

- `teardownTimeout`: мс на разборку среды (закрытие браузера и т.п.)
  - Type: `number`
  - Default: `30000`

#### Поведение

- `build`: отдельный шаг сборки
  - Type: `boolean`
  - Default: `true` (`false` если выключены `browser` или `server`, или задан `host`)

- `server`: поднять сервер для запросов в suite
  - Type: `boolean`
  - Default: `true` (`false` при `host`)

- `port`: порт тестового сервера
  - Type: `number | undefined`
  - Default: `undefined`

- `host`: URL цели вместо сборки и запуска сервера — «настоящие» e2e против деплоя или уже запущенного локального сервера (часто быстрее). См. [пример с host](/docs/4.x/getting-started/testing#target-host-end-to-end-example).
  - Type: `string`
  - Default: `undefined`

- `browser`: браузерные тесты через [`playwright`](https://playwright.dev)
  - Type: `boolean`
  - Default: `false`
- `browserOptions`
  - Type: `object`
    - `type`: `chromium`, `firefox` или `webkit`
    - `launch`: опции запуска Playwright, см. [API](https://playwright.dev/docs/api/class-browsertype#browser-type-launch).
- `runner`: раннер suite; рекомендуется [Vitest](https://vitest.dev)
  - Type: `'vitest' | 'jest' | 'cucumber'`
  - Default: `'vitest'`

##### Пример e2e с `host` {#target-host-end-to-end-example}

Частый сценарий — тесты против задеплоенного приложения в прод-подобной среде.

Локально или в CI отдельный уже запущенный сервер часто эффективнее, чем пересборка между тестами.

Передайте в `setup` свойство `host` с нужным URL.

```ts
import { createPage, setup } from '@nuxt/test-utils/e2e'
import { describe, expect, it } from 'vitest'

describe('login page', async () => {
  await setup({
    host: 'http://localhost:8787',
  })

  it('displays the email and password fields', async () => {
    const page = await createPage('/login')
    expect(await page.getByTestId('email').isVisible()).toBe(true)
    expect(await page.getByTestId('password').isVisible()).toBe(true)
  })
})
```

### API

#### `$fetch(url)`

HTML SSR-страницы.

```ts twoslash
import { $fetch } from '@nuxt/test-utils/e2e'

const html = await $fetch('/')
```

#### `fetch(url)`

Ответ SSR-страницы целиком.

```ts twoslash
import { fetch } from '@nuxt/test-utils/e2e'

const res = await fetch('/')
const { body, headers } = res
```

#### `url(path)`

Полный URL страницы (с портом тестового сервера).

```ts twoslash
import { url } from '@nuxt/test-utils/e2e'

const pageUrl = url('/page')
// 'http://localhost:6840/page'
```

### Браузер

Playwright в `@nuxt/test-utils`: программно или через Playwright Test.

#### `createPage(url)`

В `vitest`, `jest` или `cucumber` — `createPage` даёт настроенный `Page`; опционально путь на запущенном сервере. Методы — в [документации Playwright](https://playwright.dev/docs/api/class-page).

```ts twoslash
import { createPage } from '@nuxt/test-utils/e2e'

const page = await createPage('/page')
// you can access all the Playwright APIs from the `page` variable
```

#### Playwright Test как раннер

Полноценная интеграция с [Playwright Test](https://playwright.dev/docs/intro).

::code-group{sync="pm"}
```bash [npm]
npm i --save-dev @playwright/test @nuxt/test-utils
```
```bash [yarn]
yarn add --dev @playwright/test @nuxt/test-utils
```
```bash [pnpm]
pnpm add -D @playwright/test @nuxt/test-utils
```
```bash [bun]
bun add --dev @playwright/test @nuxt/test-utils
```
```bash [deno]
deno add --dev npm:@playwright/test npm:@nuxt/test-utils
```
::

Глобальный конфиг Nuxt — те же поля, что у `setup()` выше.

```ts [playwright.config.ts]
import { fileURLToPath } from 'node:url'
import { defineConfig, devices } from '@playwright/test'
import type { ConfigOptions } from '@nuxt/test-utils/playwright'

export default defineConfig<ConfigOptions>({
  use: {
    nuxt: {
      rootDir: fileURLToPath(new URL('.', import.meta.url)),
    },
  },
  // ...
})
```

::read-more{title="Полный пример конфига" to="https://github.com/nuxt/test-utils/blob/main/examples/app-playwright/playwright.config.ts" target="_blank"}
::

В тестах импортируйте `expect` и `test` из `@nuxt/test-utils/playwright`:

```ts [tests/example.test.ts]
import { expect, test } from '@nuxt/test-utils/playwright'

test('test', async ({ page, goto }) => {
  await goto('/', { waitUntil: 'hydration' })
  await expect(page.getByRole('heading')).toHaveText('Welcome to Playwright!')
})
```

Или задайте сервер Nuxt в самом тестовом файле:

```ts [tests/example.test.ts]
import { expect, test } from '@nuxt/test-utils/playwright'

test.use({
  nuxt: {
    rootDir: fileURLToPath(new URL('..', import.meta.url)),
  },
})

test('test', async ({ page, goto }) => {
  await goto('/', { waitUntil: 'hydration' })
  await expect(page.getByRole('heading')).toHaveText('Welcome to Playwright!')
})
```

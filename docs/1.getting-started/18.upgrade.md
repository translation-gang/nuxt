## Обновление Nuxt

### Последний релиз

Чтобы перейти на [последний релиз](https://github.com/nuxt/nuxt/releases), используйте `nuxt upgrade`.

::code-group{sync="pm"}

```bash [npm]
npx nuxt upgrade
```

```bash [yarn]
yarn nuxt upgrade
```

```bash [pnpm]
pnpm nuxt upgrade
```

```bash [bun]
bun x nuxt upgrade
```

```bash [deno]
deno x nuxt upgrade
```

::

### Канал nightly

Чтобы собирать свежий Nuxt и пробовать фичи до релиза, см. [канал nightly](/docs/4.x/guide/going-further/nightly-release-channel).

## Проба Nuxt 5 {#testing-nuxt-5}

Nuxt 5 **в разработке**. До релиза часть breaking changes можно включить из Nuxt 4.2+.

### Включение поведения Nuxt 5

Сначала обновитесь до [последнего релиза](https://github.com/nuxt/nuxt/releases).

Затем задайте `future.compatibilityVersion` под Nuxt 5:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  future: {
    compatibilityVersion: 5,
  },
})
```

При `future.compatibilityVersion: 5` меняются дефолты конфигурации под Nuxt v5, в том числе:

- **Vite Environment API**: новая [Vite Environment API](/docs/4.x/getting-started/upgrade#migration-to-vite-environment-api) для сборки
- **Имена страниц**: имена компонентов страниц [совпадают с именами маршрутов](/docs/4.x/getting-started/upgrade#normalized-page-component-names) — предсказуемый `<KeepAlive>`
- **`clearNuxtState` и дефолты**: `clearNuxtState` [сбрасывает к начальному значению](/docs/4.x/getting-started/upgrade#respect-defaults-when-clearing-usestate), а не в `undefined`
- **Не-async `callHook`**: [`callHook` может вернуть `void`](/docs/4.x/getting-started/upgrade#non-async-callhook) вместо всегда `Promise`
- прочие изменения по мере появления

::note
Раздел может меняться до финального релиза; периодически перечитывайте, если тестируете Nuxt 5 с `future.compatibilityVersion: 5`.
::

Ниже — breaking и важные изменения с шагами миграции.

### Миграция на Vite Environment API {#migration-to-vite-environment-api}

🚦 **Уровень влияния**: средний

#### Что изменилось

Nuxt 5 переходит на [Environment API](https://vite.dev/guide/api-environment) Vite 6: явные окружения и более точная настройка на каждое из них.

Раньше были отдельные конфиги Vite для client и server. Теперь — общий конфиг и плагины с `applyToEnvironment()` для выбора окружения.

::note
Vite Environment API в Nuxt 5 всегда включён. Опция `experimental.viteEnvironmentApi` удалена.
::

**Ключевые изменения:**

1. **Устарели `server`/`client` в `extendViteConfig()`** — будут предупреждения.

2. **Регистрация плагинов**: плагины из `addVitePlugin()` только под одно окружение (`server: false` или `client: false`) не получат вызовов хуков `config` и `configResolved`.

3. **Общий конфиг**: хуки `vite:extendConfig` и `vite:configResolved` работают с одним общим конфигом, а не с раздельными client/server.

#### Причины изменения

Environment API даёт:
- единообразие dev и prod;
- более тонкую настройку по окружениям;
- лучшую архитектуру плагинов и производительность;
- кастомные окружения помимо client/server.

#### Шаги миграции

##### 1. Перейти на Vite-плагины

Рекомендуем Vite-плагин вместо `extendViteConfig`, `vite:configResolved` и `vite:extendConfig`.

```ts
// Было
extendViteConfig((config) => {
  config.optimizeDeps.include.push('my-package')
}, { server: false })

nuxt.hook('vite:extendConfig' /* или vite:configResolved */, (config, { isClient }) => {
  if (isClient) {
    config.optimizeDeps.include.push('my-package')
  }
})

// Стало
addVitePlugin(() => ({
  name: 'my-plugin',
  config (config) {
    // глобальная конфигурация Vite
  },
  configResolved (config) {
    // полностью резолвнутый конфиг Vite
  },
  configEnvironment (name, config) {
    // настройки под конкретное окружение
    if (name === 'client') {
      config.optimizeDeps ||= {}
      config.optimizeDeps.include ||= []
      config.optimizeDeps.include.push('my-package')
    }
  },
  applyToEnvironment (environment) {
    return environment.name === 'client'
  },
}))
```

##### 2. Плагины Vite и окружения

Вместо `addVitePlugin` с `server: false` или `client: false` используйте хук `applyToEnvironment` в плагине.

```ts
// Было
addVitePlugin(() => ({
  name: 'my-plugin',
  config (config) {
    config.optimizeDeps.include.push('my-package')
  },
}), { client: false })

// Стало
addVitePlugin(() => ({
  name: 'my-plugin',
  config (config) {
    // глобальная конфигурация Vite
  },
  configResolved (config) {
    // полностью резолвнутый конфиг Vite
  },
  configEnvironment (name, config) {
    // настройки под конкретное окружение
    if (name === 'client') {
      config.optimizeDeps ||= {}
      config.optimizeDeps.include ||= []
      config.optimizeDeps.include.push('my-package')
    }
  },
  applyToEnvironment (environment) {
    return environment.name === 'client'
  },
}))
```

### Миграция на Vite 8

🚦 **Уровень влияния**: средний

#### Что изменилось

Nuxt 5 поднимает Vite с 7 до [Vite 8](https://main.vite.dev/guide/migration): вместо esbuild и Rollup на нижнем уровне — [Rolldown](https://rolldown.rs). Сборки быстрее, но есть breaking changes.

::note
В отличие от миграции на Vite Environment API, это нельзя рано включить через `future.compatibilityVersion: 5`. Для проверки совместимости с Vite 8 заранее добавьте в `package.json` resolution, например `"vite": "^8.0.0-beta.15"`.
::

Большую часть миграции делает Nuxt, но есть заметные изменения для приложения:

- **`vite.esbuild` и `vite.optimizeDeps.esbuildOptions`** уступают место `vite.oxc` и `vite.optimizeDeps.rolldownOptions`. Vite 8 пока конвертирует старые поля, позже их уберут.
- **`build.rollupOptions`** → `build.rolldownOptions`.
- **Поведение CommonJS interop** изменилось; при импорте CJS см. [гайд по миграции Vite 8](https://main.vite.dev/guide/migration#consistent-commonjs-interop).

::read-more{to="https://main.vite.dev/guide/migration" target="_blank"}
Полный гайд по миграции Vite 8
::

### Не-async `callHook` {#non-async-callhook}

🚦 **Уровень влияния**: минимальный

#### Что изменилось

После перехода на [hookable v6](https://github.com/unjs/hookable) `callHook` может вернуть `void`, а не всегда `Promise<void>`: меньше лишних `Promise`, если хуков нет или все синхронны.

По умолчанию при `compatibilityVersion: 4` Nuxt оборачивает `callHook` в `Promise.resolve()`, цепочки `.then()`/`.catch()` продолжают работать. При `compatibilityVersion: 5` обёртка снимается.

::tip
Затрагивает и build-time хуки (модули Nuxt), и runtime-хуки в коде приложения.
::

#### Причины изменения

В hookable v6 `callHook` примерно в 20–40 раз быстрее за счёт отказа от `Promise`, когда он не нужен — выгодно при частых вызовах хуков.

#### Шаги миграции

Если вы или модули цепляете `.then()`/`.catch()` к `callHook`, перейдите на `await`:

```diff
- nuxtApp.callHook('my:hook', data).then(() => { ... })
+ await nuxtApp.callHook('my:hook', data)
```

```diff
- nuxtApp.hooks.callHook('my:hook', data).catch(err => { ... })
+ try { await nuxtApp.hooks.callHook('my:hook', data) } catch (err) { ... }
```

::tip
Проверить поведение заранее: `future.compatibilityVersion: 5` (см. [Проба Nuxt 5](/docs/4.x/getting-started/upgrade#testing-nuxt-5)) или явно `experimental.asyncCallHook: false`.
::

Либо принудительно держите `Promise` у `callHook`:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    asyncCallHook: true,
  },
})
```

### Новая структура каталогов

🚦 **Уровень влияния**: значительный

По умолчанию — новая структура каталогов; при обнаружении старой (например `app/pages/` в корне) новая схема не применяется — обратная совместимость.

👉 [Полный RFC](https://github.com/nuxt/nuxt/issues/26444)

#### Что изменилось

- новый дефолт Nuxt для `srcDir` — каталог `app/`, большинство путей резолвится оттуда;
- `serverDir` по умолчанию — `<rootDir>/server`, а не `<srcDir>/server`;
- `layers/`, `modules/` и `public/` по умолчанию резолвятся относительно `<rootDir>`;
- при использовании [Nuxt Content v2.13+](https://github.com/nuxt/content/pull/2649) каталог `content/` резолвится относительно `<rootDir>`;
- добавлен `dir.app` — каталог, где ищутся `router.options.ts` и `spa-loading-template.html`; по умолчанию это `<srcDir>/`;
- новый каталог [`shared/`](/docs/4.x/directory-structure/shared) для кода, общего между Vue-приложением и сервером Nitro, с автоимпортом из `shared/utils/` и `shared/types/`.

<details>

<summary>Пример структуры папок в v4.</summary>

```sh
.output/
.nuxt/
app/
  assets/
  components/
  composables/
  layouts/
  middleware/
  pages/
  plugins/
  utils/
  app.config.ts
  app.vue
  router.options.ts
content/
layers/
modules/
node_modules/
public/
shared/
  types/
  utils/
server/
  api/
  middleware/
  plugins/
  routes/
  utils/
nuxt.config.ts
```

::note
При такой структуре алиас `~` по умолчанию указывает на каталог `app/` (ваш `srcDir`). То есть `~/components` → `app/components/`, `~/pages` → `app/pages/` и т.д.
::

</details>

👉 Подробнее — в [PR с реализацией](https://github.com/nuxt/nuxt/pull/27029).

#### Причины изменения

1. **Производительность** — если весь код лежит в корне репозитория, вотчеры ФС часто сканируют `.git/` и `node_modules/`, что на системах кроме macOS сильно замедляет старт.
1. **Типобезопасность в IDE** — `server/` и остальное приложение работают в разных контекстах с разным набором глобальных импортов; вынести `server/` из той же папки, что и приложение — важный шаг к нормальным автодополнениям в IDE.

:video-accordion{title="Видео Vue School о новой структуре каталогов" videoId="1031028378" platform="vimeo"}

#### Шаги миграции

1. Создайте каталог `app/`.
1. Перенесите в него `assets/`, `components/`, `composables/`, `layouts/`, `middleware/`, `pages/`, `plugins/` и `utils/`, а также `app.vue`, `error.vue`, `app.config.ts`. Если у вас `router.options.ts` или `spa-loading-template.html` в `app/`, пути остаются теми же.
1. Оставьте `nuxt.config.ts`, `content/`, `layers/`, `modules/`, `public/`, `shared/` и `server/` **вне** `app/`, в корне проекта.
1. Обновите сторонние конфиги под новую структуру (например Tailwind или ESLint при необходимости; `@nuxtjs/tailwindcss` обычно настраивает Tailwind сам).

::tip
Миграцию можно автоматизировать: `npx codemod@latest nuxt/4/file-structure`
::

Миграция **не обязательна**. Если хотите сохранить текущую структуру папок, Nuxt должен определить её автоматически. (Если нет — создайте issue.) Исключение: если у вас уже задан свой `srcDir` — тогда `modules/`, `public/`, `shared/` и `server/` резолвятся от `rootDir`, а не от вашего `srcDir`. При необходимости переопределите через `dir.modules`, `dir.public` и `serverDir`.

Структуру как в v3 можно принудительно задать так:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  // Возвращает новый дефолт srcDir с `app` обратно в корень проекта
  srcDir: '.',
  // Префикс каталога для `router.options.ts` и `spa-loading-template.html`
  dir: {
    app: 'app',
  },
})
```

### Единый слой загрузки данных

🚦 **Уровень влияния**: умеренный

#### Что изменилось

Система загрузки данных Nuxt (`useAsyncData` и `useFetch`) существенно перестроена ради производительности и согласованности:

1. **Общие ref для одного ключа**: все вызовы `useAsyncData` или `useFetch` с одним ключом разделяют одни и те же ref `data`, `error` и `status`. Важно, чтобы при явном ключе не конфликтовали опции `deep`, `transform`, `pick`, `getCachedData` или `default`.

2. **Больше контроля над `getCachedData`**: `getCachedData` вызывается при каждой загрузке данных, в том числе из-за watcher или `refreshNuxtData`. (Раньше в этих случаях всегда тянули новые данные и эта функция не вызывалась.) Чтобы гибко выбирать кэш и повторный запрос, в функцию передаётся объект контекста с причиной запроса.

3. **Реактивные ключи**: ключом могут быть computed, обычный ref или геттер — данные автоматически перезапрашиваются и хранятся раздельно.

4. **Очистка данных**: когда размонтируется последний компонент, использовавший данные из `useAsyncData`, Nuxt удаляет их, чтобы не раздувать память.

#### Причины изменения

Изменения снижают потребление памяти и выравнивают поведение состояний загрузки между вызовами `useAsyncData`.

#### Шаги миграции

1. **Проверьте несовместимые опции**: просмотрите компоненты с одним ключом, но разными опциями или функциями fetch.

   ```ts
   // Теперь будет предупреждение
   const { data: users1 } = useAsyncData('users', () => $fetch('/api/users'), { deep: false })
   const { data: users2 } = useAsyncData('users', () => $fetch('/api/users'), { deep: true })
   ```

   Имеет смысл вынести вызовы `useAsyncData` с общим явным ключом (и своими опциями) в отдельный композабл:

   ```ts [app/composables/useUserData.ts]
   export function useUserData (userId: string) {
     return useAsyncData(
       `user-${userId}`,
       () => fetchUser(userId),
       {
         deep: true,
         transform: user => ({ ...user, lastAccessed: new Date() }),
       },
     )
   }
   ```

2. **Обновите реализации `getCachedData`**:

   ```diff
   useAsyncData('key', fetchFunction, {
   -  getCachedData: (key, nuxtApp) => {
   -    return cachedData[key]
   -  }
   +  getCachedData: (key, nuxtApp, ctx) => {
   +    // ctx.cause — 'initial' | 'refresh:hook' | 'refresh:manual' | 'watch'
   +
   +    // Пример: не брать кэш при ручном refresh
   +    if (ctx.cause === 'refresh:manual') return undefined
   +
   +    return cachedData[key]
   +  }
   })
   ```

Пока можно отключить это поведение так:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    granularCachedData: false,
    purgeCachedData: false,
  },
})
```

### Порядок загрузки модулей в layers

🚦 **Уровень влияния**: минимальный

#### Что изменилось

Порядок загрузки модулей при использовании [Nuxt layers](/docs/4.x/guide/going-further/layers) исправлен. Раньше модули из корня проекта загружались **до** модулей из расширяемых слоёв — наоборот ожидаемому.

Теперь порядок такой:

1. **Сначала модули слоёв** (в порядке `extends` — сначала более «глубокие» слои)
2. **Затем модули проекта** (наивысший приоритет)

Это касается:
- модулей из массива `modules` в `nuxt.config.ts`;
- автообнаруженных модулей в каталоге `modules/`.

#### Причины изменения

Так достигается, что:
- расширяемые слои имеют меньший приоритет, чем потребляющий проект;
- порядок выполнения модулей соответствует интуитивному наследованию слоёв;
- конфигурация и хуки модулей ведут себя предсказуемо при нескольких слоях.

#### Шаги миграции

**Большинству проектов ничего менять не нужно** — порядок просто приводится к ожидаемому.

Если же вы полагались на **старый** (неверный) порядок:

1. **Проверьте зависимости модулей** — не требуют ли они конкретного порядка загрузки.
2. **Подстройте конфигурацию модулей**, если она обходила старый баг.
3. **Тщательно протестируйте** работу с новым порядком.

Пример корректного порядка:
```ts
// Слой: my-layer/nuxt.config.ts
export default defineNuxtConfig({
  modules: ['layer-module-1', 'layer-module-2'],
})

// Проект: nuxt.config.ts
export default defineNuxtConfig({
  extends: ['./my-layer'],
  modules: ['project-module-1', 'project-module-2'],
})

// Порядок загрузки (исправленный):
// 1. layer-module-1
// 2. layer-module-2
// 3. project-module-1 (может переопределить модули слоя)
// 4. project-module-2 (может переопределить модули слоя)
```

Если порядок модулей мешает зарегистрировать хук, используйте хук [`modules:done`](/docs/4.x/guide/modules/recipes-advanced) — он выполняется после загрузки всех остальных модулей.

👉 Подробнее: [PR #31507](https://github.com/nuxt/nuxt/pull/31507) и [issue #25719](https://github.com/nuxt/nuxt/issues/25719).

### Дедупликация метаданных маршрутов

🚦 **Уровень влияния**: минимальный

#### Что изменилось

Часть метаданных маршрута задаётся через `definePageMeta` (`name`, `path` и т.д.). Раньше они дублировались и на объекте маршрута, и в `route.meta` (например `route.name` и `route.meta.name`).

Теперь они доступны только на объекте маршрута.

#### Причины изменения

Следствие включения по умолчанию `experimental.scanPageMeta` и оптимизации производительности.

#### Шаги миграции

Миграция обычно простая:

```diff
  const route = useRoute()

- console.log(route.meta.name)
+ console.log(route.name)
```

### Нормализованные имена компонентов

🚦 **Уровень влияния**: умеренный

Vue теперь генерирует имена компонентов в соответствии с соглашением Nuxt об именовании.

#### Что изменилось

По умолчанию, если имя не задано вручную, Vue присваивает компоненту имя по **имени файла**.

```bash [Directory structure]
├─ components/
├─── SomeFolder/
├───── MyComponent.vue
```

Для Vue имя компонента в этом случае — `MyComponent`. Для `<KeepAlive>` или Vue DevTools нужно было бы использовать именно его.

А для автоимпорта — `SomeFolderMyComponent`.

После изменения эти два значения **совпадают**: имя компонента Vue совпадает с паттерном именования Nuxt.

#### Шаги миграции

Обновите имена в тестах с `findComponent` из `@vue/test-utils` и в `<KeepAlive>`, если они завязаны на имя компонента.

Пока можно отключить поведение так:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    normalizeComponentNames: false,
  },
})
```

### Unhead v2

🚦 **Уровень влияния**: минимальный

#### Что изменилось

[Unhead](https://unhead.unjs.io/) (генерация тегов `<head>`) обновлён до версии 2. В целом совместим, но в низкоуровневых API есть breaking changes.

- Удалены пропсы: `vmid`, `hid`, `children`, `body`.
- Вход в виде `Promise` больше не поддерживается.
- Теги по умолчанию сортируются через Capo.js.

#### Шаги миграции

Для большинства приложений влияние минимально.

При проблемах проверьте:

- что не используются удалённые пропсы;

```diff
useHead({
  meta: [{
    name: 'description',
    // у meta-тегов не нужны vmid и key
-   vmid: 'description'
-   hid: 'description'
  }]
})
```

- при использовании [Template Params](https://unhead.unjs.io/docs/head/guides/plugins/template-params) или [Alias Tag Sorting](https://unhead.unjs.io/docs/head/guides/plugins/alias-sorting) нужно явно подключить эти возможности;

```ts
import { AliasSortingPlugin, TemplateParamsPlugin } from '@unhead/vue/plugins'

export default defineNuxtPlugin({
  setup () {
    const unhead = injectHead()
    unhead.use(TemplateParamsPlugin)
    unhead.use(AliasSortingPlugin)
  },
})
```

Необязательно, но рекомендуется заменить импорты из `@unhead/vue` на `#imports` или `nuxt/app`.

```diff
-import { useHead } from '@unhead/vue'
+import { useHead } from '#imports'
```

При сохранении проблем можно вернуться к поведению v1, включив `unhead.legacy` в конфиге.

```ts
export default defineNuxtConfig({
  unhead: {
    legacy: true,
  },
})
```

### Новое место в DOM для SPA loading screen

🚦 **Уровень влияния**: минимальный

#### Что изменилось

Для клиентской страницы (`ssr: false`) по желанию показывается экран загрузки (из `~/app/spa-loading-template.html` — в Nuxt 4 путь стал `~/spa-loading-template.html`) **внутри** корня приложения Nuxt:

```html
<div id="__nuxt">
  <!-- spa loading template -->
</div>
```

Теперь по умолчанию шаблон рендерится **рядом** с корнем приложения:

```html
<div id="__nuxt"></div>
<!-- spa loading template -->
```

#### Причины изменения

Шаблон загрузки остаётся в DOM до завершения suspense Vue-приложения, без вспышки белого экрана.

#### Шаги миграции

Если вы нацеливали стили или `document.querySelector` на шаблон загрузки, обновите селекторы. Для этого есть опции `app.spaLoaderTag` и `app.spaLoaderAttrs`.

Вернуть прежнее поведение можно так:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    spaLoadingTemplateLocation: 'within',
  },
})
```

### Разбор `error.data`

🚦 **Уровень влияния**: минимальный

Раньше при ошибке с полем `data` оно не разбиралось. Теперь разбирается и попадает в объект `error`. Это исправление, но технически breaking change, если вы сами парсили `data` по-старому.

#### Шаги миграции

В кастомном `error.vue` уберите лишний разбор `error.data`:

```diff
  <script setup lang="ts">
  import type { NuxtError } from '#app'

  const props = defineProps({
    error: Object as () => NuxtError
  })

- const data = JSON.parse(error.data)
+ const data = error.data
  </script>
```

### Более точное инлайнирование стилей

🚦 **Уровень влияния**: умеренный

Nuxt теперь инлайнит стили только у Vue-компонентов, а не весь глобальный CSS.

#### Что изменилось

Раньше Nuxt инлайнил весь CSS, включая глобальные стили, и убирал `<link>` на отдельные файлы. Теперь так делается только для стилей Vue-компонентов (которые раньше попадали в отдельные чанки). Это компромисс: меньше лишних запросов при первой загрузке (как и раньше — без отдельных запросов на каждый `.css` на страницу/компонент), плюс можно кэшировать один глобальный CSS и уменьшить размер начального HTML.

#### Шаги миграции

Поведение настраивается: чтобы снова инлайнить и глобальный CSS, задайте `inlineStyles: true`.

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  features: {
    inlineStyles: true,
  },
})
```

### Сканирование page meta после резолва

🚦 **Уровень влияния**: минимальный

#### Что изменилось

Метаданные страниц (из `definePageMeta`) теперь сканируются **после** хука `pages:extend`, а не до него.

#### Причины изменения

Чтобы учитывать страницы, добавленные в `pages:extend`. Переопределить метаданные по-прежнему можно в хуке `pages:resolved`.

#### Шаги миграции

Переопределяйте метаданные страниц в `pages:resolved`, а не в `pages:extend`.

```diff
  export default defineNuxtConfig({
    hooks: {
-     'pages:extend'(pages) {
+     'pages:resolved'(pages) {
        const myPage = pages.find(page => page.path === '/')
        myPage.meta ||= {}
        myPage.meta.layout = 'overridden-layout'
      }
    }
  })
```

Вернуть прежнее поведение можно так:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    scanPageMeta: true,
  },
})
```

### Общие данные пререндера

🚦 **Уровень влияния**: средний

#### Что изменилось

Включена ранее экспериментальная возможность: общие данные из вызовов `useAsyncData` и `useFetch` между разными страницами. См. [исходный PR](https://github.com/nuxt/nuxt/pull/24894).

#### Причины изменения

Payload **данных** автоматически шарится между пререндеримыми страницами — заметный выигрыш, если на сайте в `useAsyncData`/`useFetch` на разных страницах запрашиваются одни и те же данные.

Например, если на каждой странице нужен `useFetch` (меню, настройки из CMS и т.д.), при пререндере запрос выполнится один раз для первой страницы, затем результат переиспользуется.

#### Шаги миграции

Убедитесь, что уникальный ключ всегда однозначно соответствует одним и тем же данным. Для данных, зависящих от страницы, задайте ключ явно (`useFetch` обычно делает это сам).

```ts [app/pages/test/[slug\\].vue]
// Небезопасно на динамической странице (например `[slug].vue`): от slug зависят данные,
// но Nuxt этого не видит, если ключ это не отражает.
const route = useRoute()
const { data } = await useAsyncData(async () => {
  return await $fetch(`/api/my-page/${route.params.slug}`)
})
// Нужен ключ, однозначно идентифицирующий загружаемые данные.
const { data } = await useAsyncData(route.params.slug, async () => {
  return await $fetch(`/api/my-page/${route.params.slug}`)
})
```

Отключить функцию можно так:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    sharedPrerenderData: false,
  },
})
```

### Значения по умолчанию `data` и `error` в `useAsyncData` и `useFetch`

🚦 **Уровень влияния**: минимальный

#### Что изменилось

Объекты `data` и `error` из `useAsyncData` по умолчанию теперь `undefined`.

#### Причины изменения

Раньше `data` инициализировался как `null`, а в `clearNuxtData` сбрасывался в `undefined`; `error` был `null`. Изменение выравнивает поведение.

#### Шаги миграции

Если вы проверяли `data.value` или `error.value` на `null`, замените проверки на `undefined`.

::tip
Шаг можно автоматизировать: `npx codemod@latest nuxt/4/default-data-error-value`
::

### Удаление устаревших `boolean` у `dedupe` при вызове `refresh` в `useAsyncData` и `useFetch`

🚦 **Уровень влияния**: минимальный

#### Что изменилось

Раньше в `refresh` можно было передать `dedupe: boolean` — это были синонимы `cancel` (`true`) и `defer` (`false`).

```ts twoslash [app/app.vue]
// @errors: 2322
const { refresh } = await useAsyncData(() => Promise.resolve({ message: 'Hello, Nuxt!' }))

async function refreshData () {
  await refresh({ dedupe: true })
}
```

#### Причины изменения

Синонимы убраны ради ясности.

При добавлении `dedupe` в опции `useAsyncData` выяснилось, что булевы значения в `refresh` и в опциях означали **противоположное**.

`refresh({ dedupe: false })` — **не отменять** уже идущие запросы в пользу нового. А `dedupe: true` в опциях `useAsyncData` — **не делать новый запрос**, если уже есть ожидающий. (См. [PR](https://github.com/nuxt/nuxt/pull/24564#pullrequestreview-1764584361).)

#### Шаги миграции

Миграция обычно тривиальна:

```diff
  const { refresh } = await useAsyncData(async () => ({ message: 'Hello, Nuxt 3!' }))

  async function refreshData () {
-   await refresh({ dedupe: true })
+   await refresh({ dedupe: 'cancel' })

-   await refresh({ dedupe: false })
+   await refresh({ dedupe: 'defer' })
  }
```

::tip
Автоматизация: `npx codemod@latest nuxt/4/deprecated-dedupe-value`
::

### Дефолты при сбросе `data` в `useAsyncData` и `useFetch`

🚦 **Уровень влияния**: минимальный

#### Что изменилось

Если задан свой `default` для `useAsyncData`, при `clear` / `clearNuxtData` значение сбрасывается к этому дефолту, а не просто «снимается».

#### Причины изменения

Часто задают пустой массив и т.п., чтобы не проверять `null`/`undefined` при обходе — такое значение нужно сохранять и при сбросе.

### Дефолты при сбросе `useState` {#respect-defaults-when-clearing-usestate}

🚦 **Уровень влияния**: минимальный

#### Что изменилось

При `compatibilityVersion: 5` `clearNuxtState` сбрасывает состояние к начальному значению из `init` в `useState`, а не в `undefined` — как уже делает `clearNuxtData` для данных.

#### Причины изменения

Если после `clearNuxtState` состояние становится `undefined`, зависящие композаблы могут падать (обращение к полям у `undefined`). Сброс к значению `init` сохраняет допустимую форму.

#### Шаги миграции

Если вам нужен именно `undefined` после очистки, передайте `{ reset: false }`:

```diff
- clearNuxtState('myKey')
+ clearNuxtState('myKey', { reset: false })
```

Вернуть старое поведение:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    defaults: {
      useState: {
        resetOnClear: false,
      },
    },
  },
})
```

Включить новое поведение уже сейчас, без `compatibilityVersion: 5`:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    defaults: {
      useState: {
        resetOnClear: true,
      },
    },
  },
})
```

### Согласованность `pending` в `useAsyncData` и `useFetch`

🚦 **Уровень влияния**: средний

`pending` из `useAsyncData`, `useFetch`, `useLazyAsyncData` и `useLazyFetch` теперь вычисляемое свойство: `true` только когда `status` тоже в состоянии ожидания.

#### Что изменилось

При `immediate: false` `pending` будет `false` до первого запроса. Раньше до первого запроса `pending` всегда был `true`.

#### Причины изменения

Смысл `pending` совпадает с `status`: оба отражают «запрос в процессе».

#### Шаги миграции

Если логика завязана на `pending`, учтите: `pending === true` только когда статус тоже pending.

```diff
  <template>
-   <div v-if="!pending">
+   <div v-if="status === 'success'">
      <p>Data: {{ data }}</p>
    </div>
    <div v-else>
      <p>Загрузка…</p>
    </div>
  </template>
  <script setup lang="ts">
  const { data, pending, execute, status } = await useAsyncData(() => fetch('/api/data'), {
    immediate: false
  })
  onMounted(() => execute())
  </script>
```

Временно вернуть старое поведение:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    pendingWhenIdle: true,
  },
})
```

### Поведение при смене ключа в `useAsyncData` и `useFetch`

🚦 **Уровень влияния**: средний

#### Что изменилось

При реактивном ключе Nuxt перезапрашивает данные при его смене. При `immediate: false` повтор при смене ключа происходит **только если** данные уже хотя бы раз загружались.

Раньше у `useFetch` было иначе: при смене ключа запрос шёл всегда.

Теперь `useFetch` и `useAsyncData` ведут себя одинаково: при смене ключа — только после первой успешной загрузки.

#### Причины изменения

Единое поведение и меньше неожиданных запросов. При `immediate: false` без `refresh`/`execute` данные так и не подтянутся.

#### Шаги миграции

Обычно это ожидаемее; если вы рассчитывали, что смена ключа у не immediate `useFetch` сама запустит первый запрос — вызовите его вручную.

```diff
  const id = ref('123')
  const { data, execute } = await useFetch('/api/test', {
    query: { id },
    immediate: false
  )
+ watch(id, () => execute(), { once: true })
```

Отключить новое поведение:

```ts
// Или глобально в nuxt.config
export default defineNuxtConfig({
  experimental: {
    alwaysRunFetchOnKeyChange: true,
  },
})
```

### Shallow-реактивность данных в `useAsyncData` и `useFetch`

🚦 **Уровень влияния**: минимальный

`data` из `useAsyncData`, `useFetch`, `useLazyAsyncData` и `useLazyFetch` теперь `shallowRef`, а не глубокий `ref`.

#### Что изменилось

После нового fetch зависимости от `data` обновятся (ссылка меняется). Изменение **внутренних** полей без замены объекта реактивность не триггерит.

#### Причины изменения

Заметный прирост производительности для глубоко вложенных структур: Vue не отслеживает каждое поле. В типичном случае `data` лучше считать неизменяемым.

#### Шаги миграции

Часто ничего менять не нужно. Если нужна глубокая реактивность:

1. Включайте на вызов:
   ```diff
   - const { data } = useFetch('/api/test')
   + const { data } = useFetch('/api/test', { deep: true })
   ```
1. Или задайте дефолт для проекта (не рекомендуется):
   ```ts twoslash [nuxt.config.ts]
   export default defineNuxtConfig({
     experimental: {
       defaults: {
         useAsyncData: {
           deep: true,
         },
       },
     },
   })
   ```

::tip
Автоматизация: `npx codemod@latest nuxt/4/shallow-function-reactivity`
::

### Абсолютные пути в `builder:watch`

🚦 **Уровень влияния**: минимальный

#### Что изменилось

Хук `builder:watch` теперь передаёт **абсолютный** путь, а не относительный к `srcDir`.

#### Причины изменения

Можно следить за путями вне `srcDir` и надёжнее работать с layers и сложными схемами.

#### Шаги миграции

Публичные модули Nuxt уже обновлены. См. [issue #25339](https://github.com/nuxt/nuxt/issues/25339).

Авторам модулей, нужна совместимость с v3 и v4, можно нормализовать путь так:

```diff
+ import { relative, resolve } from 'node:fs'
  // ...
  nuxt.hook('builder:watch', async (event, path) => {
+   path = relative(nuxt.options.srcDir, resolve(nuxt.options.srcDir, path))
    // ...
  })
```

::tip
Автоматизация: `npx codemod@latest nuxt/4/absolute-watch-path`
::

### Сканирование index в каталогах

🚦 **Уровень влияния**: средний

#### Что изменилось

Вложенные папки в `app/middleware/` тоже сканируются на `index`-файлы; они регистрируются как middleware.

#### Причины изменения

Nuxt автоматически сканирует ряд каталогов, в т.ч. `app/middleware/` и `app/plugins/`.

Для `app/plugins/` вложенные папки с `index` уже учитывались; поведение выровняли для остальных сканируемых директорий.

#### Шаги миграции

Обычно ничего менять не нужно. Чтобы вернуть старое поведение, отфильтруйте такие middleware хуком:

```ts
export default defineNuxtConfig({
  hooks: {
    'app:resolve' (app) {
      app.middleware = app.middleware.filter(mw => !/\/index\.[^/]+$/.test(mw.path))
    },
  },
})
```

### Изменения компиляции шаблонов

🚦 **Уровень влияния**: минимальный

#### Что изменилось

Раньше Nuxt компилировал шаблоны с диска в формате `.ejs` через `lodash/template`.

Также были утилиты для шаблонов (`serialize`, `importName`, `importSources`) для кодогенерации — они удаляются.

#### Причины изменения

В Nuxt 3 перешли на «виртуальный» синтаксис с `getContents()` — гибче и быстрее.

У `lodash/template` были серии уязвимостей; для Nuxt это в основном build-time и доверенный код, но аудиты всё равно ругаются. Плюс тяжёлый `lodash` почти никому не нужен.

Сериализацию кода лучше не тащить в ядро Nuxt — есть пакеты вроде [unjs/knitwork](https://github.com/unjs/knitwork) как зависимость вашего проекта, с отдельным циклом исправлений без апгрейда Nuxt.

#### Шаги миграции

Модули с EJS мы обновляли PR-ами; если делаете сами, есть три совместимых варианта:

- Перенести интерполяцию строк прямо в `getContents()`.
- Своя функция замены, как в https://github.com/nuxt-modules/color-mode/pull/240.
- `es-toolkit/compat` (замена lodash template) как **ваша** зависимость, не Nuxt:

```diff
+ import { readFileSync } from 'node:fs'
+ import { template } from 'es-toolkit/compat'
  // ...
  addTemplate({
    fileName: 'appinsights-vue.js'
    options: { /* some options */ },
-   src: resolver.resolve('./runtime/plugin.ejs'),
+   getContents({ options }) {
+     const contents = readFileSync(resolver.resolve('./runtime/plugin.ejs'), 'utf-8')
+     return template(contents)({ options })
+   },
  })
```

Если используете утилиты шаблонов (`serialize`, `importName`, `importSources`), замените их на аналоги из `knitwork`:

```ts
import { genDynamicImport, genImport, genSafeVariableName } from 'knitwork'

const serialize = (data: any) => JSON.stringify(data, null, 2).replace(/"\{(.+)\}"(?=,?$)/gm, r => JSON.parse(r).replace(/^\{(.*)\}$/, '$1'))

const importSources = (sources: string | string[], { lazy = false } = {}) => {
  return toArray(sources).map((src) => {
    if (lazy) {
      return `const ${genSafeVariableName(src)} = ${genDynamicImport(src, { comment: `webpackChunkName: ${JSON.stringify(src)}` })}`
    }
    return genImport(src, genSafeVariableName(src))
  }).join('\n')
}

const importName = genSafeVariableName
```

::tip
Автоматизация: `npx codemod@latest nuxt/4/template-compilation-changes`
::

### Изменения дефолтного TypeScript-конфига

🚦 **Уровень влияния**: минимальный

#### Что изменилось

`compilerOptions.noUncheckedIndexedAccess` теперь `true` вместо `false`.

#### Причины изменения

Продолжение [обновления дефолтов в 3.12](https://github.com/nuxt/nuxt/pull/27485), в духе [рекомендаций TotalTypeScript](https://www.totaltypescript.com/tsconfig-cheat-sheet).

#### Шаги миграции

Два пути:

1. Прогнать typecheck и поправить новые ошибки (рекомендуется).

2. Переопределить дефолт в `nuxt.config.ts`:

   <!-- @case-police-ignore tsConfig -->

   ```ts
   export default defineNuxtConfig({
     typescript: {
       tsConfig: {
         compilerOptions: {
           noUncheckedIndexedAccess: false,
         },
       },
     },
   })
   ```

### Разделение TypeScript-конфигурации

🚦 **Уровень влияния**: минимальный

#### Что изменилось

Nuxt генерирует отдельные TS-конфиги под разные контексты для более точной проверки типов:

1. **Новые файлы**: дополнительно создаются:
   - `.nuxt/tsconfig.app.json` — код приложения (Vue, композаблы и т.д.)
   - `.nuxt/tsconfig.server.json` — сервер (Nitro/`server/`)
   - `.nuxt/tsconfig.node.json` — build-time (модули, `nuxt.config.ts` и т.д.)
   - `.nuxt/tsconfig.shared.json` — общий код между app и server (типы, утилиты без привязки к среде)
   - `.nuxt/tsconfig.json` — прежний вариант для обратной совместимости

2. **Обратная совместимость**: проекты с `extends` на `.nuxt/tsconfig.json` работают как раньше.

3. **Опционально — project references**: новые проекты или желающие строже проверку могут включить ссылки между конфигами.

4. **Проверка по контексту**: у каждого контекста свои `compilerOptions`, include/exclude.

5. **Опция `typescript.nodeTsConfig`**: настройка TS для Node/build-time.

#### Причины изменения

1. **Точнее типы**: у app, server и build-time свои глобалы и API.
2. **Удобнее IDE**: IntelliSense и ошибки ближе к реальному контексту файла.
3. **Разделение**: сервер не «подсказывает» клиентские API и наоборот.
4. **Скорость**: проверка по сфокусированным конфигам эффективнее.

Например, auto-imports недоступны в `nuxt.config.ts` — раньше TypeScript это не подсвечивал. Отдельный `tsconfig` в `server/` IDE понимали, но единый typecheck этого не отражал без доп. шагов.

#### Шаги миграции

**Менять ничего не обязательно** — старые проекты продолжают работать.

Чтобы использовать улучшенную проверку, подключите project references:

1. **Корневой `tsconfig.json`** — ссылки на сгенерированные конфиги:

   ::note
   Если сейчас есть `"extends": "./.nuxt/tsconfig.json"`, **удалите** эту строку перед добавлением `references`. `extends` и project references вместе не используются.
   ::

   ```json
   {
     // Удалите "extends": "./.nuxt/tsconfig.json", если была такая строка
     "files": [],
     "references": [
       { "path": "./.nuxt/tsconfig.app.json" },
       { "path": "./.nuxt/tsconfig.server.json" },
       { "path": "./.nuxt/tsconfig.shared.json" },
       { "path": "./.nuxt/tsconfig.node.json" }
     ]
   }
   ```

2. **Удалите** ручные `server/tsconfig.json`, которые только расширяли `.nuxt/tsconfig.server.json`.

3. **Скрипты typecheck** — флаг сборки для project references:

   ```diff
   - "typecheck": "nuxt prepare && vue-tsc --noEmit"
   + "typecheck": "nuxt prepare && vue-tsc -b --noEmit"
   ```

4. **Аугментации типов** — в нужный контекст:
   - app — каталог `app/`
   - server — `server/`
   - **общие** для app и server — `shared/`

    ::warning
    Аугментации вне `app/`, `server/` и `shared/` с новой схемой project references не подхватятся.
    ::

5. **Опции TypeScript** при необходимости:
   <!-- @case-police-ignore tsConfig -->

   ```ts
   export default defineNuxtConfig({
     typescript: {
       // настройка tsconfig.app.json
       tsConfig: {
         // ...
       },
       // настройка tsconfig.shared.json
       sharedTsConfig: {
         // ...
       },
       // настройка tsconfig.node.json
       nodeTsConfig: {
         // ...
       },
     },
     nitro: {
       typescript: {
         // настройка tsconfig.server.json
         tsConfig: {
           // ...
         },
       },
     },
   })
   ```

6. **CI/скрипты сборки**, где гоняется проверка TS, переведите на project references, если перешли на неё.

Новая схема даёт лучше типы и IntelliSense тем, кто подключился; остальные остаются на полной обратной совместимости.

### Удаление экспериментальных фич

🚦 **Уровень влияния**: минимальный

#### Что изменилось

В Nuxt 4 больше нельзя настраивать ряд экспериментальных флагов:

- `experimental.treeshakeClientOnly` — всегда `true` (как по умолчанию с v3.0)
- `experimental.configSchema` — всегда `true` (с v3.3)
- `experimental.polyfillVueUseHead` — всегда `false` (с v3.4)
- `experimental.respectNoSSRHeader` — всегда `false` (с v3.4)
- `experimental.viteEnvironmentApi` — всегда включён, отключить нельзя

#### Причины изменения

Значения давно зафиксированы; смысла держать их переключаемыми нет.

#### Шаги миграции

- `polyfillVueUseHead` — в пользовательском коде, например [этот плагин](https://github.com/nuxt/nuxt/blob/f209158352b09d1986aa320e29ff36353b91c358/packages/nuxt/src/head/runtime/plugins/vueuse-head-polyfill.ts#L10-L11)

- `respectNoSSRHeader` — в пользовательском коде через [server middleware](https://github.com/nuxt/nuxt/blob/c660b39447f0d5b8790c0826092638d321cd6821/packages/nuxt/src/core/runtime/nitro/no-ssr.ts#L8-L9)

### Удаление top-level опции `generate`

🚦 **Уровень влияния**: минимальный

#### Что изменилось

В Nuxt 4 нет верхнеуровневой опции `generate` и всех её полей:

- `generate.exclude` — исключение маршрутов из пререндера
- `generate.routes` — явный список маршрутов для пререндера

#### Причины изменения

`generate` на верхнем уровне — наследие Nuxt 2. Для пререндера давно рекомендуется `nitro.prerender` (Nuxt 3+).

#### Шаги миграции

Замените `generate` на соответствующие опции `nitro.prerender`:

```diff
export default defineNuxtConfig({
- generate: {
-   exclude: ['/admin', '/private'],
-   routes: ['/sitemap.xml', '/robots.txt']
- }
+ nitro: {
+   prerender: {
+     ignore: ['/admin', '/private'],
+     routes: ['/sitemap.xml', '/robots.txt']
+   }
+ }
})
```

Опции пререндера в Nitro.

### Нормализованные имена компонентов страниц {#normalized-page-component-names}

🚦 **Уровень влияния**: минимальный

#### Что изменилось

При `future.compatibilityVersion: 5` (или `experimental.normalizePageNames`) имя компонента страницы совпадает с **именем маршрута**, а не с именем файла. Например, у `pages/foo/index.vue` имя компонента `foo`, а не `index`.

#### Причины изменения

Раньше Vue брал имя из файла: и `pages/foo/index.vue`, и `pages/bar/index.vue` были `index` — `<KeepAlive>` с `include`/`exclude` вёл себя непредсказуемо, приходилось вешать `defineOptions({ name: '...' })` на каждую страницу.

#### Шаги миграции

Если вы опираетесь на старые имена (списки `include`/`exclude` у `<KeepAlive>`), замените их на **имена маршрутов**, а не файлов.

```diff
<template>
  <NuxtPage :keepalive="{
-   include: ['index']
+   include: ['foo']
  }" />
</template>
```

Отключить поведение:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    normalizePageNames: false,
  },
})
```

В Nuxt для загрузки данных в браузере и на сервере есть два композабла и встроенная обёртка: `useFetch`, [`useAsyncData`](/docs/4.x/api/composables/use-async-data) и `$fetch`.

Кратко:

- [`$fetch`](/docs/4.x/api/utils/dollarfetch) — простейший способ сетевого запроса.
- [`useFetch`](/docs/4.x/api/composables/use-fetch) — обёртка над `$fetch`: данные запрашиваются один раз при [универсальном рендеринге](/docs/4.x/guide/concepts/rendering#universal-rendering).
- [`useAsyncData`](/docs/4.x/api/composables/use-async-data) — похож на `useFetch`, но с более тонким контролем.

У `useFetch` и `useAsyncData` общие опции и паттерны — они разобраны в конце страницы.

## Зачем нужны `useFetch` и `useAsyncData`

Nuxt выполняет изоморфный (универсальный) код и на сервере, и на клиенте. Если в `setup` компонента вызывать только [`$fetch`](/docs/4.x/api/utils/dollarfetch), запрос может выполниться дважды: на сервере (для HTML) и снова на клиенте (при гидратации). Это грозит проблемами гидратации, лишней задержкой до интерактивности и нестабильным поведением.

Композаблы [`useFetch`](/docs/4.x/api/composables/use-fetch) и [`useAsyncData`](/docs/4.x/api/composables/use-async-data) передают результат серверного запроса клиенту через **payload**.

Payload — JavaScript-объект из [`useNuxtApp().payload`](/docs/4.x/api/composables/use-nuxt-app#payload). На клиенте он позволяет не запрашивать те же данные повторно при [гидратации](/docs/4.x/guide/concepts/rendering#universal-rendering).

::tip
В [Nuxt DevTools](https://devtools.nuxt.com) эти данные смотрите на вкладке **Payload**.
::

```vue [app/app.vue]
<script setup lang="ts">
const { data } = await useFetch('/api/data')

async function handleFormSubmit () {
  const res = await $fetch('/api/submit', {
    method: 'POST',
    body: {
      // My form data
    },
  })
}
</script>

<template>
  <div v-if="data == undefined">
    No data
  </div>
  <div v-else>
    <form @submit="handleFormSubmit">
      <!-- form input tags -->
    </form>
  </div>
</template>
```

В примере `useFetch` гарантирует запрос на сервере и передачу данных в браузер. У `$fetch` такого механизма нет — его логичнее использовать для запросов только с клиента (по событиям).

### Suspense

Под капотом Nuxt использует [`<Suspense>`](https://vuejs.org/guide/built-ins/suspense) Vue, чтобы не завершать навигацию, пока асинхронные данные для представления не готовы. Композаблы загрузки данных помогают использовать это поведение и настраивать его под каждый вызов.

::note
Можно добавить [`<NuxtLoadingIndicator>`](/docs/4.x/api/components/nuxt-loading-indicator) — индикатор прогресса между переходами страниц.
::

## `$fetch`

В Nuxt встроена библиотека [ofetch](https://github.com/unjs/ofetch); глобально доступен алиас `$fetch`.

```vue twoslash [pages/todos.vue]
<script setup lang="ts">
async function addTodo () {
  const todo = await $fetch('/api/todos', {
    method: 'POST',
    body: {
      // My todo data
    },
  })
}
</script>
```

::warning
Только `$fetch` не даёт [дедупликации запросов и блокировки навигации](/docs/4.x/getting-started/data-fetching#the-need-for-usefetch-and-useasyncdata). :br
Рекомендуется `$fetch` для клиентских действий (по событиям) или вместе с [`useAsyncData`](/docs/4.x/getting-started/data-fetching#useasyncdata) для начальных данных компонента.
::

### Проброс клиентских заголовков к API

При вызове `useFetch` на сервере Nuxt через [`useRequestFetch`](/docs/4.x/api/composables/use-request-fetch) проксирует заголовки и cookie клиента (кроме тех, что не следует пересылать, например `host`).

```vue
<script setup lang="ts">
const { data } = await useFetch('/api/echo')
</script>
```

```ts
// /api/echo.ts
export default defineEventHandler(event => parseCookies(event))
```

Ниже — как через [`useRequestHeaders`](/docs/4.x/api/composables/use-request-headers) передать cookie в API при серверном запросе (инициированном с клиента). Изоморфный `$fetch` даёт эндпоинту тот же заголовок `cookie`, что и у браузера. Нужно только если не используете `useFetch`.

```vue
<script setup lang="ts">
const headers = useRequestHeaders(['cookie'])

async function getCurrentUser () {
  return await $fetch('/api/me', { headers })
}
</script>
```

::tip
Можно использовать [`useRequestFetch`](/docs/4.x/api/composables/use-request-fetch) — заголовки пробросятся автоматически.
::

::caution
Осторожно с проксированием заголовков во внешние API: пересылайте только нужные. Не все заголовки безопасно пробрасывать. Типичные заголовки, которые **не** стоит проксировать:

- `host`, `accept`
- `content-length`, `content-md5`, `content-type`
- `x-forwarded-host`, `x-forwarded-port`, `x-forwarded-proto`
- `cf-connecting-ip`, `cf-ray`
::

## `useFetch`

Композабл [`useFetch`](/docs/4.x/api/composables/use-fetch) внутри использует `$fetch` и делает SSR-безопасные запросы из `setup`.

```vue twoslash [app/app.vue]
<script setup lang="ts">
const { data: count } = await useFetch('/api/count')
</script>

<template>
  <p>Page visits: {{ count }}</p>
</template>
```

Это обёртка над [`useAsyncData`](/docs/4.x/api/composables/use-async-data) и утилитой `$fetch`.

## `useAsyncData`

Композабл `useAsyncData` оборачивает асинхронную логику и возвращает результат после завершения.

::tip
`useFetch(url)` почти эквивалентен `useAsyncData(url, () => event.$fetch(url))`. :br
Это синтаксический сахар для самого частого случая. (Подробнее про `event.fetch` — [`useRequestFetch`](/docs/4.x/api/composables/use-request-fetch).)
::

:video-accordion{title="Видео Александра Лихтера: разница между useFetch и useAsyncData" videoId="0X-aOpSGabA"}

Иногда [`useFetch`](/docs/4.x/api/composables/use-fetch) не подходит — например, когда CMS или сторонний слой даёт свой API запросов. Тогда оборачивайте вызовы в [`useAsyncData`](/docs/4.x/api/composables/use-async-data), сохраняя преимущества композабла.

```vue [app/pages/users.vue]
<script setup lang="ts">
const { data, error } = await useAsyncData('users', () => myGetFunction('users'))

// This is also possible:
const { data, error } = await useAsyncData(() => myGetFunction('users'))
</script>
```

::note
Первый аргумент [`useAsyncData`](/docs/4.x/api/composables/use-async-data) — уникальный ключ кэша для ответа второго аргумента (функции запроса). Ключ можно не передавать, указав сразу функцию — тогда ключ сгенерируется автоматически.
:br :br
Автоключ учитывает только файл и строку вызова `useAsyncData`, поэтому лучше задавать свой ключ, чтобы избежать сюрпризов — например в своём композабле вокруг `useAsyncData`.
:br :br
Явный ключ полезен для общих данных между компонентами через [`useNuxtData`](/docs/4.x/api/composables/use-nuxt-data) и для [точечного обновления данных](/docs/4.x/api/utils/refresh-nuxt-data#refresh-specific-data).
::

```vue [app/pages/users/[id\\].vue]
<script setup lang="ts">
const { id } = useRoute().params

const { data, error } = await useAsyncData(`user:${id}`, () => {
  return myGetFunction('users', { id })
})
</script>
```

`useAsyncData` удобен, чтобы дождаться нескольких `$fetch` и обработать результат вместе.

```vue
<script setup lang="ts">
const { data: discounts, status } = await useAsyncData('cart-discount', async (_nuxtApp, { signal }) => {
  const [coupons, offers] = await Promise.all([
    $fetch('/cart/coupons', { signal }),
    $fetch('/cart/offers', { signal }),
  ])

  return { coupons, offers }
})
// discounts.value.coupons
// discounts.value.offers
</script>
```

::note
`useAsyncData` — для загрузки и кэширования данных, а не для побочных эффектов вроде вызова экшенов Pinia — иначе возможны повторные запуски и странные значения. Для side effects используйте [`callOnce`](/docs/4.x/api/utils/call-once).

```vue
<script setup lang="ts">
const offersStore = useOffersStore()

// you can't do this
await useAsyncData(() => offersStore.getOffer(route.params.slug))
</script>
```
::

## Возвращаемые значения

У `useFetch` и `useAsyncData` одинаковый набор полей:

- `data`: результат асинхронной функции.
- `refresh`/`execute`: обновить данные, возвращённые `handler`.
- `clear`: сбросить `data` в `undefined` (или в `options.default()`, если задано), обнулить `error`, выставить `status` в `idle` и отменить текущие запросы.
- `error`: объект ошибки при неудачной загрузке.
- `status`: состояние запроса (`"idle"`, `"pending"`, `"success"`, `"error"`).

::note
`data`, `error` и `status` — это Vue ref; в `<script setup>` доступ через `.value`
::

По умолчанию Nuxt не запускает новый `refresh`, пока не завершится предыдущий.

::note
Если данные не запрашивались на сервере (например `server: false`), они **не** загрузятся до завершения гидратации. Даже при `await useFetch` на клиенте в `<script setup>` `data` может оставаться `null`.
::

## Опции

[`useAsyncData`](/docs/4.x/api/composables/use-async-data) и [`useFetch`](/docs/4.x/api/composables/use-fetch) возвращают объект одного типа и принимают общие опции последним аргументом: блокировка навигации, кэш, момент выполнения.

### Lazy

По умолчанию композаблы ждут завершения асинхронной функции перед переходом на другую страницу (через Suspense Vue). На клиентской навигации это можно отключить опцией `lazy` — тогда состояние загрузки обрабатывайте через `status`.

```vue twoslash [app/app.vue]
<script setup lang="ts">
const { status, data: posts } = useFetch('/api/posts', {
  lazy: true,
})
</script>

<template>
  <!-- you will need to handle a loading state -->
  <div v-if="status === 'pending'">
    Loading ...
  </div>
  <div v-else>
    <div v-for="post in posts">
      <!-- do something -->
    </div>
  </div>
</template>
```

Альтернатива — [`useLazyFetch`](/docs/4.x/api/composables/use-lazy-fetch) и `useLazyAsyncData` как сокращения для того же поведения.

```vue twoslash
<script setup lang="ts">
const { status, data: posts } = useLazyFetch('/api/posts')
</script>
```

### Только на клиенте

По умолчанию асинхронная функция выполняется и на сервере, и на клиенте. `server: false` — только на клиенте. При первой загрузке данные не запросятся до гидратации — нужен UI для ожидания; при последующих клиентских переходах страница подождёт данные.

Вместе с `lazy` удобно для данных, не нужных при первом рендере (например не критичных для SEO).

```ts twoslash
/* This call is performed before hydration */
const articles = await useFetch('/api/article')

/* This call will only be performed on the client */
const { status, data: comments } = useFetch('/api/comments', {
  lazy: true,
  server: false,
})
```

`useFetch` вызывают в `setup` или на верхнем уровне в хуках жизненного цикла; иначе используйте [`$fetch`](/docs/4.x/getting-started/data-fetching#fetch).

### Уменьшение размера payload

Опция `pick` сужает payload в HTML: в ответ попадают только нужные поля.

```vue
<script setup lang="ts">
/* only pick the fields used in your template */
const { data: mountain } = await useFetch('/api/mountains/everest', {
  pick: ['title', 'description'],
})
</script>

<template>
  <h1>{{ mountain.title }}</h1>
  <p>{{ mountain.description }}</p>
</template>
```

Для большего контроля или преобразования нескольких объектов используйте `transform`.

```ts
const { data: mountains } = await useFetch('/api/mountains', {
  transform: (mountains) => {
    return mountains.map(mountain => ({ title: mountain.title, description: mountain.description }))
  },
})
```

::note
`pick` и `transform` не мешают сначала запросить «лишние» данные на сервере, но не добавят их в payload при передаче на клиент.
::

### Кэш и повторные запросы

#### Ключи

[`useFetch`](/docs/4.x/api/composables/use-fetch) и [`useAsyncData`](/docs/4.x/api/composables/use-async-data) используют ключи, чтобы не дублировать загрузку.

- [`useFetch`](/docs/4.x/api/composables/use-fetch): ключом служит URL; можно передать `key` в объекте опций последним аргументом.
- [`useAsyncData`](/docs/4.x/api/composables/use-async-data): если первый аргумент — строка, это ключ; если сразу передана функция-обработчик, ключ генерируется по файлу и строке вызова.

::tip
Кэш по ключу читается через [`useNuxtData`](/docs/4.x/api/composables/use-nuxt-data)
::

:video-accordion{title="Vue School: кэширование данных с опцией key" videoId="1026410044" platform="vimeo"}

#### Общее состояние и согласованность опций

Несколько компонентов с одним ключом в `useAsyncData` или `useFetch` делят одни и те же ref `data`, `error` и `status`. Это согласует данные, но требует одинаковых значений части опций.

Следующие опции **должны совпадать** у всех вызовов с одним ключом:
- функция `handler`
- опция `deep`
- функция `transform`
- массив `pick`
- функция `getCachedData`
- значение `default`

```ts
// ❌ This will trigger a development warning
const { data: users1 } = useAsyncData('users', (_nuxtApp, { signal }) => $fetch('/api/users', { signal }), { deep: false })
const { data: users2 } = useAsyncData('users', (_nuxtApp, { signal }) => $fetch('/api/users', { signal }), { deep: true })
```

Эти опции **могут отличаться** без предупреждений:
- `server`
- `lazy`
- `immediate`
- `dedupe`
- `watch`

```ts
// ✅ This is allowed
const { data: users1 } = useAsyncData('users', (_nuxtApp, { signal }) => $fetch('/api/users', { signal }), { immediate: true })
const { data: users2 } = useAsyncData('users', (_nuxtApp, { signal }) => $fetch('/api/users', { signal }), { immediate: false })
```

Независимые экземпляры — разные ключи:

```ts
// These are completely independent instances
const { data: users1 } = useAsyncData('users-1', (_nuxtApp, { signal }) => $fetch('/api/users', { signal }))
const { data: users2 } = useAsyncData('users-2', (_nuxtApp, { signal }) => $fetch('/api/users', { signal }))
```

#### Реактивные ключи

Ключом могут быть `computed`, обычный `ref` или геттер — данные перезапрашиваются при смене зависимостей:

```ts
// Using a computed property as a key
const userId = ref('123')
const { data: user } = useAsyncData(
  computed(() => `user-${userId.value}`),
  () => fetchUser(userId.value),
)

// When userId changes, the data will be automatically refetched
// and the old data will be cleaned up if no other components use it
userId.value = '456'
```

#### Обновление: `refresh` и `execute`

Чтобы вручную запросить или обновить данные, используйте `execute` или `refresh` из композабла.

```vue twoslash
<script setup lang="ts">
const { data, error, execute, refresh } = await useFetch('/api/users')
</script>

<template>
  <div>
    <p>{{ data }}</p>
    <button @click="() => refresh()">
      Refresh data
    </button>
  </div>
</template>
```

`execute` — синоним `refresh` с тем же поведением; удобнее, когда запрос [не immediate](/docs/4.x/getting-started/data-fetching#not-immediate).

::tip
Глобальная инвалидация и повторная загрузка: [`clearNuxtData`](/docs/4.x/api/utils/clear-nuxt-data) и [`refreshNuxtData`](/docs/4.x/api/utils/refresh-nuxt-data).
::

#### Очистка

Чтобы сбросить данные без знания ключа для `clearNuxtData`, вызовите `clear` из композабла.

```vue twoslash
<script setup lang="ts">
const { data, clear } = await useFetch('/api/users')

const route = useRoute()
watch(() => route.path, (path) => {
  if (path === '/') {
    clear()
  }
})
</script>
```

#### `watch`

Чтобы снова выполнять загрузку при изменении реактивных значений, задайте опцию `watch` — для одного или нескольких наблюдаемых источников.

```vue twoslash
<script setup lang="ts">
const id = ref(1)

const { data, error, refresh } = await useFetch('/api/users', {
  /* Changing the id will trigger a refetch */
  watch: [id],
})
</script>
```

**Наблюдение за реактивным значением не меняет URL запроса.** Например, ниже всё время запрашивается тот же начальный `id`, потому что URL собран в момент вызова.

```vue
<script setup lang="ts">
const id = ref(1)

const { data, error, refresh } = await useFetch(`/api/users/${id.value}`, {
  watch: [id],
})
</script>
```

Чтобы URL зависел от реактивного значения, используйте [вычисляемый URL](/docs/4.x/getting-started/data-fetching#computed-url).

Реактивные опции запроса по умолчанию отслеживаются и вызывают повторную загрузку. Чтобы отключить это, укажите `watch: false`.

```ts
const id = ref(1)

// Won't automatically refetch when id changes
const { data, execute } = await useFetch('/api/users', {
  query: { id }, // id is watched by default
  watch: false, // disables automatic watching of id
})

// doesn't trigger refetch
id.value = 2
```

#### Вычисляемый URL

Иногда URL нужно собрать из реактивных значений и обновлять данные при их смене. Можно передать параметры как ref — Nuxt подставит значения и перезапросит при изменении.

```vue
<script setup lang="ts">
const id = ref(null)

const { data, status } = useLazyFetch('/api/user', {
  query: {
    user_id: id,
  },
})
</script>
```

Для более сложной сборки URL передайте колбэк — [computed-геттер](https://vuejs.org/guide/essentials/computed), возвращающий строку URL.

При смене зависимостей запрос пойдёт по новому URL. Вместе с [not immediate](/docs/4.x/getting-started/data-fetching#not-immediate) можно дождаться изменения реактивного значения перед первым запросом.

```vue
<script setup lang="ts">
const id = ref(null)

const { data, status } = useLazyFetch(() => `/api/users/${id.value}`, {
  immediate: false,
})

const pending = computed(() => status.value === 'pending')
</script>

<template>
  <div>
    <!-- disable the input while fetching -->
    <input
      v-model="id"
      type="number"
      :disabled="pending"
    >

    <div v-if="status === 'idle'">
      Type an user ID
    </div>

    <div v-else-if="pending">
      Loading ...
    </div>

    <div v-else>
      {{ data }}
    </div>
  </div>
</template>
```

Чтобы принудительно обновлять при смене других реактивных значений, см. [watch](/docs/4.x/getting-started/data-fetching#watch).

### Без немедленного запроса

`useFetch` начинает загрузку сразу при вызове. `immediate: false` откладывает запрос, например до действия пользователя.

Тогда нужны `status` для жизненного цикла и `execute`, чтобы запустить загрузку.

```vue
<script setup lang="ts">
const { data, error, execute, status } = await useLazyFetch('/api/comments', {
  immediate: false,
})
</script>

<template>
  <div v-if="status === 'idle'">
    <button @click="execute">
      Get data
    </button>
  </div>

  <div v-else-if="status === 'pending'">
    Loading comments...
  </div>

  <div v-else>
    {{ data }}
  </div>
</template>
```

Возможные значения `status`:

- `idle` — запрос ещё не начинался
- `pending` — запрос начался, ответа ещё нет
- `error` — ошибка запроса
- `success` — запрос успешно завершён

## Заголовки и cookie

В браузере `$fetch` отправляет заголовки пользователя, в том числе `cookie`, напрямую в API.

При SSR из соображений безопасности `$fetch` по умолчанию не подставляет cookie браузера и не пробрасывает `Set-Cookie` из ответа.

Если на сервере вызывать `useFetch` с относительным URL, Nuxt использует [`useRequestFetch`](/docs/4.x/api/composables/use-request-fetch) для проксирования заголовков и cookie (кроме тех, что не должны пересылаться, например `host`).

### Проброс cookie из внутренних API в ответ SSR

Чтобы передать cookie от внутреннего запроса обратно клиенту, это нужно сделать вручную.

```ts [app/composables/fetch.ts]
import { appendResponseHeader } from 'h3'
import type { H3Event } from 'h3'

export const fetchWithCookie = async (event: H3Event, url: string) => {
  /* Get the response from the server endpoint */
  const res = await $fetch.raw(url)
  /* Get the cookies from the response */
  const cookies = res.headers.getSetCookie()
  /* Attach each cookie to our incoming Request */
  for (const cookie of cookies) {
    appendResponseHeader(event, 'set-cookie', cookie)
  }
  /* Return the data of the response */
  return res._data
}
```

```vue
<script setup lang="ts">
// This composable will automatically pass cookies to the client
const event = useRequestEvent()

const { data: result } = await useAsyncData(() => fetchWithCookie(event!, '/api/with-cookie'))

onMounted(() => console.log(document.cookie))
</script>
```

## Сериализация данных с сервера на клиент

`useAsyncData` и `useLazyAsyncData` передают данные с сервера на клиент через [payload Nuxt](/docs/4.x/api/composables/use-nuxt-app#payload), сериализуя его [`devalue`](https://github.com/sveltejs/devalue). Так можно передавать не только JSON, но и RegExp, Date, Map, Set, `ref`, `reactive`, `shallowRef`, `shallowReactive`, `NuxtError` и другое.

Для неподдерживаемых типов можно задать свои сериализатор/десериализатор — см. [`useNuxtApp`](/docs/4.x/api/composables/use-nuxt-app#payload).

::note
Это **не относится** к данным из server routes при запросе через `$fetch` или `useFetch` — см. следующий раздел.
::

## Сериализация данных из API routes

Ответы из каталога `server` сериализуются через `JSON.stringify`. Из-за ограничений примитивами Nuxt старается привести тип результата `$fetch` и [`useFetch`](/docs/4.x/api/composables/use-fetch) к фактическому значению.

::read-more{icon="i-simple-icons-mdnwebdocs" to="https://developer.mozilla.org/en-US/docs/Web/JavaScript/Reference/Global_Objects/JSON/stringify#description" target="_blank"}
Ограничения `JSON.stringify`.
::

### Пример

```ts [server/api/foo.ts]
export default defineEventHandler(() => {
  return new Date()
})
```

```vue [app/app.vue]
<script setup lang="ts">
// Type of `data` is inferred as string even though we returned a Date object
const { data } = await useFetch('/api/foo')
</script>
```

### Своя функция сериализации

Задайте на возвращаемом объекте метод `toJSON`: Nuxt возьмёт его результат как тип и не будет дополнительно преобразовывать поля.

```ts [server/api/bar.ts]
export default defineEventHandler(() => {
  const data = {
    createdAt: new Date(),

    toJSON () {
      return {
        createdAt: {
          year: this.createdAt.getFullYear(),
          month: this.createdAt.getMonth(),
          day: this.createdAt.getDate(),
        },
      }
    },
  }
  return data
})
```

```vue [app/app.vue]
<script setup lang="ts">
// Type of `data` is inferred as
// {
//   createdAt: {
//     year: number
//     month: number
//     day: number
//   }
// }
const { data } = await useFetch('/api/bar')
</script>
```

### Другой сериализатор

Встроенной замены `JSON.stringify` нет, но можно вернуть строку и распарсить на клиенте, сохраняя типы через `toJSON` при необходимости.

Ниже — [superjson](https://github.com/flightcontrolhq/superjson).

```ts [server/api/superjson.ts]
import superjson from 'superjson'

export default defineEventHandler(() => {
  const data = {
    createdAt: new Date(),

    // Workaround the type conversion
    toJSON () {
      return this
    },
  }

  // Serialize the output to string, using superjson
  return superjson.stringify(data) as unknown as typeof data
})
```

```vue [app/app.vue]
<script setup lang="ts">
import superjson from 'superjson'

// `date` is inferred as { createdAt: Date } and you can safely use the Date object methods
const { data } = await useFetch('/api/superjson', {
  transform: (value) => {
    return superjson.parse(value as unknown as string)
  },
})
</script>
```

## Рецепты

### SSE (Server-Sent Events) через POST

::tip
Для SSE по GET подойдут [`EventSource`](https://developer.mozilla.org/en-US/docs/Web/API/EventSource) или композабл VueUse [`useEventSource`](https://vueuse.org/core/useeventsource/).
::

При POST нужно вручную обработать поток:

```ts
// Make a POST request to the SSE endpoint
const response = await $fetch<ReadableStream>('/chats/ask-ai', {
  method: 'POST',
  body: {
    query: 'Hello AI, how are you?',
  },
  responseType: 'stream',
})

// Create a new ReadableStream from the response with TextDecoderStream to get the data as text
const reader = response.pipeThrough(new TextDecoderStream()).getReader()

// Read the chunk of data as we get it
while (true) {
  const { value, done } = await reader.read()

  if (done) { break }

  console.log('Received:', value)
}
```

### Параллельные запросы

Если запросы независимы, выполняйте их параллельно через `Promise.all()`.

```ts
const { data } = await useAsyncData((_nuxtApp, { signal }) => {
  return Promise.all([
    $fetch('/api/comments/', { signal }),
    $fetch('/api/author/12', { signal }),
  ])
})

const comments = computed(() => data.value?.[0])
const author = computed(() => data.value?.[1])
```

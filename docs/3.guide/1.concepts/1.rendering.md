Nuxt поддерживает разные режимы рендеринга: [универсальный рендеринг](/docs/4.x/guide/concepts/rendering#universal-rendering), [рендеринг на клиенте](/docs/4.x/guide/concepts/rendering#client-side-rendering), а также [гибридный рендеринг](/docs/4.x/guide/concepts/rendering#hybrid-rendering) и возможность отдавать приложение с [CDN на границе сети](/docs/4.x/guide/concepts/rendering#edge-side-rendering).

И браузер, и сервер могут выполнять JavaScript и превращать компоненты Vue.js в HTML. Этот шаг называется **рендерингом**. Nuxt поддерживает **универсальный** и **клиентский** рендеринг; у каждого подхода есть плюсы и минусы, которые мы разберём.

По умолчанию Nuxt использует **универсальный рендеринг** для UX, производительности и SEO, но режим можно сменить [одной строкой конфигурации](/docs/4.x/api/nuxt-config#ssr).

## Универсальный рендеринг

Это близко к классическому **рендерингу на сервере** (SSR), как у PHP или Ruby. При запросе URL с включённым универсальным рендерингом Nuxt выполняет код Vue.js на сервере и отдаёт браузеру полностью отрисованный HTML. Страница может отдаваться из кеша, если она была сгенерирована заранее. Пользователь сразу видит начальное содержимое приложения — в отличие от чистого CSR.

После загрузки HTML браузер его разбирает, и Vue.js берёт документ под контроль. Тот же JavaScript, что выполнялся на сервере, на клиенте (в браузере) выполняется **снова** в фоне и включает интерактивность (поэтому режим и называется **универсальным**): вешаются обработчики на уже существующий HTML. Это **гидратация**. После неё доступны динамический интерфейс и переходы между страницами.

Универсальный рендеринг даёт быструю первую отрисовку и при этом сохраняет плюсы клиентского рендеринга. Контент уже в HTML, поэтому краулеры индексируют его без лишних затрат.

![Пользователь видит статический контент при загрузке HTML; гидратация включает интерактивность страницы](/assets/docs/concepts/rendering/ssr.svg)

**Что рендерится на сервере, а что на клиенте?**

Естественно спросить, какие части `.vue` выполняются на сервере и/или клиенте в универсальном режиме.

```vue [app/app.vue]
<script setup lang="ts">
const counter = ref(0) // выполняется и на сервере, и на клиенте

const handleClick = () => {
  counter.value++ // только в браузере
}
</script>

<template>
  <div>
    <p>Count: {{ counter }}</p>
    <button @click="handleClick">
      Increment
    </button>
  </div>
</template>
```

При первом запросе `counter` инициализируется на сервере, так как попадает в `<p>`. Тело `handleClick` на сервере не выполняется. При гидратации в браузере `counter` инициализируется снова, `handleClick` привязывается к кнопке — значит, тело `handleClick` всегда выполняется в браузере.

[Middleware](/docs/4.x/directory-structure/app/middleware) и [страницы](/docs/4.x/directory-structure/app/pages) выполняются на сервере и на клиенте при гидратации. [Плагины](/docs/4.x/directory-structure/app/plugins) могут работать на сервере, на клиенте или в обоих местах. [Компоненты](/docs/4.x/directory-structure/app/components) можно принудительно оставить только на клиенте. [Композаблы](/docs/4.x/directory-structure/app/composables) и [утилиты](/docs/4.x/directory-structure/app/utils) рендерятся в зависимости от контекста вызова.

**Плюсы рендеринга на сервере:**
- **Производительность**: контент виден сразу, статический HTML отображается быстрее, чем сгенерированный только на клиенте. При этом Nuxt сохраняет интерактивность после гидратации.
- **SEO**: в браузер приходит полный HTML, как у классического серверного приложения. Краулеры индексируют контент напрямую — удобно для всего, что нужно быстро попасть в поиск.

**Минусы рендеринга на сервере:**
- **Ограничения разработки:** API сервера и браузера различаются; писать код, одинаково работающий в обеих средах, бывает непросто. Nuxt даёт подсказки и переменные, чтобы понять, где выполняется код.
- **Стоимость:** для SSR нужен работающий сервер — как у любого классического бэкенда. Но при универсальном рендеринге запросов к серверу меньше: при клиентской навигации работу берёт на себя браузер. Снизить затраты помогает [edge-side-rendering](/docs/4.x/guide/concepts/rendering#edge-side-rendering).

Универсальный рендеринг гибок и подходит почти для любых задач, особенно для контентных сайтов: **блоги, маркетинг, портфолио, e-commerce и маркетплейсы.**

::tip
Больше примеров по Vue без рассогласования гидратации — в [документации Vue](https://vuejs.org/guide/scaling-up/ssr#hydration-mismatch).
::

::important
Если библиотека зависит от браузерных API и имеет побочные эффекты при импорте, убедитесь, что компонент, который её импортирует, вызывается только на клиенте. Бандлеры не вырезают (treeshake) импорты модулей с побочными эффектами.
::

## Рендеринг на клиенте

В классическом SPA Vue.js рендерится в браузере (**клиенте**). HTML появляется после того, как браузер загрузит и разберёт весь JavaScript с инструкциями по интерфейсу.

![Пользователь ждёт загрузки, разбора и выполнения JavaScript, прежде чем увидеть контент](/assets/docs/concepts/rendering/csr.svg)

**Плюсы клиентского рендеринга:**
- **Скорость разработки:** можно не думать о серверной совместимости и свободно использовать только браузерные API вроде `window`.
- **Дешевле:** сервер с выполнением JS стоит денег. Чисто клиентское приложение можно разместить на любом статическом хостинге с HTML, CSS и JS.
- **Офлайн:** код целиком в браузере, приложение может работать без сети.

**Минусы клиентского рендеринга:**
- **Производительность:** нужно дождаться загрузки, разбора и выполнения JS — от сети и устройства пользователя.
- **SEO:** индексация и обновление контента при CSR обычно медленнее, чем при готовом HTML на сервере. Краулеры не всегда ждут полной отрисовки интерфейса; контент дольше появляется и обновляется в выдаче.

CSR уместен для сильно интерактивных **веб-приложений**, где индексация не критична или пользователи заходят часто; можно использовать кеш браузера и пропускать повторную загрузку — **SaaS, админки, онлайн-игры**.

Включить только клиентский рендеринг в `nuxt.config.ts`:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  ssr: false,
})
```

::note
При `ssr: false` положите HTML в `~/spa-loading-template.html` — экран загрузки до гидратации приложения.
:read-more{title="SPA Loading Template" to="/docs/4.x/api/configuration/nuxt-config#spaloadingtemplate"}
::

### Деплой статического CSR-приложения

При деплое на [статический хостинг](/docs/4.x/getting-started/deployment#static-hosting) с `nuxt generate` или `nuxt build --prerender` по умолчанию Nuxt пререндерит каждую страницу в отдельный статический HTML.

::warning
При пререндере через `nuxt generate` или `nuxt build --prerender` серверные эндпоинты недоступны — в выходной папке нет сервера. Нужен сервер — используйте `nuxt build`.
::

При чистом CSR иногда достаточно одного `index.html` и fallback `200.html` / `404.html`, которые хост отдаёт на все запросы.

Для этого меняют список пререндеримых маршрутов — добавьте в [хуки](/docs/4.x/api/advanced/hooks#nuxt-hooks-build-time) в `nuxt.config.ts`:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  hooks: {
    'prerender:routes' ({ routes }) {
      routes.clear() // Не генерировать маршруты (кроме значений по умолчанию)
    },
  },
})
```

Получатся три файла:

- `index.html`
- `200.html`
- `404.html`

`200.html` и `404.html` могут пригодиться вашему хостингу.

#### Без клиентских fallback-файлов

При пререндере CSR Nuxt по умолчанию создаёт `index.html`, `200.html` и `404.html`. Чтобы не генерировать часть или все эти файлы, используйте хук `'prerender:generate'` в [Nitro](/docs/4.x/getting-started/prerendering#prerendergenerate-nitro-hook).

```ts twoslash [nuxt.config.ts]
// @errors: 2353 7006
export default defineNuxtConfig({
  ssr: false,
  nitro: {
    hooks: {
      'prerender:generate' (route) {
        const routesToSkip = ['/index.html', '/200.html', '/404.html']
        if (routesToSkip.includes(route.route)) {
          route.skip = true
        }
      },
    },
  },
})
```

## Гибридный рендеринг

Гибридный рендеринг задаёт разные правила кеширования и ответа сервера для маршрутов через **Route Rules**.

Раньше весь сайт был либо универсальным, либо CSR. На практике часть страниц хочется собрать на билде, а часть — только в браузере. Пример: контентный сайт и админка: публичные страницы статичны, админка — динамическое приложение с авторизацией.

В Nuxt есть route rules и гибридный рендеринг: правила для групп маршрутов, смена режима рендеринга и стратегии кеша.

Сервер Nuxt регистрирует middleware и оборачивает маршруты в обработчики кеша через [слой кеширования Nitro](https://nitro.build/guide/cache).

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  routeRules: {
    // Главная — пререндер при сборке
    '/': { prerender: true },
    // Каталог товаров — по требованию, SWR, кеш до изменения ответа API
    '/products': { swr: true },
    // Карточки товаров — по требованию, фоновое обновление, кеш 1 час
    '/products/**': { swr: 3600 },
    // Блог — по требованию, ISR на CDN на 1 час
    '/blog': { isr: 3600 },
    // Посты — по требованию до следующего деплоя, кеш на CDN
    '/blog/**': { isr: true },
    // Админка — только на клиенте
    '/admin/**': { ssr: false },
    // CORS для API
    '/api/**': { cors: true },
    // Редиректы со старых URL
    '/old-page': { redirect: '/new-page' },
  },
})
```

### Route Rules

Доступные свойства:

- `redirect: string`{lang=ts} — серверные редиректы.
- `ssr: boolean`{lang=ts} — отключить SSR HTML для части приложения (`ssr: false` — только браузер).
- `cors: boolean`{lang=ts} — CORS-заголовки с `cors: true`; можно переопределить через `headers`.
- `headers: object`{lang=ts} — заголовки для части сайта (например, для ассетов).
- `swr: number | boolean`{lang=ts} — кеш ответа на сервере/прокси с TTL; у пресета `node-server` Nitro кеширует весь ответ. После истечения TTL отдаётся старый ответ, страница пересобирается в фоне. `true` — заголовок stale-while-revalidate без MaxAge.
- `isr: number | boolean`{lang=ts} — как `swr`, но ответ может попасть в кеш CDN (Netlify, Vercel). `true` — контент в CDN до следующего деплоя.
- `prerender: boolean`{lang=ts} — пререндер маршрутов при сборке как статика.
- `noScripts: boolean`{lang=ts} — не подключать скрипты Nuxt и подсказки к ресурсам для части сайта.
- `appMiddleware: string | string[] | Record<string, boolean>`{lang=ts} — какой app-middleware запускать для путей во Vue-приложении (не Nitro-маршруты).

::note
Маршруты с `isr` или `swr` также создают `_payload.json` рядом с HTML. При клиентской навигации подгружаются эти payload вместо повторного запроса данных. Динамические маршруты вроде `pages/[...slug].vue` задавайте шаблонами: `'/**': { isr: true }`.
::

По возможности правила маппятся на нативные механизмы платформы (Netlify, Vercel).

::important
Гибридный рендеринг недоступен при [`nuxt generate`](/docs/4.x/api/commands/generate).
::

## Рендеринг на границе (ESR)

Edge-Side Rendering (ESR) — рендеринг приложения ближе к пользователю на edge-серверах CDN. Меньше задержка и быстрее ответ.

ESR переносит рендеринг на «край» сети — edge CDN. Это скорее цель деплоя, чем отдельный «режим» внутри Nuxt.

Запрос перехватывает ближайший edge-сервер, генерирует HTML и отдаёт клиенту — короче путь данных, **ниже задержка и быстрее загрузка**.

Edge-рендеринг обеспечивает [Nitro](https://nitro.build/) — [серверный движок](/docs/4.x/guide/concepts/server-engine) Nuxt: Node.js, Deno, Cloudflare Workers и др.

Типичные платформы:

- [Cloudflare Pages](https://pages.cloudflare.com) — git-интеграция и `nuxt build`
- [Vercel Cloud](https://vercel.com/home) — `nuxt build` и `NITRO_PRESET=vercel-edge`
- [Netlify Edge Functions](https://www.netlify.com/platform/#netlify-edge-functions) — `nuxt build` и `NITRO_PRESET=netlify-edge`

**Гибридный рендеринг** с route rules совместим и с ESR.

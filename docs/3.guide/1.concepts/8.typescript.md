## Проверка типов

По умолчанию Nuxt не проверяет типы при `nuxt dev` и `nuxt build` ради скорости.

Чтобы включить проверку, установите `vue-tsc` и `typescript` как dev-зависимости:

::code-group{sync="pm"}

  ```bash [npm]
  npm install --save-dev vue-tsc typescript
  ```

  ```bash [yarn]
  yarn add --dev vue-tsc typescript
  ```

  ```bash [pnpm]
  pnpm add -D vue-tsc typescript
  ```

  ```bash [bun]
  bun add -D vue-tsc typescript
  ```

  ```bash [deno]
  deno add -D npm:vue-tsc npm:typescript
  ```

::

Затем выполните [`nuxt typecheck`](/docs/4.x/api/commands/typecheck):

```bash [Terminal]
npx nuxt typecheck
```

Проверку при dev/сборке можно включить опцией [`typescript.typeCheck`](/docs/4.x/api/nuxt-config#typecheck) в `nuxt.config`:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  typescript: {
    typeCheck: true,
  },
})
```

## Автогенерируемые типы

Проекты Nuxt опираются на автогенерируемые типы в каталоге [`.nuxt`](/docs/4.x/directory-structure/nuxt); они создаются при dev/сборке или вручную через `nuxt prepare`.

Сгенерированные `tsconfig.json` в `.nuxt` задают **базовую рекомендуемую конфигурацию TypeScript**, ссылки на [автоимпорты](/docs/4.x/guide/concepts/auto-imports), [типы маршрутов API](/docs/4.x/guide/concepts/server-engine#typed-api-routes), алиасы `#imports`, `~/file`, `#build/file` и т.д.

::warning
Nuxt и [модули](/docs/4.x/guide/modules) опираются на эту конфигурацию. Не редактируйте корневой `tsconfig.json` напрямую — можно затереть важные настройки. Расширяйте через `nuxt.config.ts`. [Как расширять конфигурацию](/docs/4.x/directory-structure/tsconfig).
::

## Project References

Nuxt использует [project references TypeScript](https://www.typescriptlang.org/docs/handbook/project-references.html) для ускорения проверки типов и лучшей поддержки в IDE — код делится на более мелкие части.

### Как Nuxt использует project references

При `nuxt dev`, `nuxt build` или `nuxt prepare` генерируется несколько `tsconfig.json` для разных частей приложения.

- **`.nuxt/tsconfig.app.json`** — код приложения в `app/`
- **`.nuxt/tsconfig.node.json`** — `nuxt.config.ts` и файлы вне других контекстов
- **`.nuxt/tsconfig.server.json`** — серверный код (если есть)
- **`.nuxt/tsconfig.shared.json`** — общий код между app и server (типы, нейтральные утилиты)

В каждом файле — свои ссылки и оптимальная проверка для контекста.

::note
Для обратной совместимости по-прежнему есть `.nuxt/tsconfig.json`. Рекомендуем [project references](/docs/4.x/directory-structure/tsconfig) с новыми файлами. Устаревший файл будет удалён в будущих версиях Nuxt.
::

### Плюсы project references

- **Быстрее сборки**: TypeScript не пересобирает неизменённые проекты
- **Быстрее IDE**: IntelliSense и ошибки обрабатываются быстрее
- **Изолированная компиляция**: ошибки в одной части не блокируют остальные
- **Явные зависимости**: каждый проект декларирует, от чего зависит

### Расширение типов в нужном контексте

Типы нужно **дополнять в правильном контексте**, иначе TypeScript их не увидит.

Например, для контекста `app` — файл дополнения в `app/`.

- Сервер — в `server/`.
- Общие типы — в `shared/`.

Как дополнять контексты из **файлов вне этих каталогов** — в руководстве для авторов модулей.

## Строгие проверки

TypeScript даёт дополнительные проверки безопасности и анализа.

При включённой [`typescript.typeCheck`](/docs/4.x/guide/concepts/typescript#type-checking) [строгие проверки](https://www.typescriptlang.org/docs/handbook/migrating-from-javascript.html#getting-stricter-checks) в Nuxt включены по умолчанию.

При постепенном переходе на TypeScript можно временно отключить strict, задав `strict: false` в `nuxt.config`:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  typescript: {
    strict: false,
  },
})
```

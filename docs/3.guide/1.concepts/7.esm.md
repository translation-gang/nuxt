Здесь объясняется, что такое ES-модули и как сделать приложение Nuxt (или upstream-библиотеку) совместимым с ESM.

## Предыстория

### Модули CommonJS

CommonJS (CJS) — формат Node.js для обмена кодом между модулями ([подробнее](https://nodejs.org/api/modules.html)).
Типичный синтаксис:

```js
const a = require('./a')

module.exports.a = a
```

Бандлеры вроде webpack и Rollup понимают CJS и позволяют использовать такие модули в браузере.

### Синтаксис ESM

Часто под «ESM против CJS» имеют в виду другой синтаксис [модулей](https://developer.mozilla.org/en-US/docs/Web/JavaScript/Guide/Modules).

```js
import a from './a'

export { a }
```

До стандарта ECMAScript Modules (ESM) прошло больше 10 лет; [webpack](https://webpack.js.org/guides/ecma-script-modules/) и TypeScript уже поддерживали **синтаксис ESM**.
Есть отличия от спецификации; наглядное объяснение — [на MDN/Hacks](https://hacks.mozilla.org/2018/03/es-modules-a-cartoon-deep-dive/).

### Что значит «нативный» ESM?

Вы могли давно писать приложение в синтаксисе ESM: в браузере он нативный, в Nuxt 2 весь ваш код собирался в нужный формат (CJS для сервера, ESM для клиента).

С npm-пакетами иначе: библиотека могла отдавать и CJS, и ESM:

```json
{
  "name": "sample-library",
  "main": "dist/sample-library.cjs.js",
  "module": "dist/sample-library.esm.js"
}
```

В Nuxt 2 бандлер (webpack) для сервера брал `main` (CJS), для клиента — `module` (ESM).

::note
Поле `module` — договорённость бандлеров, Node.js его не использует. Node смотрит на [`exports`](https://nodejs.org/api/packages.html#exports) и [`main`](https://nodejs.org/api/packages.html#main).
::

В современных LTS Node.js можно [использовать нативный ESM](https://nodejs.org/api/esm.html): Node сам разбирает синтаксис ESM, но не по умолчанию. Чаще всего включают так:

- `"type": "module"` в `package.json` и расширение `.js`
- или расширение `.mjs` (рекомендуется)

Так сделано в Nitro: выход — `.output/server/index.mjs`, Node трактует файл как нативный ES-модуль.

### Какие импорты допустимы в Node.js?

При `import` вместо `require` разрешение модулей другое: для `sample-library` Node смотрит `exports` в `package.json`, иначе `main`.

То же для динамического `import('sample-library')`.

Поддерживаемые варианты ([документация](https://nodejs.org/api/packages.html#determining-module-system)):

1. файлы `.mjs` — ожидается синтаксис ESM
1. файлы `.cjs` — ожидается CJS
1. файлы `.js` — CJS, если в `package.json` нет `"type": "module"`

### Какие бывают проблемы?

Долгое время авторы собирали ESM в файлы `.esm.js` / `.es.js` и указывали их в `module`. Для webpack это не мешало.

В нативном ESM-контексте Node импорт пакета с `.esm.js` может упасть:

```bash [Terminal]

/path/to/index.js:1

export default {}
```

Похожая ошибка при именованном импорте из сборки, которую Node считает CJS:

## Устранение проблем ESM

Если видите такие ошибки, почти наверняка виновата upstream-библиотека — ей нужно [починить пакет](/docs/4.x/guide/concepts/esm#library-author-guide) для импорта из Node.

### Транспиляция библиотек

Пока библиотека не исправлена, добавьте её в `build.transpile`, чтобы Nuxt не тянул проблемный ESM напрямую:

```ts twoslash
export default defineNuxtConfig({
  build: {
    transpile: ['sample-library'],
  },
})
```

Иногда нужно добавить и транзитивные зависимости.

### Алиасы

Иногда помогает явный алиас на CJS-сборку:

```ts twoslash
export default defineNuxtConfig({
  alias: {
    'sample-library': 'sample-library/dist/sample-library.cjs.js',
  },
})
```

### Default exports

В CJS зависимости default задают через `module.exports` или `exports`:

```js [node_modules/cjs-pkg/index.js]
module.exports = { test: 123 }
// or
exports.test = 123
```

С `require` это обычно работает:

```js [test.cjs]
const pkg = require('cjs-pkg')

console.log(pkg) // { test: 123 }
```

[Нативный ESM в Node](https://nodejs.org/api/esm.html#interoperability-with-commonjs), [TypeScript с `esModuleInterop`](https://www.typescriptlang.org/tsconfig/#esModuleInterop) и бандлеры дают совместимость для `import pkg from 'cjs-pkg'` («interop require default»):

```js
import pkg from 'cjs-pkg'

console.log(pkg) // { test: 123 }
```

Из-за различий форматов иногда interop ломается:

```js
import pkg from 'cjs-pkg'

console.log(pkg) // { default: { test: 123 } }
```

При динамическом `import` часто так:

```js
import('cjs-pkg').then(console.log) // [Module: null prototype] { default: { test: '123' } }
```

Тогда вручную нормализуйте default:

<!-- eslint-disable import/no-named-default -->

```js
// Статический импорт
import { default as pkg } from 'cjs-pkg'

// Динамический импорт
import('cjs-pkg').then(m => m.default || m).then(console.log)
```

Для сложных случаев в Nuxt рекомендуем [mlly](https://github.com/unjs/mlly) с сохранением именованных экспортов.

```js
import { interopDefault } from 'mlly'

// Допустим, { default: { foo: 'bar' }, baz: 'qux' }
import myModule from 'my-module'

console.log(interopDefault(myModule)) // { foo: 'bar', baz: 'qux' }
```

## Руководство для авторов библиотек

Исправить совместимость с ESM обычно несложно. Два основных пути:

1. **Переименовать ESM-файлы в `.mjs`.**

   _Рекомендуемый и простой вариант._ Возможны правки зависимостей и сборки; CJS-файлы лучше назвать `.cjs` для ясности.

1. **Сделать библиотеку только ESM.**

   `"type": "module"` в `package.json` и ESM в сборке. Возможны проблемы с зависимостями; потребители должны быть в ESM-контексте.

### Миграция

Замените `require` на `import`:

::code-group

```ts [Before]
module.exports = function () { /* ... */ }

exports.hello = 'world'
```

```ts [After]
export default function () { /* ... */ }

export const hello = 'world'
```

::

::code-group

```js [Before]
const myLib = require('my-lib')
```

```js [After]
import myLib from 'my-lib'
// or
const dynamicMyLib = await import('my-lib').then(lib => lib.default || lib)
```

::

В ESM нет глобалей `require`, `require.resolve`, `__filename`, `__dirname` — используйте `import()` и `import.meta.url` (и производные).

::code-group

```js [Before]
const { join } = require('node:path')

const newDir = join(__dirname, 'new-dir')
```

```js [After]
import { fileURLToPath } from 'node:url'

const newDir = fileURLToPath(new URL('./new-dir', import.meta.url))
```

::

::code-group

```js [Before]
const someFile = require.resolve('./lib/foo.js')
```

```js [After]
import { resolvePath } from 'mlly'

const someFile = await resolvePath('my-lib', { url: import.meta.url })
```

::

### Лучшие практики

- Предпочитайте именованные экспорты default — меньше конфликтов с CJS (см. [Default exports](/docs/4.x/guide/concepts/esm#default-exports)).

- По возможности не завязывайтесь на встроенные модули Node и CJS/только-Node зависимости, чтобы библиотека работала в браузере и Edge Workers без полифиллов Nitro.

- Используйте поле `exports` с условными экспортами ([подробнее](https://nodejs.org/api/packages.html#conditional-exports)).

```json
{
  "exports": {
    ".": {
      "import": "./dist/mymodule.mjs"
    }
  }
}
```

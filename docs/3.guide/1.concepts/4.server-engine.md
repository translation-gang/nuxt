Разрабатывая Nuxt, мы создали новый серверный движок: [Nitro](https://nitro.build/).

Он поставляется с множеством возможностей:

- Кроссплатформенная поддержка Node.js, браузеров, service workers и др.
- Поддержка serverless из коробки.
- Маршруты API.
- Автоматическое разбиение кода и асинхронная подгрузка чанков.
- Гибридный режим для статики и serverless.
- Dev-сервер с горячей перезагрузкой модулей.

## Слой API

Серверные [эндпоинты API](/docs/4.x/directory-structure/server#server-routes) и [middleware](/docs/4.x/directory-structure/server#server-middleware) добавляет Nitro на базе [h3](https://github.com/h3js/h3).

Ключевые возможности:

- Обработчики могут возвращать объекты и массивы — ответ в JSON формируется автоматически
- Обработчики могут возвращать промисы, они будут ожидаться (поддерживаются и `res.end()`, и `next()`)
- Вспомогательные функции для разбора тела запроса, cookies, редиректов, заголовков и т.д.

Подробнее — в [документации h3](https://github.com/h3js/h3).

## Прямые вызовы API

Nitro позволяет «напрямую» вызывать маршруты через глобальный хелпер [`$fetch`](/docs/4.x/api/utils/dollarfetch): в браузере это HTTP-запрос к серверу, на сервере — прямой вызов соответствующей функции, **без лишнего HTTP**.

API [`$fetch`](/docs/4.x/api/utils/dollarfetch) построен на [ofetch](https://github.com/unjs/ofetch), в том числе:

- Автоматический разбор JSON-ответов (при необходимости доступен сырой ответ)
- Тело и параметры запроса обрабатываются автоматически, с корректными заголовками `Content-Type`

Подробнее о возможностях `$fetch` — в [ofetch](https://github.com/unjs/ofetch).

## Типизированные маршруты API

Для маршрутов API (или middleware) Nitro генерирует типы, если вы возвращаете значение, а не отвечаете через `res.end()`.

Эти типы доступны при использовании [`$fetch()`](/docs/4.x/api/utils/dollarfetch) или [`useFetch()`](/docs/4.x/api/composables/use-fetch).

## Автономный сервер

Nitro собирает автономный дистрибутив сервера, не зависящий от `node_modules`.

Сервер в Nuxt 2 не был автономным и требовал часть ядра Nuxt при запуске `nuxt start` (дистрибутивы [`nuxt-start`](https://www.npmjs.com/package/nuxt-start) или [`nuxt`](https://www.npmjs.com/package/nuxt)) или программного API — это хрупко, ломалось и плохо подходило для serverless и service worker.

Nuxt генерирует этот дистрибутив командой `nuxt build` в каталог [`.output`](/docs/4.x/directory-structure/output).

В выходе — runtime для запуска сервера Nuxt в любой среде (включая экспериментальные service worker в браузере) и отдачи статики, то есть полноценный гибридный фреймворк для JAMstack. Плюс встроенный слой хранилища с драйверами из разных источников и локальными ассетами.

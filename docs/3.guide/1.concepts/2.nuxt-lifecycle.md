Цель этой главы — дать обзор этапов работы фреймворка, порядка их выполнения и того, как они согласуются друг с другом.

## Сервер

На сервере для **каждого** первичного запроса к приложению выполняются следующие шаги.

### Шаг 1: запуск Nitro и серверных плагинов (один раз)

Nuxt использует [Nitro](https://nitro.build/) — серверный движок.

При старте Nitro инициализирует и выполняет плагины из каталога `/server/plugins`. Они могут:
- перехватывать и обрабатывать ошибки приложения;
- подписываться на остановку Nitro;
- вешать хуки на жизненный цикл запроса (например менять ответ).

::callout{icon="i-lucide-lightbulb"}
Плагины Nitro выполняются один раз при старте сервера. В бессерверной среде экземпляр поднимается на каждый запрос — тогда и плагины запускаются заново, но их **не ждут** (`await`).
::

:read-more{to="/docs/3.x/directory-structure/server#server-plugins"}

### Шаг 2: middleware Nitro

После инициализации Nitro для **каждого** запроса выполняется middleware из `server/middleware/`. Типичные задачи: аутентификация, логирование, преобразование запроса.

::warning
Если middleware **возвращает значение**, запрос завершается и это значение уходит как ответ. Так делать обычно не стоит — ломается нормальная обработка запроса.
::

:read-more{to="/docs/3.x/directory-structure/server#server-middleware"}

### Шаг 3: инициализация Nuxt и плагинов приложения

Сначала создаются инстансы Vue и Nuxt, затем выполняются **серверные** плагины Nuxt:
- встроенные (Vue Router, `unhead` и др.);
- ваши из `plugins/`: без суффикса (например `myPlugin.ts`) и с `.server` (например `myServerPlugin.server.ts`).

Порядок запуска и параллелизм описаны в [документации по плагинам](/docs/3.x/directory-structure/plugins).

::callout{icon="i-lucide-lightbulb"}
После этого Nuxt вызывает хук [`app:created`](/docs/3.x/api/advanced/hooks#app-hooks-runtime) — в нём можно выполнить дополнительную логику.
::

:read-more{to="/docs/3.x/directory-structure/plugins"}

### Шаг 4: валидация маршрута

После плагинов и до middleware приложения Nuxt вызывает `validate` из `definePageMeta`, если он задан. Метод может быть синхронным или асинхронным и обычно проверяет параметры динамического маршрута.

- при успехе верните `true`;
- при ошибке — `false` или объект с `statusCode` и/или `statusMessage`, чтобы прервать запрос.

Подробнее: [валидация маршрута](/docs/3.x/getting-started/routing#route-validation).

:read-more{to="/docs/3.x/getting-started/routing#route-validation"}

### Шаг 5: middleware приложения Nuxt

Middleware выполняет код **до** перехода на маршрут: часто аутентификация, редиректы, логирование.

В Nuxt три вида:
- **глобальный** route middleware;
- **именованный**;
- **анонимный (inline)**.

Глобальный middleware запускается при первом входе в приложение и перед каждой навигацией. Именованный и анонимный — только на маршрутах, где они указаны в `definePageMeta`.

Детали и примеры: [middleware](/docs/3.x/directory-structure/middleware).

Редирект на сервере отдаёт заголовок `Location:`; браузер делает новый запрос. Состояние приложения сбрасывается, если его не сохранить (например в cookie).

:read-more{to="/docs/3.x/directory-structure/middleware"}

### Шаг 6: рендер страницы и компонентов

Nuxt рендерит страницу и компоненты и в этом шаге подгружает данные через `useFetch` / `useAsyncData`. На сервере нет динамических обновлений DOM, поэтому хуки Vue вроде `onBeforeMount`, `onMounted` и последующие **не** выполняются при SSR.

По умолчанию Vue на SSR приостанавливает трекинг зависимостей ради производительности.

::callout{icon="i-lucide-lightbulb"}
На сервере нет «живой» реактивности в привычном смысле: SSR идёт сверху вниз в статический HTML, уже отрендеренное нельзя «переделать».
::

::important
Избегайте побочных эффектов в корне `<script setup>`, которые требуют очистки. Пример: `setInterval` в корне — при SSR `onBeforeUnmount` / `onUnmounted` не вызовутся, таймер останется навсегда. Переносите такой код в `onMounted`.
::

::tip{icon="i-lucide-video" to="https://youtu.be/dZSNW07sO-A" target="_blank"}
Видео Дэниела Ро про SSR и глобальное состояние.
::

### Шаг 7: вывод HTML

Когда данные получены и компоненты отрендерены, Nuxt собирает HTML вместе с настройками `unhead` в полный документ. HTML и связанные данные отправляются клиенту — SSR завершён.

::callout{icon="i-lucide-lightbulb"}
После рендера Vue-приложения в HTML Nuxt вызывает хук [`app:rendered`](/docs/3.x/api/advanced/hooks#app-hooks-runtime).
::

::callout{icon="i-lucide-lightbulb"}
Перед финальной отправкой HTML Nitro вызывает [`render:html`](/docs/3.x/api/advanced/hooks#nitro-app-hooks-runtime-server-side) — можно подправить HTML (скрипты, meta и т.д.).
::

## Клиент (браузер)

Эта часть цикла выполняется **полностью в браузере**, независимо от выбранного режима Nuxt.

### Шаг 1: инициализация Nuxt и плагинов

Аналогично серверу: встроенные и пользовательские плагины.

На клиенте выполняются плагины из `plugins/` без суффикса и с `.client` (например `myClientPlugin.client.ts`).

::callout{icon="i-lucide-lightbulb"}
После этого снова вызывается [`app:created`](/docs/3.x/api/advanced/hooks#app-hooks-runtime).
::

:read-more{to="/docs/3.x/directory-structure/plugins"}

### Шаг 2: валидация маршрута

Как на сервере: при наличии `validate` в `definePageMeta`.

### Шаг 3: middleware приложения

Middleware Nuxt выполняется и на сервере, и на клиенте. Чтобы ограничить среду, используйте `import.meta.client` и `import.meta.server`.

:read-more{to="/docs/3.x/directory-structure/middleware#when-middleware-runs"}

### Шаг 4: монтирование Vue и гидратация

Вызов `app.mount('#__nuxt')` монтирует приложение в DOM. При SSR/SSG Vue выполняет **гидратацию**: сопоставляет компоненты с узлами DOM и вешает обработчики (кроме [серверных компонентов](/docs/3.x/directory-structure/components#server-components)).

Чтобы не было ошибок гидратации, данные на сервере и клиенте должны совпадать. Для запросов к API лучше `useAsyncData`, `useFetch` или другие композаблы, дружественные к SSR: данные с сервера переиспользуются при гидратации. Новые запросы без необходимости до гидратации лучше не запускать.

::callout{icon="i-lucide-lightbulb"}
Перед монтированием Nuxt вызывает [`app:beforeMount`](/docs/3.x/api/advanced/hooks#app-hooks-runtime).
::

::callout{icon="i-lucide-lightbulb"}
После монтирования — [`app:mounted`](/docs/3.x/api/advanced/hooks#app-hooks-runtime).
::

### Шаг 5: жизненный цикл Vue

В браузере выполняется полный [жизненный цикл Vue](https://vuejs.org/guide/essentials/lifecycle).

Слои Nuxt — способ делиться и переиспользовать части приложения в монорепозитории, из git или npm. Структура слоя почти как у обычного проекта Nuxt, поэтому его проще писать и поддерживать.

:read-more{title="Слои в Nuxt" to="/docs/4.x/getting-started/layers"}

Минимальный слой содержит [`nuxt.config.ts`](/docs/4.x/directory-structure/nuxt-config), чтобы Nuxt распознал каталог как слой.

```ts [base/nuxt.config.ts]
export default defineNuxtConfig({})
```

Остальные файлы в каталоге слоя при необходимости подхватываются автоматически.

- [`app/components/*`](/docs/4.x/directory-structure/app/components) — компоненты
- [`app/composables/*`](/docs/4.x/directory-structure/app/composables) — композаблы
- [`app/layouts/*`](/docs/4.x/directory-structure/app/layouts) — layouts
- [`app/middleware/*`](/docs/4.x/directory-structure/app/middleware) — middleware
- [`app/pages/*`](/docs/4.x/directory-structure/app/pages) — страницы
- [`app/plugins/*`](/docs/4.x/directory-structure/app/plugins) — плагины
- [`app/utils/*`](/docs/4.x/directory-structure/app/utils) — утилиты
- [`app/app.config.ts`](/docs/4.x/directory-structure/app/app-config) — app config
- [`server/*`](/docs/4.x/directory-structure/server) — серверные эндпоинты и middleware
- [`nuxt.config.ts`](/docs/4.x/directory-structure/nuxt-config) — конфигурация Nuxt

## Простой пример

::code-tree{defaultValue="nuxt.config.ts" expandAll}

  ```ts [nuxt.config.ts]
  export default defineNuxtConfig({
    extends: [
      './base',
    ],
  })
  ```

  ```vue [app/app.vue]
  <template>
    <BaseComponent />
  </template>
  ```

  ```ts [base/nuxt.config.ts]
  export default defineNuxtConfig({
    // Extending from base nuxt.config.ts!
    app: {
      head: {
        title: 'Extending Configs is Fun!',
        meta: [
          { name: 'description', content: 'I am using the extends feature in Nuxt!' },
        ],
      },
    },
  })
  ```

  ```vue [base/app/components/BaseComponent.vue]
  <template>
    <h1>Extending Components is Fun!</h1>
  </template>
  ```

::

## Приоритет слоёв

При нескольких слоях важен порядок переопределения: **более приоритетный** слой перекрывает менее приоритетный при совпадении файлов или компонентов.

От большего приоритета к меньшему:

1. **Файлы вашего проекта** — всегда выше всех
2. **Автопоиск в `~~/layers`** — по алфавиту (Z выше A)
3. **Слои из `extends`** — первый элемент важнее второго

### Когда что использовать

- **`extends`** — внешние зависимости (npm, удалённый репозиторий), слои вне каталога проекта
- **`~~/layers`** — локальные слои внутри проекта

::tip
Чтобы задать порядок автопоиска, префиксы с номерами: `~/layers/1.z-layer`, `~/layers/2.a-layer` — тогда `2.a-layer` важнее `1.z-layer`.
::

### Пример

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  extends: [
    // Local layer outside the project
    '../base',
    // NPM package
    '@my-themes/awesome',
    // Remote repository
    'github:my-themes/awesome#v1',
  ],
})
```

Если есть ещё `~~/layers/custom`, порядок такой:
- файлы проекта (максимум)
- `~~/layers/custom`
- `../base`
- `@my-themes/awesome`
- `github:my-themes/awesome#v1` (минимум)

То есть проект перекрывает любой слой, а `~~/layers/custom` — всё из `extends`.

## Публикация слоёв

Слой можно отдавать через удалённый источник или npm-пакет.

### Git-репозиторий

Примеры `extends`:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  extends: [
    // GitHub Remote Source
    'github:username/repoName',
    // GitHub Remote Source within /base directory
    'github:username/repoName/base',
    // GitHub Remote Source from dev branch
    'github:username/repoName#dev',
    // GitHub Remote Source from v1.0.0 tag
    'github:username/repoName#v1.0.0',
    // GitLab Remote Source example
    'gitlab:username/repoName',
    // Bitbucket Remote Source example
    'bitbucket:username/repoName',
  ],
})
```

::tip
Приватный remote: переменная `GIGET_AUTH=<token>`.
::

::tip
Самохостинг GitHub/GitLab: `GIGET_GITHUB_URL=<url>` или `GIGET_GITLAB_URL=<url>` — либо опция [`auth`](https://github.com/unjs/c12#extending-config-layer-from-remote-sources) в `nuxt.config`.
::

::warning
Зависимости удалённого слоя (например eslint-плагин) не видны вашему eslint: они лежат в `node_modules/.c12/layer_name/node_modules/`, менеджер пакетов туда не смотрит.
::

::note
Чтобы при git-слое установить npm-зависимости слоя, укажите `install: true`:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  extends: [
    ['github:username/repoName', { install: true }],
  ],
})
```
::

### npm-пакет

Слой можно опубликовать как пакет с нужными файлами и зависимостями — для шаринга конфигурации между проектами.

Пакет должен быть в npm и установлен в проекте как devDependency; в `extends` указывается имя модуля:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  extends: [
    // Node Module with scope
    '@scope/moduleName',
    // or just the module name
    'moduleName',
  ],
})
```

В `package.json` слоя заполните поля, чтобы при публикации попали нужные файлы:

```json [package.json]
{
  "name": "my-theme",
  "version": "1.0.0",
  "type": "module",
  "main": "./nuxt.config.ts",
  "dependencies": {},
  "devDependencies": {
    "nuxt": "^3.0.0"
  }
}
```

::important
Все импорты слоя должны быть в **dependencies**. `nuxt` и всё только для локальной проверки слоя — в **devDependencies**.
::

Публикация в npm — публичная или приватная.

::important
Для приватного пакета нужна авторизация в npm при установке.
::

## Советы

### Именованные алиасы слоёв

Автопоиск в `~~/layers` создаёт алиасы: слой `~~/layers/test` доступен как `#layers/test`.

Для других слоёв имя задаётся через `$meta`:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  $meta: {
    name: 'example',
  },
})
```

Появится алиас `#layers/example` на корень слоя.

### Относительные пути и алиасы

Глобальные алиасы `~/` и `@/` в компонентах и композаблах слоя резолвятся от **проекта пользователя**. Обход: **относительные импорты** или именованные алиасы слоёв.

В `nuxt.config` слоя относительные пути (кроме вложенных `extends`) тоже считаются от проекта пользователя. Используйте абсолютные пути:

```ts [nuxt.config.ts]
import { fileURLToPath } from 'node:url'
import { dirname, join } from 'node:path'

const currentDir = dirname(fileURLToPath(import.meta.url))

export default defineNuxtConfig({
  css: [
    join(currentDir, './app/assets/main.css'),
  ],
})
```

## Отключение модулей из слоя

Модуль слоя можно выключить, задав ключ конфигурации модуля в `false`:

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  extends: ['./base-layer'],
  // Disable modules from the layer by setting their config key to false
  image: false, // Disables @nuxt/image
  pinia: false, // Disables @pinia/nuxt
})
```

::note
Ключ задаёт сам модуль: часто `image` для `@nuxt/image`, `pinia` для `@pinia/nuxt`, `content` для `@nuxt/content`. Смотрите документацию модуля.
::

Полезно, когда:
- в слое есть ненужные вам модули
- нужна другая реализация
- аналитику и т.п. надо отключить в части окружений

::tip
Так же можно отключать модули и в самом проекте: `false` останавливает setup модуля, но типы для него всё равно генерируются.
::

## Многослойность в модулях Nuxt

Для обхода всех слоёв в модуле используйте [`getLayerDirectories`](/docs/4.x/api/kit/layers#getlayerdirectories) из Nuxt Kit.

```ts [modules/my-module.ts]
import { defineNuxtModule, getLayerDirectories } from 'nuxt/kit'

export default defineNuxtModule({
  setup (_options, nuxt) {
    const layerDirs = getLayerDirectories()

    for (const [index, layer] of layerDirs.entries()) {
      console.log(`Layer ${index}:`)
      console.log(`  Root: ${layer.root}`)
      console.log(`  App: ${layer.app}`)
      console.log(`  Server: ${layer.server}`)
      console.log(`  Pages: ${layer.appPages}`)
      // ... other directories
    }
  },
})
```

**Замечания:**
- Ранние элементы массива приоритетнее и перекрывают последующие
- Первый элемент — проект пользователя

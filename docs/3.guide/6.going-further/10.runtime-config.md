## Публикация значений

Чтобы передать настройки и переменные окружения в приложение, опишите их в [`nuxt.config`](/docs/4.x/directory-structure/nuxt-config) в [`runtimeConfig`](/docs/4.x/api/nuxt-config#runtimeconfig).

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  runtimeConfig: {
    // The private keys which are only available within server-side
    apiSecret: '123',
    // Keys within public, will be also exposed to the client-side
    public: {
      apiBase: '/api',
    },
  },
})
```

Поля в `runtimeConfig.public` (например `apiBase`) попадают в payload страниц и доступны и на сервере, и в браузере.

```ts
const runtimeConfig = useRuntimeConfig()

console.log(runtimeConfig.apiSecret)
console.log(runtimeConfig.public.apiBase)
```

::tip
Публичный runtime config в шаблонах Vue: `$config.public`.
::

### Сериализация

Перед передачей в Nitro runtime config сериализуется. Функции, `Set`, `Map` и подобное в `nuxt.config` задавать нельзя.

Несериализуемую логику перенесите в плагин или middleware Nuxt/Nitro.

### Переменные окружения

Чаще всего конфиг задают через переменные окружения.

::note
Nuxt CLI подхватывает `.env` в dev, build и generate. Уже **собранный** сервер `.env` сам не читает.
:read-more{to="/docs/4.x/directory-structure/env"}
::

Значения runtime config **подставляются из подходящих переменных окружения в runtime**.

Условия два:

1. Ключи должны быть объявлены в `nuxt.config`, чтобы случайные переменные не утекли в код.

2. Переменная окружения в верхнем регистре, с префиксом `NUXT_`, сегменты разделяются `_`, отражая вложенность ключей.

::warning
Если умолчание в `runtimeConfig` брать из *другой* переменной (например `myVar` из `process.env.OTHER_VARIABLE`), это сработает только на этапе сборки и сломается в runtime. Лучше именовать переменные в духе структуры `runtimeConfig`.
::

::tip{icon="i-lucide-video" to="https://youtu.be/_FYV5WfiWvs" target="_blank"}
Видео Alexander Lichter о типичной ошибке с runtimeConfig.
::

#### Пример

```ini [.env]
NUXT_API_SECRET=api_secret_token
NUXT_PUBLIC_API_BASE=https://nuxtjs.org
```

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  runtimeConfig: {
    apiSecret: '', // can be overridden by NUXT_API_SECRET environment variable
    public: {
      apiBase: '', // can be overridden by NUXT_PUBLIC_API_BASE environment variable
    },
  },
})
```

## Чтение

### Vue-приложение

В клиентской части Nuxt вызывайте [`useRuntimeConfig()`](/docs/4.x/api/composables/use-runtime-config).

::important
Поведение на клиенте и сервере разное:

- На клиенте доступны только `runtimeConfig.public` и `runtimeConfig.app` (внутреннее у Nuxt); объект реактивный и перезаписываемый.

- На сервере доступен весь runtime config, объект только для чтения, чтобы не смешивать контексты запросов.
::

```vue [app/pages/index.vue]
<script setup lang="ts">
const config = useRuntimeConfig()

console.log('Runtime config:', config)
if (import.meta.server) {
  console.log('API secret:', config.apiSecret)
}
</script>

<template>
  <div>
    <div>Check developer console!</div>
  </div>
</template>
```

::caution
**Безопасность:** не выводите на клиент секретные ключи runtime config и не кладите их в `useState`.
::

### Плагины

В плагине вызывайте [`useRuntimeConfig()`](/docs/4.x/api/composables/use-runtime-config) внутри `defineNuxtPlugin`.

```ts [app/plugins/config.ts]
export default defineNuxtPlugin((nuxtApp) => {
  const config = useRuntimeConfig()

  console.log('API base URL:', config.public.apiBase)
})
```

### Серверные маршруты

В обработчиках используйте `useRuntimeConfig`.

```ts [server/api/test.ts]
export default defineEventHandler(async (event) => {
  const { apiSecret } = useRuntimeConfig(event)
  const result = await $fetch('https://my.api.com/test', {
    headers: {
      Authorization: `Bearer ${apiSecret}`,
    },
  })
  return result
})
```

::note
Аргумент `event` у `useRuntimeConfig` необязателен, но с ним для server routes корректнее подхватываются переопределения из [переменных окружения](/docs/4.x/guide/going-further/runtime-config#environment-variables) в runtime.
::

## Типизация

Nuxt генерирует интерфейс TypeScript из `runtimeConfig` через [unjs/untyped](https://github.com/unjs/untyped).

Типы можно задать вручную:

```ts [index.d.ts]
declare module 'nuxt/schema' {
  interface RuntimeConfig {
    apiSecret: string
  }
  interface PublicRuntimeConfig {
    apiBase: string
  }
}
// It is always important to ensure you import/export something when augmenting a type
export {}
```

::note
`nuxt/schema` — для приложений. Авторы модулей расширяют `@nuxt/schema`.
::

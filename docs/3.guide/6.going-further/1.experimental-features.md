В Nuxt есть экспериментальные возможности, которые включаются в файле конфигурации.

Внутри они описаны в `@nuxt/schema`. Подробности — в [документации API](/docs/4.x/guide/going-further/experimental-features) и [исходном коде](https://github.com/nuxt/nuxt/blob/main/packages/schema/src/config/experimental.ts).

::note
Экспериментальные функции могут быть удалены или изменены в будущем.
::

## alwaysRunFetchOnKeyChange

Запускать ли `useFetch` при смене ключа, даже при `immediate: false`, если запрос ещё не выполнялся.

`useFetch` и `useAsyncData` при смене ключа всегда выполняются, если `immediate: true` или запрос уже был.

По умолчанию выключено; можно включить:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    alwaysRunFetchOnKeyChange: true,
  },
})
```

## appManifest

Манифесты приложения, чтобы на клиенте соблюдались route rules.

По умолчанию включено; можно отключить:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    appManifest: false,
  },
})
```

## asyncContext

Нативный async context для вложенных композаблов в Nuxt и Nitro: можно вызывать композаблы внутри async и реже получать ошибку `Nuxt instance is unavailable`.

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    asyncContext: true,
  },
})
```

## extractAsyncDataHandlers

Выносит обработчики из вызовов `useAsyncData` и `useLazyAsyncData` в отдельные чанки для лучшего code splitting и кеширования.

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    extractAsyncDataHandlers: true,
  },
})
```

Inline-обработчики превращаются в динамические импорты:

```vue
<!-- Before -->
<script setup>
const { data } = await useAsyncData('user', async () => {
  return await $fetch('/api/user')
})
</script>
```

```vue
<!-- After transformation -->
<script setup>
const { data } = await useAsyncData('user', () =>
  import('/generated-chunk.js').then(r => r.default()),
)
</script>
```

Так логику загрузки данных можно вынести в отдельный чанк и при необходимости подгрузить.

::important
Рекомендуется только для **статических сборок** с извлечением payload, когда данные не нужно заново запрашивать в runtime.
::

## emitRouteChunkError

Вызывает хук `app:chunkError` при ошибке загрузки чанка Vite/webpack. По умолчанию при навигации на новый маршрут и падении чанка выполняется перезагрузка нового маршрута.

В режиме `automatic` Nuxt также перезагружает новый маршрут при ошибке чанка при переходе.

`automatic-immediate` — перезагрузка текущего маршрута сразу при ошибке чанка (без ожидания навигации). Полезно, если ошибка не из-за перехода, например не загрузился [lazy-компонент](/docs/4.x/directory-structure/app/components#dynamic-imports). Минус — лишние перезагрузки, если проблемный чанк приложению не нужен.

`false` отключает автообработку; `manual` — обрабатывать ошибки чанков самостоятельно.

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    emitRouteChunkError: 'automatic', // or 'automatic-immediate', 'manual' or false
  },
})
```

```ts twoslash [nuxt.config.ts]
    enforceModuleCompatibility: true,
  },

```

## restoreState

Восстанавливать состояние приложения из `sessionStorage` после перезагрузки из-за ошибки чанка или вызова [`reloadNuxtApp()`](/docs/4.x/api/utils/reload-nuxt-app).

Чтобы не ломать гидратацию, применяется только после монтирования Vue — возможен краткий «миг» при первой загрузке.

::important
Включайте осознанно: возможно неожиданное поведение.
Задавайте явные ключи [`useState`](/docs/4.x/api/composables/use-state) — автогенерируемые могут не совпадать между сборками.
::

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    restoreState: true,
  },
})
```

## payloadExtraction

Как отдаётся payload для пререндеренных и закешированных (ISR/SWR) страниц.

- `'client'` — payload в HTML при первом ответе сервера, для клиентской навигации — в `_payload.json`. Меньше запросов при первой загрузке, навигация остаётся эффективной.
- `true` — отдельный `_payload.json` и для первого ответа, и для навигации.
- `false` — без извлечения: payload только в HTML, файлов `_payload.json` нет.

По умолчанию `true`, при `compatibilityVersion: 5` — `'client'`.

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    // Payload в HTML, извлечение только для клиентской навигации
    payloadExtraction: 'client',
  },
})
```

Извлечение payload работает и для ISR/SWR: CDN может кешировать `_payload.json` вместе с HTML и ускорять клиентскую навигацию.

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    payloadExtraction: 'client',
  },
  routeRules: {
    // Файлы payload для этих кешируемых маршрутов
    '/products/**': { isr: 3600 },
    '/blog/**': { swr: true },
  },
})
```

## typedPages

Включить экспериментальный типизированный роутер на базе [`unplugin-vue-router`](https://github.com/posva/unplugin-vue-router).

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    typedPages: true,
  },
})
```

Из коробки появляется типизация для [`navigateTo`](/docs/4.x/api/utils/navigate-to), [`<NuxtLink>`](/docs/4.x/api/components/nuxt-link), [`router.push()`](/docs/4.x/api/composables/use-router) и др.

::important
Если используете `pnpm` без `shamefully-hoist=true`, добавьте `unplugin-vue-router` в `publicHoistPattern` в `pnpm-workspace.yaml`, иначе функция не заработает.
```yaml
publicHoistPattern:
  - "unplugin-vue-router"
```
::

## watcher

Альтернативный watcher для отслеживания файлов в Nuxt.

По умолчанию — `chokidar-granular`: игнорирует верхнеуровневые каталоги вроде `node_modules` и `.git`, если они исключены из watching.

Можно указать `parcel` и использовать `@parcel/watcher` — иногда быстрее на больших проектах или Windows.

Или `chokidar`, чтобы следить за всеми файлами в исходниках.

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    watcher: 'chokidar-granular', // 'chokidar' or 'parcel' are also options
  },
})
```

## sharedPrerenderData

Nuxt автоматически разделяет *данные* payload между пререндеренными страницами. Это заметно ускоряет пререндер, если на разных страницах через `useAsyncData` или `useFetch` запрашиваются одни и те же данные.

При необходимости можно отключить.

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    sharedPrerenderData: false,
  },
})
```

:video-accordion{title="Видео Alexander Lichter об экспериментальной опции sharedPrerenderData" videoId="1jUupYHVvrU"}

Важно: при включении у каждого уникального ключа данные должны однозначно соответствовать одному и тому же результату. Например, для `useAsyncData` по данным конкретной страницы задайте ключ, уникальный для этих данных (`useFetch` обычно делает это сам).

```ts
// This would be unsafe in a dynamic page (e.g. `[slug].vue`) because the route slug makes a difference
// to the data fetched, but Nuxt can't know that because it's not reflected in the key.
const route = useRoute()
const { data } = await useAsyncData(async (_nuxtApp, { signal }) => {
  return await $fetch(`/api/my-page/${route.params.slug}`, { signal })
})
// Instead, you should use a key that uniquely identifies the data fetched.
const { data } = await useAsyncData(route.params.slug, async (_nuxtApp, { signal }) => {
  return await $fetch(`/api/my-page/${route.params.slug}`, { signal })
})
```

## clientNodeCompat

Nuxt автоматически полифиллит импорты Node.js в клиентской сборке через [`unenv`](https://github.com/unjs/unenv).

::note
Чтобы глобалы вроде `Buffer` работали в браузере, их нужно вручную подключить к `globalThis`.

```ts
import { Buffer } from 'node:buffer'

globalThis.Buffer ||= Buffer
```

## scanPageMeta

На этапе сборки Nuxt отдаёт модулям часть метаданных маршрута из `definePageMeta`: `alias`, `name`, `path`, `redirect`, `props`, `middleware`.

Работает только со статическими значениями, строками и массивами — не с переменными и условиями. Контекст — в [исходном issue](https://github.com/nuxt/nuxt/issues/24770).

По умолчанию метаданные страниц сканируются после регистрации всех маршрутов в `pages:extend`, затем вызывается хук `pages:resolved`.

Можно отключить, если мешает в проекте.

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    scanPageMeta: false,
  },
})
```

## buildCache

Кеширует артефакты сборки Nuxt по хешу конфигурации и исходников.

Учитываются файлы внутри `srcDir` и `serverDir` для Vue- и Nitro-частей приложения.

По умолчанию выключено; можно включить:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    buildCache: true,
  },
})
```

При включении изменение перечисленных файлов вызывает полную пересборку:

```bash [Directory structure]
.nuxtrc
.npmrc
package.json
package-lock.json
yarn.lock
pnpm-lock.yaml
tsconfig.json
bun.lock
bun.lockb
```

Кроме того, изменения файлов в `srcDir` пересобирают клиентский и серверный Vue-бандлы. Nitro всегда пересобирается (ведётся работа над кешируемыми артефактами Nitro и их хешами).

::note
Хранится не более 10 tar-архивов кеша.
::

## checkOutdatedBuildInterval

Интервал проверки новых сборок (мс). Не работает, если `experimental.appManifest` равен `false`.

`false` — отключить проверку.

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    checkOutdatedBuildInterval: 3600000, // 1 hour, or false to disable
  },
})
```

## extraPageMetaExtractionKeys

Макрос `definePageMeta()` задаёт метаданные страниц на этапе сборки. Nuxt поддерживает фиксированный набор ключей для редиректов, алиасов и кастомных путей.

Эта опция добавляет свои ключи к извлечению метаданных при `scanPageMeta`.

```vue
<script lang="ts" setup>
definePageMeta({
  foo: 'bar',
})
</script>
```

```ts
export default defineNuxtConfig({
  experimental: {
    extraPageMetaExtractionKeys: ['foo'],
  },
  hooks: {
    'pages:resolved' (ctx) {
      // ✅ foo is available
    },
  },
})
```

Модули могут читать эти поля в контексте сборки. В модуле имеет смысл [расширить типы `NuxtPage` своими ключами](/docs/4.x/directory-structure/app/pages#typing-custom-metadata).

## normalizeComponentNames

Nuxt подстраивает автогенерируемые имена компонентов Vue под полное имя, как при автоимпорте.

При проблемах можно отключить.

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    normalizeComponentNames: false,
  },
})
```

Если имя не задано вручную, Vue по умолчанию берёт имя из имени файла компонента.

```bash [Directory structure]
├─ components/
├─── SomeFolder/
├───── MyComponent.vue
```

Тогда для Vue имя — `MyComponent`: его видит `<KeepAlive>` и Vue DevTools.

Для автоимпорта нужно имя `SomeFolderMyComponent`.

С `experimental.normalizeComponentNames` оба совпадают: Vue даёт имя в стиле Nuxt.

## normalizePageNames

Имена компонентов страниц совпадают с именами маршрутов: задаётся `__name`, чтобы `<KeepAlive>` отличал страницы по имени.

По умолчанию Vue берёт имя из файла: у `pages/foo/index.vue` и `pages/bar/index.vue` оба будут `index`, и фильтрация `<KeepAlive>` по имени ломается.

С `normalizePageNames` имена совпадают с маршрутом (`foo`, `bar`), можно использовать `include`/`exclude` без `defineOptions({ name: '...' })` на каждой странице.

Включается при `future.compatibilityVersion` ≥ `5`; при необходимости отключите:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    normalizePageNames: false,
  },
})
```

```vue [app.vue]
<template>
  <NuxtPage :keepalive="{ include: ['foo'] }" />
</template>
```

## spaLoadingTemplateLocation

Для клиентских страниц (`ssr: false`) можно показать экран загрузки из `~/spa-loading-template.html`.

Значение `within` — внутри корня:

```html
<div id="__nuxt">
  <!-- spa loading template -->
</div>
```

Или `body` — шаблон рядом с корнем приложения:

```html
<div id="__nuxt"></div>
<!-- spa loading template -->
```

Так меньше белой вспышки при гидратации клиент-only страницы.

## lazyHydration

Стратегии гидратации для `<Lazy>`: откладывают гидратацию до необходимости и ускоряют первый рендер.

По умолчанию включено; можно отключить:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    lazyHydration: false,
  },
})
```

## templateImportResolution

Отключить разрешение импортов в шаблонах Nuxt относительно модуля, добавившего шаблон.

По умолчанию Nuxt резолвит импорты в шаблонах от каталога этого модуля. `false` отключает это — полезно при конфликтах резолва.

По умолчанию включено; можно отключить:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    templateImportResolution: false,
  },
})
```

## templateRouteInjection

Объект маршрута из автоимпортируемого `useRoute()` синхронизирован с текущей страницей в `<NuxtPage>`. У экспортируемого `vue-router` `useRoute` и у `$route` в шаблонах так не бывает.

При включении подмешивается mixin, чтобы `$route` в шаблонах совпадал с `useRoute()` из Nuxt.

По умолчанию включено; можно отключить:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    templateRouteInjection: false,
  },
})
```

## decorators

Синтаксис декораторов во всём приложении Nuxt/Nitro.

Со сборщиком Vite (по умолчанию) декораторы обрабатываются [Babel](https://babeljs.io/) и [`@babel/plugin-proposal-decorators`](https://babeljs.io/docs/babel-plugin-proposal-decorators). С webpack или rspack — через [esbuild](https://github.com/evanw/esbuild/releases/tag/v0.21.3).

Раньше TypeScript поддерживал декораторы через `compilerOptions.experimentalDecorators` до стандарта TC39. Сейчас это [предложение Stage 3](https://github.com/tc39/proposal-decorators), в TS 5.0+ без отдельной настройки (см. https://github.com/microsoft/TypeScript/pull/52582 и https://devblogs.microsoft.com/typescript/announcing-typescript-5-0-beta/#decorators).

`experimental.decorators` включает именно TC39-вариант, **а не** старый `compilerOptions.experimentalDecorators`.

::warning
До финального стандарта в JS спецификация ещё может меняться.
::

### Использование

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    decorators: true,
  },
})
```

Для Vite или серверной сборки Nitro установите дополнительные пакеты Babel как dev-зависимости:

::code-group
```bash [npm]
npm install -D @babel/plugin-proposal-decorators @babel/plugin-syntax-jsx
```
```bash [pnpm]
pnpm add -D @babel/plugin-proposal-decorators @babel/plugin-syntax-jsx
```
```bash [yarn]
yarn add -D @babel/plugin-proposal-decorators @babel/plugin-syntax-jsx
```
::

::tip
Если пакетов нет, Nuxt предложит установить их автоматически.
::

```ts [app/app.vue]
function something (_method: () => unknown) {
  return () => 'decorated'
}

class SomeClass {
  @something
  public someMethod () {
    return 'initial'
  }
}

const value = new SomeClass().someMethod()
// this will return 'decorated'
```

## defaults

Значения по умолчанию для встроенных компонентов и композаблов Nuxt.

В будущем их, скорее всего, перенесут, например в `app.config` или каталог `app/`.

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    defaults: {
      nuxtLink: {
        componentName: 'NuxtLink',
        prefetch: true,
        prefetchOn: {
          visibility: true,
        },
      },
      useAsyncData: {
        deep: true,
      },
      useState: {
        resetOnClear: true,
      },
    },
  },
})
```

`useState.resetOnClear` задаёт, сбрасывает ли [`clearNuxtState`](/docs/4.x/api/utils/clear-nuxt-state) состояние к начальному значению из `init` в [`useState`](/docs/4.x/api/composables/use-state), а не в `undefined`. При `compatibilityVersion: 5` по умолчанию `true`.

## purgeCachedData

Очищать ли кеши static и asyncData при смене маршрута.

Nuxt сбрасывает кеш `useAsyncData` и `nuxtApp.static.data`, что снижает утечки памяти и подгружает актуальные данные; при необходимости отключите.

По умолчанию включено; можно отключить:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    purgeCachedData: false,
  },
})
```

## entryImportMap

По умолчанию Nuxt стабилизирует чанки через import map для entry-чанка бандла.

В начало `<head>` вставляется import map:

```html
<script type="importmap">{"imports":{"#entry":"/_nuxt/DC5HVSK5.js"}}</script>
```

В чанках скриптов от Vite импорты идут с `#entry`: смена entry не инвалидирует остальные неизменённые чанки.

::note
Nuxt отключает это, если в `vite.build.target` есть браузер без import maps или если `vite.build.rolldownOptions.output.entryFileNames` без `[hash]`.
::

При необходимости отключите вручную:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    entryImportMap: false,
  },
  // or, better, simply tell vite your desired target
  // which nuxt will respect
  vite: {
    build: {
      target: 'safari13',
    },
  },
})
```

## typescriptPlugin

Расширенный DX для TypeScript через модуль `@dxup/nuxt`.

Экспериментальный плагин улучшает интеграцию TypeScript и инструменты разработки в Nuxt.

По умолчанию выключено; можно включить:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    typescriptPlugin: true,
  },
})
```

::important
Требования:
- зависимость `typescript` в проекте;
- в VS Code включить использование TypeScript из workspace ([документация VS Code](https://code.visualstudio.com/docs/typescript/typescript-compiling#_using-the-workspace-version-of-typescript)).

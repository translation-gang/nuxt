Это руководство помогает разобраться во внутреннем устройстве Nuxt при разработке решений и модульных интеграций.

## Интерфейс Nuxt

При [`nuxt dev`](/docs/4.x/api/commands/dev) или [`nuxt build`](/docs/4.x/api/commands/build) создаётся общий контекст — внутри он называется `nuxt`. В нём нормализованные опции из `nuxt.config`, внутреннее состояние и [система хуков](/docs/4.x/api/advanced/hooks) на [unjs/hookable](https://github.com/unjs/hookable), через которую части сборки обмениваются сообщениями. Условно это **ядро сборки**.

К нему обращаются композаблы [Nuxt Kit](/docs/4.x/guide/going-further/kit). В одном процессе допустим только один экземпляр Nuxt.

Расширять интерфейс Nuxt и этапы сборки можно [модулями Nuxt](/docs/4.x/guide/modules).

Подробнее — [исходный код](https://github.com/nuxt/nuxt/blob/main/packages/nuxt/src/core/nuxt.ts).

## Интерфейс NuxtApp

При рендере в браузере или на сервере создаётся общий контекст `nuxtApp`: экземпляр Vue, runtime-хуки, `ssrContext`, payload для гидратации и т.д. Это **ядро runtime**.

Доступ через [`useNuxtApp()`](/docs/4.x/api/composables/use-nuxt-app) в плагинах, `<script setup>` и композаблах Vue. Глобально на клиенте можно, на сервере нельзя — чтобы не смешивать контексты пользователей.

Если [`useNuxtApp`](/docs/4.x/api/composables/use-nuxt-app) бросает исключение из‑за отсутствия контекста, а `nuxtApp` не обязателен, используйте [`tryUseNuxtApp`](/docs/4.x/api/composables/use-nuxt-app#tryusenuxtapp) — вернёт `null`.

Расширять `nuxtApp` и реагировать на этапы runtime позволяют [плагины Nuxt](/docs/4.x/directory-structure/app/plugins).

См. [Nuxt App](/docs/4.x/api/composables/use-nuxt-app).

`nuxtApp` включает в том числе:

```ts
interface NuxtApp {
  vueApp // the global Vue application: https://vuejs.org/api/application.html#application-api

  versions // an object containing Nuxt and Vue versions

  // These let you call and add runtime NuxtApp hooks
  // https://github.com/nuxt/nuxt/blob/main/packages/nuxt/src/app/nuxt.ts#L18
  hooks
  hook
  callHook

  // Only accessible on server-side
  ssrContext: {
    url
    req
    res
    runtimeConfig
    noSSR
  }

  // This will be stringified and passed from server to client
  payload: {
    serverRendered: true
    data: {}
    state: {}
  }

  provide: (name: string, value: any) => void
}
```

Исходники: [nuxt.ts](https://github.com/nuxt/nuxt/blob/main/packages/nuxt/src/app/nuxt.ts).

## Runtime и сборка

Nuxt собирает проект в Node.js и имеет отдельный runtime.

Контексты изолированы: runtime не должен делить с этапом сборки состояние, код или контекст, кроме runtime-конфигурации.

`nuxt.config` и [модули](/docs/4.x/guide/modules) расширяют сборку, [плагины](/docs/4.x/directory-structure/app/plugins) — runtime.

При `nuxt build` для production в `.output` попадает автономная сборка без зависимости от `nuxt.config` и [модулей](/docs/4.x/guide/modules) в runtime.

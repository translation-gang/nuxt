---
title: Старый Composition API
description: 'Как мигрировать на Composition API с Nuxt Bridge.'
---

Nuxt Bridge даёт доступ к синтаксису Composition API и рассчитан на согласованность с Nuxt 3. Поэтому при включении Bridge, если вы уже использовали Composition API, потребуется несколько дополнительных шагов.

## Удалите модули

- Удалите `@vue/composition-api` из зависимостей.
- Удалите `@nuxtjs/composition-api` из зависимостей (и из `modules` в `nuxt.config`).

## Использование `@vue/composition-api`

Если вы использовали только `@vue/composition-api`, без `@nuxtjs/composition-api`, всё просто.

1. Удалите плагин, где вручную регистрируется Composition API — Nuxt Bridge сделает это за вас.

   ```diff
   - import Vue from 'vue'
   - import VueCompositionApi from '@vue/composition-api'
   -
   - Vue.use(VueCompositionApi)
   ```

2. Больше ничего не требуется. При желании можно убрать явные импорты из `@vue/composition-api` и полагаться на автоимпорт от Nuxt Bridge.

## Миграция с `@nuxtjs/composition-api`

Nuxt Bridge реализует Composition API иначе, чем `@nuxtjs/composition-api`, и предлагает другие композаблы (выровненные с Nuxt 3).

Часть композаблов удалена и пока без замены — процесс будет чуть сложнее.

### Удалите `@nuxtjs/composition-api/module` из `buildModules`

Импорты можно не менять сразу — Nuxt Bridge подставит «shim» для большинства текущих импортов, чтобы дать время перейти на новые композаблы, совместимые с Nuxt 3, за исключением:

- `withContext` удалён. См. [ниже](/docs/4.x/bridge/nuxt3-compatible-api#usecontext-and-withcontext).
- `useStatic` удалён. Замены пока нет. Можно обсудить кейс в discussion, если он нужен.
- `reqRef` и `reqSsrRef` (устаревшие) полностью удалены. См. ниже про [ssrRef](/docs/4.x/bridge/nuxt3-compatible-api#ssrref-and-shallowssrref).

### Установите `bridge.capi`

```ts
import { defineNuxtConfig } from '@nuxt/bridge'

export default defineNuxtConfig({
  bridge: {
    capi: true,
    nitro: false, // If migration to Nitro is complete, set to true
  },
})
```

Для остальных композаблов из `@nuxtjs/composition-api` следуйте шагам ниже.

### useFetch

`$fetchState` и `$fetch` удалены.

```diff
const {
- $fetch,
- $fetchState,
+ fetch,
+ fetchState,
} = useFetch(() => { posts.value = await $fetch('/api/posts') })
```

### `defineNuxtMiddleware`

Это была заглушка для типов — теперь удалена.

Уберите обёртку `defineNuxtMiddleware`:

```diff
- import { defineNuxtMiddleware } from '@nuxtjs/composition-api`
- export default defineNuxtMiddleware((ctx) => {})
+ export default (ctx) => {}
```

Для поддержки TypeScript можно использовать `@nuxt/types`:

```ts
import type { Middleware } from '@nuxt/types'

export default <Middleware> function (ctx) { }
```

### `defineNuxtPlugin`

Это была заглушка для типов — теперь удалена.

Можно по-прежнему использовать плагины в стиле Nuxt 2, убрав обёртку (как для [defineNuxtMiddleware](/docs/4.x/bridge/bridge-composition-api#definenuxtmiddleware)).

Уберите обёртку `defineNuxtPlugin`:

```diff
- import { defineNuxtPlugin } from '@nuxtjs/composition-api'
- export default defineNuxtPlugin((ctx, inject) => {})
+ export default (ctx, inject) => {}
```

Для поддержки TypeScript можно использовать `@nuxt/types`:

```ts
import type { Plugin } from '@nuxt/types'

export default <Plugin> function (ctx, inject) {}
```

::warning
Пример корректен, но в Nuxt 3 появляется новая функция defineNuxtPlugin с немного другой сигнатурой.
::

:ReadMore{link="/docs/4.x/directory-structure/app/plugins#creating-plugins"}

### `useRouter` и `useRoute`

В Nuxt Bridge есть прямые аналоги через [`useRouter`](/docs/4.x/api/composables/use-router) и `useRoute`.

Главное отличие: [`useRoute`](/docs/4.x/api/composables/use-route) больше не возвращает computed.

```diff
- import { useRouter, useRoute } from '@nuxtjs/composition-api'

  const router = useRouter()
  const route = useRoute()

- console.log(route.value.path)
+ console.log(route.path)
```

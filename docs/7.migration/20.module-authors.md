## Совместимость модулей

В Nuxt 3 есть базовый слой совместимости с модулями Nuxt 2 через автообёртки `@nuxt/kit`. Обычно всё равно нужны шаги для полной совместимости с Nuxt 3; иногда требуется Nuxt Bridge.

Есть [отдельное руководство](/docs/4.x/guide/modules) по модулям для Nuxt 3 на `@nuxt/kit`. Сейчас лучший путь — следовать ему и переписать модуль. Ниже — подготовительные шаги, если полный рефакторинг пока нежелателен, но нужна совместимость с Nuxt 3.

::tip{icon="i-lucide-puzzle" to="/modules"}
Модули, совместимые с Nuxt 3.
::

### Совместимость плагинов

Плагины Nuxt 3 **не** полностью обратно совместимы с Nuxt 2.

:read-more{to="/docs/4.x/directory-structure/app/plugins"}

### Совместимость с Vue

Плагины или компоненты на Composition API должны явно поддерживать Vue 2 или Vue 3.

С [vue-demi](https://github.com/vueuse/vue-demi) можно сделать код совместимым с Nuxt 2 и 3.

## Миграция модуля

Когда пользователи Nuxt 3 подключают ваш модуль, контейнера модуля (`this.*`) нет — используйте утилиты из `@nuxt/kit`.

### Тестирование с `@nuxt/bridge`

Переход на `@nuxt/bridge` — первый и главный шаг к поддержке Nuxt 3.

Если в модуле есть fixture или пример, добавьте пакет `@nuxt/bridge` в его конфиг (см. [пример](/docs/4.x/bridge/overview#update-nuxtconfig))

### С CommonJS на ESM

Nuxt 3 нативно поддерживает TypeScript и ECMAScript Modules. См. [нативные ES-модули](/docs/4.x/guide/concepts/esm).

### Обязательный default export у плагинов

Если подключаемый плагин без `export default` (например глобальный плагин Vue), добавьте в конец `export default () => { }`.

::code-group

```js [Before]
// ~/plugins/vuelidate.js
import Vue from 'vue'
import Vuelidate from 'vuelidate'

Vue.use(Vuelidate)
```

```js [After]
// ~/plugins/vuelidate.js
import Vue from 'vue'
import Vuelidate from 'vuelidate'

Vue.use(Vuelidate)

export default () => { }
```

::

### Избегайте runtime-модулей

В Nuxt 3 Nuxt — зависимость только времени сборки; модули не должны вешаться на runtime Nuxt.

Модуль должен работать, если добавлен только в [`buildModules`](/docs/4.x/api/nuxt-config#runtimeconfig) (а не в `modules`). Например:

- Не обновляйте `process.env` в модуле и не читайте это в плагине; используйте [`runtimeConfig`](/docs/4.x/api/nuxt-config#runtimeconfig).
- (*) Не полагайтесь на runtime-хуки вроде `vue-renderer:*` в продакшене
- (*) Не добавляйте `serverMiddleware` через импорт внутри модуля; указывайте путь к файлу, чтобы middleware не зависел от контекста модуля

(*) Допустимо только для `nuxt dev` и под `if (nuxt.options.dev) { }`.

::tip
Продолжайте в [руководстве для авторов модулей](/docs/4.x/guide/modules).
::

### TypeScript (по желанию)

Не обязательно, но экосистема Nuxt всё чаще на TypeScript — имеет смысл планировать миграцию.

::tip
Начните с переименования `.js` в `.ts` — TypeScript допускает постепенное внедрение.
::

::tip
Синтаксис TypeScript для модулей и плагинов Nuxt 2 и 3 можно использовать без лишних зависимостей.
::

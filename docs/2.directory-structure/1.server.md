Nuxt автоматически сканирует файлы в этих каталогах и регистрирует API и серверные обработчики с поддержкой горячей замены модулей (HMR).

```bash [Directory structure]
-| server/
---| api/
-----| hello.ts      # /api/hello
---| routes/
-----| bonjour.ts    # /bonjour
---| middleware/
-----| log.ts        # log all requests
```

Каждый файл должен экспортировать функцию по умолчанию, объявленную через `defineEventHandler()` или `eventHandler()` (алиас).

Обработчик может напрямую вернуть JSON, `Promise` или отправить ответ через `event.node.res.end()`.

```ts twoslash [server/api/hello.ts]
export default defineEventHandler((event) => {
  return {
    hello: 'world',
  }
})
```

Этот API можно вызывать из страниц и компонентов:

```vue [app/pages/index.vue]
<script setup lang="ts">
const { data } = await useFetch('/api/hello')
</script>

<template>
  <pre>{{ data }}</pre>
</template>
```

## Серверные маршруты

Файлы в `~~/server/api` автоматически получают префикс `/api` в пути маршрута.

:video-accordion{title="Видео Vue School про маршруты API" videoId="761468863" platform="vimeo"}

Чтобы добавить серверные маршруты без префикса `/api`, поместите их в каталог `~~/server/routes`.

**Пример:**

```ts [server/routes/hello.ts]
export default defineEventHandler(() => 'Hello World!')
```

В примере выше маршрут `/hello` будет доступен по адресу <http://localhost:3000/hello>.

::note
Сейчас серверные маршруты не поддерживают весь набор возможностей динамических маршрутов, как [страницы](/docs/4.x/directory-structure/app/pages#dynamic-routes).
::

## Серверный middleware

Nuxt автоматически подхватывает файлы в `~~/server/middleware` и создаёт для проекта серверный middleware.

Обработчики middleware выполняются при каждом запросе до остальных серверных маршрутов: можно добавлять или проверять заголовки, логировать запросы или расширять объект запроса события.

::note
Обработчики middleware не должны ничего возвращать (и не должны закрывать или отвечать на запрос) — только проверять или расширять контекст запроса либо выбрасывать ошибку.
::

**Примеры:**

```ts [server/middleware/log.ts]
export default defineEventHandler((event) => {
  console.log('New request: ' + getRequestURL(event))
})
```

```ts [server/middleware/auth.ts]
export default defineEventHandler((event) => {
  event.context.auth = { user: 123 }
})
```

## Плагины сервера

Nuxt автоматически читает файлы в `~~/server/plugins` и регистрирует их как плагины Nitro. Так можно расширять поведение рантайма Nitro и подключаться к событиям жизненного цикла.

**Пример:**

```ts [server/plugins/nitroPlugin.ts]
export default defineNitroPlugin((nitroApp) => {
  console.log('Nitro plugin', nitroApp)
})
```

:read-more{to="https://nitro.build/guide/plugins" title="Плагины Nitro" target="_blank"}

## Утилиты сервера

Серверные маршруты работают на [h3js/h3](https://github.com/h3js/h3) с набором вспомогательных функций.

:read-more{to="https://www.jsdocs.io/package/h3#package-index-functions" title="Доступные хелперы запросов H3" target="_blank"}

Свои хелперы можно добавить в каталог `~~/server/utils`.

Например, можно обернуть исходный обработчик и выполнить дополнительные действия до возврата ответа.

**Пример:**

```ts [server/utils/handler.ts]
export const defineWrappedResponseHandler = <T extends EventHandlerRequest, D> (
  handler: EventHandler<T, D>,
): EventHandler<T, D> =>
  defineEventHandler<T>(async (event) => {
    try {
      // do something before the route handler
      const response = await handler(event)
      // do something after the route handler
      return { response }
    } catch (err) {
      // Error handling
      return { err }
    }
  })
```

```ts [server/api/hello.get.ts]
export default defineWrappedResponseHandler(event => 'hello world')
```

## Алиас сервера

Алиас `#server` позволяет импортировать файлы из любого места внутри `server/`, независимо от расположения импортирующего файла.

```ts [server/api/users/[id]/profile.ts]
// Instead of relative paths like this:
// import { formatUser } from '../../../utils/formatUser'

// Use the #server alias:
import { formatUser } from '#server/utils/formatUser'
```

Алиас обеспечивает единообразные импорты в серверном коде, что особенно удобно в глубоко вложенных обработчиках.

::note
Алиас `#server` можно использовать только внутри каталога `server/`. Импорт из `#server` в клиентском коде приведёт к ошибке.
::

## Типы сервера

Автоимпорты и типы для каталога `server/` отличаются от `app/`, так как контекст выполнения другой.

По умолчанию Nuxt 4 генерирует [`tsconfig.json`](/docs/4.x/directory-structure/tsconfig) с project reference на папку `server/` для корректной типизации.

## Рецепты

### Параметры маршрута

В имени файла можно задавать динамические параметры в квадратных скобках, например `/api/hello/[name].ts`, и читать их через `event.context.params`.

```ts [server/api/hello/[name\\].ts]
export default defineEventHandler((event) => {
  const name = getRouterParam(event, 'name')

  return `Hello, ${name}!`
})
```

::tip{to="https://h3.dev/examples/validate-data#validate-params"}
Альтернатива — `getValidatedRouterParams` со схемой валидации, например Zod, для проверки во время выполнения и типобезопасности.
::

Этот API можно вызвать по `/api/hello/nuxt` и получить `Hello, nuxt!`.

### Соответствие HTTP-методу

К имени файла обработчика можно добавить суффиксы `.get`, `.post`, `.put`, `.delete` и т.д., чтобы сопоставить [HTTP-метод](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Methods) запроса.

```ts [server/api/test.get.ts]
export default defineEventHandler(() => 'Test get handler')
```

```ts [server/api/test.post.ts]
export default defineEventHandler(() => 'Test post handler')
```

В примере выше запрос к `/test`:

- метод **GET**: вернёт `Test get handler`
- метод **POST**: вернёт `Test post handler`
- любой другой метод: ошибка 405

Внутри каталога можно использовать `index.[method].ts` для другой структуры кода — удобно для пространств имён API.

::code-group
```ts [server/api/foo/index.get.ts]
export default defineEventHandler((event) => {
  // handle GET requests for the `api/foo` endpoint
})
```
```ts [server/api/foo/index.post.ts]
export default defineEventHandler((event) => {
  // handle POST requests for the `api/foo` endpoint
})
```
```ts [server/api/foo/bar.get.ts]
export default defineEventHandler((event) => {
  // handle GET requests for the `api/foo/bar` endpoint
})
```
::

### Маршрут catch-all

Catch-all маршруты удобны для запасной обработки.

Например, файл `~~/server/api/foo/[...].ts` зарегистрирует catch-all для запросов, не попавших в другие обработчики, например `/api/foo/bar/baz`.

```ts [server/api/foo/[...\\].ts]
export default defineEventHandler((event) => {
  // event.context.path to get the route path: '/api/foo/bar/baz'
  // event.context.params._ to get the route segment: 'bar/baz'
  return `Default foo handler`
})
```

Имя для catch-all задаётся файлом `~~/server/api/foo/[...slug].ts`; сегмент доступен как `event.context.params.slug`.

```ts [server/api/foo/[...slug\\].ts]
export default defineEventHandler((event) => {
  // event.context.params.slug to get the route segment: 'bar/baz'
  return `Default foo handler`
})
```

### Тело запроса

```ts [server/api/submit.post.ts]
export default defineEventHandler(async (event) => {
  const body = await readBody(event)
  return { body }
})
```

::tip{to="https://unjs.io/blog/2023-08-15-h3-towards-the-edge-of-the-web/#runtime-type-safe-request-utils"}
Альтернатива — `readValidatedBody` со схемой валидации, например Zod, для проверки во время выполнения и типобезопасности.
::

Вызов API:

```vue [app/app.vue]
<script setup lang="ts">
async function submit () {
  const { body } = await $fetch('/api/submit', {
    method: 'post',
    body: { test: 123 },
  })
}
</script>
```

::note
В имени файла `submit.post.ts` суффикс `.post` нужен, чтобы обрабатывать только запросы с методом `POST`, принимающие тело. При `readBody` в GET-запросе `readBody` выбросит HTTP-ошибку `405 Method Not Allowed`.
::

### Параметры запроса

Пример запроса `/api/query?foo=bar&baz=qux`

```ts [server/api/query.get.ts]
export default defineEventHandler((event) => {
  const query = getQuery(event)

  return { a: query.foo, b: query.baz }
})
```

::tip{to="https://unjs.io/blog/2023-08-15-h3-towards-the-edge-of-the-web#runtime-type-safe-request-utils"}
Альтернатива — `getValidatedQuery` со схемой валидации, например Zod, для проверки во время выполнения и типобезопасности.
::

### Обработка ошибок

Если ошибок нет, возвращается статус `200 OK`.

Необработанные ошибки дают HTTP-ошибку `500 Internal Server Error`.

Для других кодов ошибок выбросьте исключение через [`createError`](/docs/4.x/api/utils/create-error):

```ts [server/api/validation/[id\\].ts]
export default defineEventHandler((event) => {
  const id = Number.parseInt(event.context.params.id) as number

  if (!Number.isInteger(id)) {
    throw createError({
      status: 400,
      statusText: 'ID should be an integer',
    })
  }
  return 'All good'
})
```

### Коды статуса

Для других кодов статуса используйте утилиту [`setResponseStatus`](/docs/4.x/api/utils/set-response-status).

Например, чтобы вернуть `202 Accepted`

```ts [server/api/validation/[id\\].ts]
export default defineEventHandler((event) => {
  setResponseStatus(event, 202)
})
```

### Runtime Config

::code-group
```ts [server/api/foo.ts]
export default defineEventHandler(async (event) => {
  const config = useRuntimeConfig(event)

  const repo = await $fetch('https://api.github.com/repos/nuxt/nuxt', {
    headers: {
      Authorization: `token ${config.githubToken}`,
    },
  })

  return repo
})
```
```ts [nuxt.config.ts]
export default defineNuxtConfig({
  runtimeConfig: {
    githubToken: '',
  },
})
```
```ini [.env]
NUXT_GITHUB_TOKEN='<my-super-token>'
```
::

::note
Передавать `event` в `useRuntimeConfig` не обязательно, но рекомендуется: так runtime config для серверных маршрутов может переопределяться [переменными окружения](/docs/4.x/guide/going-further/runtime-config#environment-variables) во время выполнения.
::

### Cookies запроса

```ts [server/api/cookies.ts]
export default defineEventHandler((event) => {
  const cookies = parseCookies(event)

  return { cookies }
})
```

### Проброс контекста и заголовков

По умолчанию при `fetch` из серверных маршрутов не пробрасываются ни заголовки входящего запроса, ни его контекст. Используйте `event.$fetch`, чтобы пробрасывать контекст и заголовки при запросах из серверных маршрутов.

```ts [server/api/forward.ts]
export default defineEventHandler((event) => {
  return event.$fetch('/api/forwarded')
})
```

::note
Заголовки, которые **не предназначены для проброса**, в запрос **не попадут**. Например:
`transfer-encoding`, `connection`, `keep-alive`, `upgrade`, `expect`, `host`, `accept`
::

### Ожидание промисов после ответа

Иногда нужны асинхронные задачи, которые не должны задерживать ответ клиенту (кэширование, логирование). Для этого используйте `event.waitUntil`: промис выполнится в фоне без задержки ответа.

`event.waitUntil` принимает промис, который будет дожидаться до завершения обработчика, чтобы задача успела выполниться даже если сервер иначе завершил бы обработчик сразу после отправки ответа. Это согласуется с возможностями рантайма для операций после ответа.

```ts [server/api/background-task.ts]
const timeConsumingBackgroundTask = async () => {
  await new Promise(resolve => setTimeout(resolve, 1000))
}

export default eventHandler((event) => {
  // schedule a background task without blocking the response
  event.waitUntil(timeConsumingBackgroundTask())

  // immediately send the response to the client
  return 'done'
})
```

## Продвинутое использование

### Конфигурация Nitro

В `nuxt.config` ключ `nitro` задаёт [конфигурацию Nitro](https://nitro.build/config) напрямую.

::warning
Это продвинутый вариант. Своя конфигурация может влиять на production: интерфейс конфигурации может меняться при обновлении Nitro в минорных версиях Nuxt.
::

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  // https://nitro.build/config
  nitro: {},
})
```

:read-more{to="/docs/4.x/guide/concepts/server-engine"}

### Вложенный роутер

```ts [server/api/hello/[...slug\\].ts]
import { createRouter, defineEventHandler, useBase } from 'h3'

const router = createRouter()

router.get('/test', defineEventHandler(() => 'Hello World'))

export default useBase('/api/hello', router.handler)
```

### Отправка потоков

::tip
Экспериментальная возможность, доступна во всех средах.
::

```ts [server/api/foo.get.ts]
import fs from 'node:fs'
import { sendStream } from 'h3'

export default defineEventHandler((event) => {
  return sendStream(event, fs.createReadStream('/path/to/file'))
})
```

### Редирект

```ts [server/api/foo.get.ts]
export default defineEventHandler(async (event) => {
  await sendRedirect(event, '/path/redirect/to', 302)
})
```

### Устаревшие обработчики и middleware

```ts [server/api/legacy.ts]
export default fromNodeMiddleware((req, res) => {
  res.end('Legacy handler')
})
```

::important
Поддержка legacy возможна через [h3js/h3](https://github.com/h3js/h3), но по возможности избегайте устаревших обработчиков.
::

```ts [server/middleware/legacy.ts]
export default fromNodeMiddleware((req, res, next) => {
  console.log('Legacy middleware')
  next()
})
```

::warning
Не сочетайте колбэк `next()` с legacy middleware, который `async` или возвращает `Promise`.
::

### Хранилище сервера

Nitro даёт кроссплатформенный [слой хранилища](https://nitro.build/guide/storage). Дополнительные точки монтирования настраиваются через `nitro.storage` или [плагины сервера](/docs/4.x/directory-structure/server#server-plugins).

**Пример: Redis через `nitro.storage`:**

```ts [nuxt.config.ts]
export default defineNuxtConfig({
  nitro: {
    storage: {
      redis: {
        driver: 'redis',
        /* redis connector options */
        port: 6379, // Redis port
        host: '127.0.0.1', // Redis host
        username: '', // needs Redis >= 6
        password: '',
        db: 0, // Defaults to 0
        tls: {}, // tls/ssl
      },
    },
  },
})
```

В обработчике API:

```ts [server/api/storage/test.ts]
export default defineEventHandler(async (event) => {
  // List all keys with
  const keys = await useStorage('redis').getKeys()

  // Set a key with
  await useStorage('redis').setItem('foo', 'bar')

  // Remove a key with
  await useStorage('redis').removeItem('foo')

  return {}
})
```

::read-more{to="https://nitro.build/guide/storage" target="_blank"}
Подробнее о слое хранилища Nitro.
::

Альтернатива — точка монтирования через плагин сервера и runtime config:

::code-group
```ts [server/plugins/storage.ts]
import redisDriver from 'unstorage/drivers/redis'

export default defineNitroPlugin(() => {
  const storage = useStorage()

  // Dynamically pass in credentials from runtime configuration, or other sources
  const driver = redisDriver({
    base: 'redis',
    host: useRuntimeConfig().redis.host,
    port: useRuntimeConfig().redis.port,
    /* other redis connector options */
  })

  // Mount driver
  storage.mount('redis', driver)
})
```

``` ts [nuxt.config.ts]
export default defineNuxtConfig({
  runtimeConfig: {
    redis: { // Default values
      host: '',
      port: 0,
      /* other redis connector options */
    },
  },
})
```
::

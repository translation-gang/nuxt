Nuxt автоматически импортирует компоненты из этого каталога (и зарегистрированные модулями).

```bash [Directory Structure]
-| components/
---| AppHeader.vue
---| AppFooter.vue
```

```html [app/app.vue]
<template>
  <div>
    <AppHeader />
    <NuxtPage />
    <AppFooter />
  </div>
</template>
```

## Имена компонентов

Компонент во вложенных каталогах, например:

```bash [Directory Structure]
-| components/
---| base/
-----| foo/
-------| Button.vue
```

... имя строится из пути и имени файла с удалением дублирующихся сегментов:

```html
<BaseFooButton />
```

::note
Для ясности имя файла лучше совпадало с именем компонента — например переименовать `Button.vue` в `BaseFooButton.vue`.
::

Чтобы автоимпорт учитывал только имя файла, а не путь, задайте `pathPrefix: false` в расширенной форме конфигурации:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  components: [
    {
      path: '~/components',
      pathPrefix: false, // [!code ++]
    },
  ],
})
```

Так же, как в Nuxt 2: `~/components/Some/MyComponent.vue` станет `<MyComponent>`, а не `<SomeMyComponent>`.

## Динамические компоненты

Для синтаксиса Vue `<component :is="someComputedComponent">`{lang=vue} используйте хелпер `resolveComponent` или импортируйте компонент из `#components` и передайте в проп `is`.

Пример:

```vue [app/pages/index.vue]
<script setup lang="ts">
import { SomeComponent } from '#components'

const MyButton = resolveComponent('MyButton')
</script>

<template>
  <component :is="clickable ? MyButton : 'div'" />
  <component :is="SomeComponent" />
</template>
```

::important
В `resolveComponent` передавайте только литеральную строку с именем компонента, не переменную и не выражение со строкой — имя анализируется на этапе компиляции.
::

:video-accordion{title="Короткое видео Daniel Roe про resolveComponent()" videoId="4kq8E5IUM2U"}

Не рекомендуется, но можно зарегистрировать все компоненты глобально — для каждого будет отдельный async-чанк.

```diff
  export default defineNuxtConfig({
    components: {
+     global: true,
+     dirs: ['~/components']
    },
  })
```

Отдельные компоненты можно сделать глобальными через каталог `~/components/global` или суффикс `.global.vue`. У каждого глобального компонента свой чанк — не злоупотребляйте.

::note
Опцию `global` можно задать отдельно для каждого каталога компонентов.
::

## Динамический импорт

Для ленивой загрузки добавьте префикс `Lazy` к имени компонента — удобно, если он нужен не всегда.

Код компонента подгрузится позже, что помогает уменьшить размер JS-бандла.

```vue [app/pages/index.vue]
<script setup lang="ts">
const show = ref(false)
</script>

<template>
  <div>
    <h1>Mountains</h1>
    <LazyMountainsList v-if="show" />
    <button
      v-if="!show"
      @click="show = true"
    >
      Show List
    </button>
  </div>
</template>
```

## Отложенная (ленивая) гидратация

Lazy-компоненты помогают с чанками, но без `v-if` они всё равно подгружаются рано. На страницах много контента и компонентов, и не всем нужна интерактивность сразу — ранняя загрузка всего вредит производительности.

Можно отложить гидратацию до появления в области просмотра или до менее загруженного момента в браузере.

Nuxt поддерживает отложенную гидратацию — вы контролируете, когда компонент становится интерактивным.

### Стратегии гидратации

Встроенные стратегии; на один lazy-компонент — одна стратегия.

::note
Любое изменение пропа у лениво гидратируемого компонента сразу запускает гидратацию (например у компонента с `hydrate-never`).
::

::warning
Встроенная ленивая гидратация Nuxt работает только в однофайловых компонентах (SFC), проп нужно задать в шаблоне явно (не через spread объекта в `v-bind`). Не работает с прямым импортом из `#components`.
::

#### `hydrate-on-visible`

Гидратация, когда элемент попадает в viewport.

```vue [app/pages/index.vue]
<template>
  <div>
    <LazyMyComponent hydrate-on-visible />
  </div>
</template>
```

::read-more{to="https://developer.mozilla.org/en-US/docs/Web/API/IntersectionObserver/IntersectionObserver" title="Опции IntersectionObserver"}
Подробнее об опциях `hydrate-on-visible`.
::

::note
Внутри используется встроенная в Vue стратегия [`hydrateOnVisible`](https://vuejs.org/guide/components/async#hydrate-on-visible).
::

#### `hydrate-on-idle`

Гидратация в простое браузера. Подходит, если компонент нужен скоро, но не должен блокировать критический путь рендера.

Можно передать число — максимальная задержка (таймаут).

```vue [app/pages/index.vue]
<template>
  <div>
    <LazyMyComponent hydrate-on-idle />
  </div>
</template>
```

::note
Внутри — [`hydrateOnIdle`](https://vuejs.org/guide/components/async#hydrate-on-idle) во Vue.
::

#### `hydrate-on-interaction`

Гидратация после заданного действия (клик, mouseover и т.д.).

```vue [app/pages/index.vue]
<template>
  <div>
    <LazyMyComponent hydrate-on-interaction="mouseover" />
  </div>
</template>
```

Без указания событий по умолчанию гидратация на `pointerenter`, `click` и `focus`.

::note
Внутри — [`hydrateOnInteraction`](https://vuejs.org/guide/components/async#hydrate-on-interaction) во Vue.
::

#### `hydrate-on-media-query`

Гидратация, когда окно соответствует media query.

```vue [app/pages/index.vue]
<template>
  <div>
    <LazyMyComponent hydrate-on-media-query="(max-width: 768px)" />
  </div>
</template>
```

::note
Внутри — [`hydrateOnMediaQuery`](https://vuejs.org/guide/components/async#hydrate-on-media-query) во Vue.
::

#### `hydrate-after`

Гидратация через указанную задержку в миллисекундах.

```vue [app/pages/index.vue]
<template>
  <div>
    <LazyMyComponent :hydrate-after="2000" />
  </div>
</template>
```

#### `hydrate-when`

Гидратация по булевому условию.

```vue [app/pages/index.vue]
<template>
  <div>
    <LazyMyComponent :hydrate-when="isReady" />
  </div>
</template>

<script setup lang="ts">
const isReady = ref(false)
function myFunction () {
  // trigger custom hydration strategy...
  isReady.value = true
}
</script>
```

#### `hydrate-never`

Гидратация не выполняется.

```vue [app/pages/index.vue]
<template>
  <div>
    <LazyMyComponent hydrate-never />
  </div>
</template>
```

### Событие гидратации

У компонентов с отложенной гидратацией есть событие `@hydrated` после гидратации.

```vue [app/pages/index.vue]
<template>
  <div>
    <LazyMyComponent
      hydrate-on-visible
      @hydrated="onHydrate"
    />
  </div>
</template>

<script setup lang="ts">
function onHydrate () {
  console.log('Component has been hydrated!')
}
</script>
```

### Ограничения и практики

Отложенная гидратация ускоряет загрузку только при правильном применении:

1. **Контент в viewport:** не откладывайте гидратацию для критичного контента «на первом экране». Стратегия для того, что не нужно сразу.

2. **Условный рендер:** при `v-if="false"` на lazy-компоненте отдельная отложенная гидратация часто не нужна — достаточно обычного lazy.

3. **Общее состояние:** при общем `v-model` обновление в одном компоненте может запустить гидратацию во всех связанных.

4. **Назначение стратегий:**
    * `hydrate-when` — когда гидратация не всегда нужна.
    * `hydrate-after` — можно подождать фиксированное время.
    * `hydrate-on-idle` — когда уместна гидратация в простое браузера.

5. **Не `hydrate-never` для интерактивных компонентов:** если нужны действия пользователя, полный отказ от гидратации не подходит.

## Прямые импорты

Явный импорт из `#components` обходит автоимпорт Nuxt.

```vue [app/pages/index.vue]
<script setup lang="ts">
import { LazyMountainsList, NuxtLink } from '#components'

const show = ref(false)
</script>

<template>
  <div>
    <h1>Mountains</h1>
    <LazyMountainsList v-if="show" />
    <button
      v-if="!show"
      @click="show = true"
    >
      Show List
    </button>
    <NuxtLink to="/">Home</NuxtLink>
  </div>
</template>
```

## Свои каталоги

По умолчанию сканируется только `~/components`. Другие каталоги и правила сканирования добавляются в конфигурацию:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  components: [
    // ~/calendar-module/components/event/Update.vue => <EventUpdate />
    { path: '~/calendar-module/components' },

    // ~/user-module/components/account/UserDeleteDialog.vue => <UserDeleteDialog />
    { path: '~/user-module/components', pathPrefix: false },

    // ~/components/special-components/Btn.vue => <SpecialBtn />
    { path: '~/components/special-components', prefix: 'Special' },

    // It's important that this comes last if you have overrides you wish to apply
    // to sub-directories of `~/components`.
    //
    // ~/components/Btn.vue => <Btn />
    // ~/components/base/Btn.vue => <BaseBtn />
    '~/components',
  ],
})
```

::note
Вложенные каталоги указывайте раньше в списке — порядок сканирования важен.
::

## Пакеты npm

Автоимпорт компонентов из npm оформляйте через [`addComponent`](/docs/4.x/api/kit/components#addcomponent) в [локальном модуле](/docs/4.x/directory-structure/modules).

::code-group

```ts twoslash [~/modules/register-component.ts]
import { addComponent, defineNuxtModule } from '@nuxt/kit'

export default defineNuxtModule({
  setup () {
    // import { MyComponent as MyAutoImportedComponent } from 'my-npm-package'
    addComponent({
      name: 'MyAutoImportedComponent',
      export: 'MyComponent',
      filePath: 'my-npm-package',
    })
  },
})
```

```vue [app/app.vue]
<template>
  <div>
    <!--  the component uses the name we specified and is auto-imported  -->
    <MyAutoImportedComponent />
  </div>
</template>
```

::

## Расширения файлов

По умолчанию компонентом считается любой файл с расширением из [ключа `extensions` в `nuxt.config.ts`](/docs/4.x/api/nuxt-config#extensions).
Чтобы ограничить расширения, используйте расширенную форму объявления каталога компонентов и ключ `extensions`:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  components: [
    {
      path: '~/components',
      extensions: ['.vue'], // [!code ++]
    },
  ],
})
```

## Только клиент

Для рендера только на клиенте добавьте суффикс `.client` к имени компонента.

```bash [Directory Structure]
| components/
--| Comments.client.vue
```

```vue [app/pages/example.vue]
<template>
  <div>
    <!-- this component will only be rendered on client side -->
    <Comments />
  </div>
</template>
```

::note
Работает только с автоимпортом Nuxt и импортом из `#components`. Явный импорт по реальному пути не превращает компонент в client-only.
::

::important
Компоненты `.client` рендерятся только после монтирования. Чтобы в `onMounted()` увидеть разметку, добавьте `await nextTick()` в колбэк.
::

::read-more{to="/docs/4.x/api/components/client-only"}
Похожий эффект даёт `<ClientOnly>`.
::

## Серверные компоненты

Серверные компоненты рендерятся на сервере внутри клиентского приложения. В Nuxt они доступны и при статической генерации — можно смешивать динамику, серверный HTML и статические фрагменты.

Их можно использовать отдельно или в паре с [клиентским компонентом](/docs/4.x/directory-structure/app/components#paired-with-a-client-component).

:video-accordion{title="Видео Learn Vue о серверных компонентах Nuxt" videoId="u1yyXe86xJM"}

::tip{icon="i-lucide-newspaper" to="https://roe.dev/blog/nuxt-server-components" target="_blank"}
Гайд Daniel Roe по серверным компонентам Nuxt.
::

### Автономные серверные компоненты

Всегда рендерятся на сервере (островные / Islands-компоненты).

При обновлении пропсов уходит сетевой запрос и HTML обновляется на месте.

Сейчас это эксперимент: в `nuxt.config` нужно включить «component islands»:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  experimental: {
    componentIslands: true,
  },
})
```

Компоненты только для сервера регистрируются суффиксом `.server` и доступны в приложении с автоимпортом.

```bash [Directory Structure]
-| components/
---| HighlightedMarkdown.server.vue
```

```vue [app/pages/example.vue]
<template>
  <div>
    <!--
      this will automatically be rendered on the server, meaning your markdown parsing + highlighting
      libraries are not included in your client bundle.
     -->
    <HighlightedMarkdown markdown="# Headline" />
  </div>
</template>
```

Под капотом — [`<NuxtIsland>`](/docs/4.x/api/components/nuxt-island): проп `lazy` и слот `#fallback` пробрасываются дальше.

::warning
У серверных компонентов и островов один корневой элемент (комментарии HTML тоже считаются узлами).
::

::warning
Пропы серверных компонентов передаются через query в URL — длина URL ограничена, не передавайте огромные данные в пропах.
::

::warning
Вложенные острова дают дополнительные накладные расходы.
::

::warning
Слоты, клиентские компоненты и большинство возможностей server-only и island доступны только в однофайловых компонентах.
::

#### Клиентские компоненты внутри серверных

::note
Нужно `experimental.componentIslands.selectiveClient: true` в конфигурации.
::

Частичная гидратация: атрибут `nuxt-client` на компоненте, который должен загрузиться на клиенте.

```vue [app/components/ServerWithClient.vue]
<template>
  <div>
    <HighlightedMarkdown markdown="# Headline" />
    <!-- Counter will be loaded and hydrated client-side -->
    <Counter
      nuxt-client
      :count="5"
    />
  </div>
</template>
```

::note
Только внутри серверного компонента. Слоты клиентских компонентов работают с `experimental.componentIsland.selectiveClient: 'deep'`; они рендерятся на сервере и после клиента остаются неинтерактивными.
::

#### Контекст серверного компонента

При рендере server-only или island `<NuxtIsland>` делает fetch и получает `NuxtIslandResponse` (внутренний запрос на сервере или видимый в Network при клиентской навигации).

Следствия:

* На сервере создаётся отдельное Vue-приложение для ответа острова.
* При рендере появляется контекст острова.
* Контекст острова недоступен остальному приложению и наоборот — остров _изолирован_.
* Плагины выполняются снова при рендере острова, кроме случая `env: { islands: false }` в плагине в виде объекта.

::important
Middleware маршрутов при рендере островов не запускается: это концепция страниц, не компонентов.
::

Внутри острова контекст: `nuxtApp.ssrContext.islandContext`. Пока острова экспериментальны, формат может меняться.

::note
Слоты могут быть интерактивными и оборачиваются в `<div style="display: contents">`.
::

### Пара с клиентским компонентом

Пара `.server` + `.client` — две «половины» одного компонента для разной реализации на сервере и клиенте в сложных сценариях.

```bash [Directory Structure]
-| components/
---| Comments.client.vue
---| Comments.server.vue
```

```vue [app/pages/example.vue]
<template>
  <div>
    <!-- this component will render Comments.server on the server then Comments.client once mounted in the browser -->
    <Comments />
  </div>
</template>
```

## Авторам библиотек

Собрать библиотеку Vue-компонентов с tree-shaking и регистрацией в Nuxt несложно.

В модуле Nuxt вызывайте [`addComponentsDir`](/docs/4.x/api/kit/components#addcomponentsdir) из `@nuxt/kit`.

Пример структуры:

```bash [Directory Structure]
-| node_modules/
---| awesome-ui/
-----| components/
-------| Alert.vue
-------| Button.vue
-----| nuxt.ts
-| pages/
---| index.vue
-| nuxt.config.ts
```

В `awesome-ui/nuxt.ts`:

```ts twoslash
import { addComponentsDir, createResolver, defineNuxtModule } from '@nuxt/kit'

export default defineNuxtModule({
  setup () {
    const resolver = createResolver(import.meta.url)

    // Add ./components dir to the list
    addComponentsDir({
      path: resolver.resolve('./components'),
      prefix: 'awesome',
    })
  },
})
```

В проекте подключите UI как модуль в `nuxt.config`:

```ts twoslash [nuxt.config.ts]
export default defineNuxtConfig({
  modules: ['awesome-ui/nuxt'],
})
```

Компоненты с префиксом `awesome-` в `app/pages/index.vue`:

```vue
<template>
  <div>
    My <AwesomeButton>UI button</AwesomeButton>!
    <awesome-alert>Here's an alert!</awesome-alert>
  </div>
</template>
```

Импорт только используемых компонентов и HMR при правках в `node_modules/awesome-ui/components/`.

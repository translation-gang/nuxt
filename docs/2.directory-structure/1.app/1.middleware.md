Nuxt предлагает настраиваемый фреймворк **middleware маршрутов**: код, который выполняется до перехода на нужный маршрут.

Три вида middleware маршрутов:

1. Анонимный (inline) — задаётся прямо на странице.
2. Именованный — в `app/middleware/`, подгружается асинхронно при использовании на странице.
3. Глобальный — в `app/middleware/` с суффиксом `.global`, выполняется при каждой смене маршрута.

Первые два типа задаются в [`definePageMeta`](/docs/4.x/api/utils/define-page-meta).

::note
Имена middleware приводятся к kebab-case: `myMiddleware` → `my-middleware`.
::

::note
Middleware маршрутов выполняется во Vue-части Nuxt. По названию они похожи на [server middleware](/docs/4.x/directory-structure/server#server-middleware), но это другой механизм — тот работает на сервере Nitro.
::

## Использование

Middleware маршрутов — это navigation guards: им передаются текущий и следующий маршрут.

```ts twoslash [middleware/my-middleware.ts]
export default defineNuxtRouteMiddleware((to, from) => {
  if (to.params.id === '1') {
    return abortNavigation()
  }
  // In a real app you would probably not redirect every route to `/`
  // however it is important to check `to.path` before redirecting or you
  // might get an infinite redirect loop
  if (to.path !== '/') {
    return navigateTo('/')
  }
})
```

Два глобальных хелпера можно вернуть из middleware:

1. [`navigateTo`](/docs/4.x/api/utils/navigate-to) — перенаправление на маршрут
2. [`abortNavigation`](/docs/4.x/api/utils/abort-navigation) — отмена навигации, опционально с сообщением об ошибке

В отличие от [navigation guards](https://router.vuejs.org/guide/advanced/navigation-guards#Global-Before-Guards) в `vue-router`, третий аргумент `next()` не передаётся — **редирект и отмена задаются возвращаемым значением**.

Возможные результаты:

* ничего (пустой `return` или отсутствие return) — навигация не блокируется, выполняется следующий middleware или завершается переход
* `return navigateTo('/')` — редирект; код [`302` Found](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status/302), если редирект на сервере
* `return navigateTo('/', { redirectCode: 301 })` — редирект с [`301` Moved Permanently](https://developer.mozilla.org/en-US/docs/Web/HTTP/Reference/Status/301) на сервере
* `return abortNavigation()` — остановка навигации
* `return abortNavigation(error)` — отклонение навигации с ошибкой

:read-more{to="/docs/4.x/api/utils/navigate-to"}
:read-more{to="/docs/4.x/api/utils/abort-navigation"}

::important
Для редиректов и остановки навигации лучше использовать перечисленные хелперы. Другие варианты из [документации vue-router](https://router.vuejs.org/guide/advanced/navigation-guards#Global-Before-Guards) могут работать, но в будущем возможны breaking changes.
::

## Порядок middleware

1. Глобальный middleware
2. Порядок middleware на странице (если указан массив)

Пример:

```bash [app/middleware/ directory]
-| middleware/
---| analytics.global.ts
---| setup.global.ts
---| auth.ts
```

```vue twoslash [pages/profile.vue]
<script setup lang="ts">
definePageMeta({
  middleware: [
    function (to, from) {
      // Custom inline middleware
    },
    'auth',
  ],
})
</script>
```

Порядок выполнения:

1. `analytics.global.ts`
2. `setup.global.ts`
3. Inline middleware
4. `auth.ts`

### Порядок глобального middleware

По умолчанию глобальные файлы выполняются в алфавитном порядке имён.

Чтобы задать свой порядок (например, `setup.global.ts` раньше `analytics.global.ts`), добавьте префикс с «алфавитной» нумерацией:

```bash [Directory structure]
-| middleware/
---| 01.setup.global.ts
---| 02.analytics.global.ts
---| auth.ts
```

::note
Имена сортируются как строки, не как числа: `10.new.global.ts` окажется раньше `2.new.global.ts`. Поэтому однозначные номера дополняют ведущим `0`.
::

## Когда выполняется middleware

При SSR или генерации middleware для первой страницы выполняется и при рендере, и снова на клиенте. Это нужно, если middleware зависит от браузера (генерация, агрессивный кэш, localStorage).

Чтобы изменить поведение:

```ts twoslash [middleware/example.ts]
export default defineNuxtRouteMiddleware((to) => {
  // skip middleware on server
  if (import.meta.server) {
    return
  }
  // skip middleware on client side entirely
  if (import.meta.client) {
    return
  }
  // or only skip middleware on initial client load
  const nuxtApp = useNuxtApp()
  if (import.meta.client && nuxtApp.isHydrating && nuxtApp.payload.serverRendered) {
    return
  }
})
```

Так будет даже если на сервере в middleware выброшена ошибка и отрендерена страница ошибки — в браузере middleware снова выполнится.

::note
Страница ошибки — отдельная загрузка, middleware запустится заново. В middleware можно вызвать [`useError`](/docs/4.x/getting-started/error-handling#useerror), чтобы проверить обработку ошибки.
::

## Доступ к маршруту в middleware

Используйте параметры `to` и `from`. Не используйте [`useRoute()`](/docs/4.x/api/composables/use-route) в middleware.

**В middleware нет понятия «текущего маршрута»**: навигация может быть отменена или перенаправлена, `useRoute()` в этом контексте будет неточным.

::warning
Composable, который внутри вызывает `useRoute()`, может вызвать предупреждение даже без прямого вызова в middleware.
Решение то же — передавайте маршрут аргументом в функции, вызываемые из middleware.
::

::code-group
```ts twoslash [middleware/access-route.ts]
// @errors: 2304
export default defineNuxtRouteMiddleware((to) => {
  // passing the route to the function to avoid calling `useRoute()` in middleware
  doSomethingWithRoute(to)

  // ❌ this will output a warning and is NOT recommended
  callsRouteInternally()
})
```

```ts twoslash [utils/handle-route.ts]
// providing the route as an argument so that it can be used in middleware correctly
export function doSomethingWithRoute (route = useRoute()) {
  // ...
}
```
```ts twoslash [utils/dont-do-this.ts]
// ❌ this function is not suitable for use in middleware
export function callsRouteInternally () {
  const route = useRoute()
  // ...
}
```

::

## Добавление middleware динамически

Глобальный или именованный middleware можно зарегистрировать вручную через [`addRouteMiddleware()`](/docs/4.x/api/utils/add-route-middleware), например из плагина.

```ts twoslash
export default defineNuxtPlugin(() => {
  addRouteMiddleware('global-test', () => {
    console.log('this global middleware was added in a plugin and will be run on every route change')
  }, { global: true })

  addRouteMiddleware('named-test', () => {
    console.log('this named middleware was added in a plugin and would override any existing middleware of the same name')
  })
})
```

## Middleware на этапе сборки

Вместо `definePageMeta` на каждой странице можно задать именованный middleware в хуке `pages:extend`.

```ts twoslash [nuxt.config.ts]
import type { NuxtPage } from 'nuxt/schema'

export default defineNuxtConfig({
  hooks: {
    'pages:extend' (pages) {
      function setMiddleware (pages: NuxtPage[]) {
        for (const page of pages) {
          if (/* some condition */ Math.random() > 0.5) {
            page.meta ||= {}
            // Note that this will override any middleware set in `definePageMeta` in the page
            page.meta.middleware = ['named']
          }
          if (page.children) {
            setMiddleware(page.children)
          }
        }
      }
      setMiddleware(pages)
    },
  },
})
```

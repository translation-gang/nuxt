## Разработка, сборка и generate

В Nuxt CLI встроена поддержка [dotenv](https://github.com/motdotla/dotenv) в режиме разработки и при [`nuxt build`](/docs/4.x/api/commands/build) и [`nuxt generate`](/docs/4.x/api/commands/generate).

Помимо переменных процесса, если в корне проекта есть `.env`, он подхватывается **при dev, build и generate**. Переменные из него доступны в `nuxt.config` и в модулях.

```ini [.env]
MY_ENV_VARIABLE=hello
```

::note
Удаление переменной из `.env` или удаление самого `.env` не сбрасывает значения, уже попавшие в окружение.
::

## Свой файл

Чтобы использовать другой файл (например `.env.local` или `.env.production`), передайте флаг `--dotenv` в Nuxt CLI.

```bash [Terminal]
npx nuxt dev --dotenv .env.local
```

При изменении `.env` в режиме разработки экземпляр Nuxt перезапускается, чтобы подтянуть новые значения в `process.env`.

::important
В коде приложения лучше использовать [Runtime Config](/docs/4.x/guide/going-further/runtime-config), а не «голые» переменные окружения.
::

## Production

**После сборки сервера** вы сами задаёте переменные окружения при запуске.

Файлы `.env` в этот момент не читаются. Способ зависит от среды.

Так сделано для совместимости с разными платформами деплоя, в том числе без классической файловой системы (serverless, edge вроде Cloudflare Workers).

Раз `.env` в production не используется, переменные нужно задавать средствами хостинга. Типичные варианты:

* Передать переменные в командной строке:

   `$ DATABASE_HOST=mydatabaseconnectionstring node .output/server/index.mjs`

* Задать в профильных файлах оболочки (`.bashrc`, `.profile`).

* У облачных провайдеров (Vercel, Netlify, AWS) — через панель, CLI или конфигурационные файлы.

::important
`runtimeConfig` [не подхватит переменные окружения в production, если они не начинаются с `NUXT_`](https://nuxt.com/docs/4.x/guide/going-further/runtime-config#environment-variables).
::

## Локальный просмотр production

Для локального просмотра production-сборки удобно [`nuxt preview`](/docs/4.x/api/commands/preview): `.env` подгружается в `process.env`. Нужны установленные зависимости в каталоге пакета.

Или передайте переменные в командной строке, например на Linux или macOS:

```bash [Terminal]
DATABASE_HOST=mydatabaseconnectionstring node .output/server/index.mjs
```

Для чисто статического сайта после пререндера нельзя задать runtime-конфигурацию снаружи.

:read-more{to="/docs/4.x/guide/going-further/runtime-config"}

::note
Если нужны переменные на этапе сборки без последующего обновления снаружи (или только реактивное обновление _внутри_ приложения), может подойти `appConfig`. Его можно задать в `nuxt.config` (через переменные окружения) и в файле `~/app.config.ts`.
:read-more{to="/docs/4.x/directory-structure/app/app-config"}
::

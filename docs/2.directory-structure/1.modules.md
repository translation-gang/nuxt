Сюда удобно класть локальные модули, которые вы пишете в процессе разработки.

Шаблоны авторегистрации:
- `modules/*/index.ts`
- `modules/*.ts`

Эти локальные модули не нужно отдельно перечислять в [`nuxt.config.ts`](/docs/4.x/directory-structure/nuxt-config).

::code-group

```ts twoslash [modules/hello/index.ts]
// `nuxt/kit` is a helper subpath import you can use when defining local modules
// that means you do not need to add `@nuxt/kit` to your project's dependencies
import { addComponentsDir, addServerHandler, createResolver, defineNuxtModule } from 'nuxt/kit'

export default defineNuxtModule({
  meta: {
    name: 'hello',
  },
  setup () {
    const resolver = createResolver(import.meta.url)

    // Add an API route
    addServerHandler({
      route: '/api/hello',
      handler: resolver.resolve('./runtime/api-route'),
    })

    // Add components
    addComponentsDir({
      path: resolver.resolve('./runtime/app/components'),
      pathPrefix: true, // Prefix your exports to avoid conflicts with user code or other modules
    })
  },
})
```

```ts twoslash [modules/hello/runtime/api-route.ts]
export default defineEventHandler(() => {
  return { hello: 'world' }
})
```

::

При старте Nuxt модуль `hello` зарегистрируется, маршрут `/api/hello` станет доступен.

::note
Все компоненты, страницы, composables и прочие файлы, которые обычно лежат в `app/`, для модуля нужно размещать в `modules/your-module/runtime/app/`, чтобы типизация работала корректно.
::

Порядок выполнения модулей:
- Сначала загружаются модули из [`nuxt.config.ts`](/docs/4.x/api/nuxt-config#modules-1).
- Затем выполняются модули из каталога `modules/` — в алфавитном порядке.

Порядок локальных модулей можно задать префиксом-номером в имени каталога:

```bash [Directory structure]
modules/
  1.first-module/
    index.ts
  2.second-module.ts
```
